# Architecture Deep Reference

Deep internals of the wowlua_ls type system, inference engine, narrowing, and cross-file resolution — plus the relocated source-file deep notes, configuration internals, inlay hints, the annotation & type-system feature reference, and the diagnostics module reference (all formerly inline in CLAUDE.md). For the always-loaded source-file map, conventions, crate layering, and testing/stub workflow, see [CLAUDE.md](../CLAUDE.md); for the full test-file catalog and assertion format, see [TESTING.md](TESTING.md).

## Code lens (in `queries.rs` + `main_loop.rs`)
`code_lens_targets(tree)` collects one entry per function definition in the file: top-level named functions (scope-0 symbols with `ValueType::Function`), class methods (from `ir.classes`), and non-class table functions (scope-0 table-typed symbols). Each `CodeLensTarget` carries the function name, definition byte range (`def_start`/`def_end`), and a `name_offset` pointing inside the function's name token (suitable for `reference_target_at`).

The "N usages" LSP flow is two-stage:
1. **`textDocument/codeLens`** returns unresolved lenses (range + `data` containing URI and name_offset, no command).
2. **`codeLens/resolve`** counts references via `find_references_across_workspace` (same `text.contains(name)` prefilter + rayon parallel disk scan as find-references) and returns a `Command` with title "N usages" / "1 usage" / "0 usages" and command `editor.action.showReferences`.

`code_lens()` returns `Vec<CodeLensData>` with two additional kinds of annotations:
1. **"N implementations"** — shown on each local `@class` declaration. Counts direct subclasses by inverting the parent→child map from all classes (local + external). Uses `ir.class_def_ranges` for positioning and `table.parent_classes` for the child count.
2. **"overrides Parent"** — shown on methods that override a parent class method. Walks all local tables' function-typed fields, uses `function_owner_class` to find the owning class, then checks the class's `parent_classes` chain for a parent defining the same method name. Parent method detection uses a `class_methods: HashMap<class_name, HashSet<method_name>>` built from both local tables (via `function_owner_class`) and external class tables (via `field.annotation` or `Expr::FunctionDef`).

LSP handler in `main_loop.rs` registers `codeLensProvider` and converts byte ranges to LSP positions. "N implementations" uses two-stage resolve (like "N usages"): initial lens has `command: None` with `className` in `data`; on `codeLens/resolve`, `find_implementations_across_workspace()` searches `ws_file_classes` for child classes and returns their `def_range`/`def_path` as locations via `wowlua-ls.showReferences`. "overrides Parent" uses `wowlua-ls.showSuperDefinition` (navigates to parent definition).

## Cross-file find-references / rename
`references_at(offset)` runs against a single tree. For workspace-wide search, the LSP handler (`lsp/main_loop.rs::find_references_across_workspace`) composes three queries:
1. `AnalysisResult::reference_target_at(offset)` returns a `ReferenceTarget` (either `Symbol { idx, name }` or `Field { table_idx, field_name }`). An index `>= EXT_BASE` is stable across every `AnalysisResult` built from the same `PreResolvedGlobals`.
2. `AnalysisResult::promote_to_cross_file(&target)` lifts a file-local symbol or table to its workspace-wide counterpart when one exists (the defining file keeps a shadowing scope-0 local for its own global functions and a local `@class` table for its own `@class` declarations — both are swapped out for the `EXT_BASE+` idx when searching elsewhere).
3. `AnalysisResult::references_for_target(tree, &target, include_declaration, strict_shadow)` runs the search over an arbitrary tree against an externally-resolved target, enabling the LSP handler to iterate every open document and every scanned workspace file (rayon-parallel, gated by a `text.contains(target.name())` prefilter).

Consumer → defining-file matching works because the `Symbol` arm of `references_for_target` also accepts a scope-0 local whose name is in `ext.scope0_symbols` when the target is external; the `Field` arm accepts a local `@class` table whose `class_name` maps to the external `table_idx`.

The shadow-acceptance rule permissively matches any scope-0 local with the same name — including a truly-local `local X = 5` in a file that also has a workspace-wide `X` — which is desirable for find-references (the user wants to see the collision) but destructive for rename. The `strict_shadow` flag on `references_for_target` filters shadows whose first-version def-node sits inside a `local` statement (detected via `is_local_declaration_site`, which walks up to a `LocalAssignStatement` or a `FunctionDefinition` with a `LocalKeyword` child). The rename handler passes `strict_shadow=true`; find-refs passes `false`.

`include_declaration=false` drops the name-token range inside the first-version def-node for both the local target and any accepted shadow local. `def_name_token_range` translates the statement-level `DefNode` to the name-token range first, since `DefNode` ranges cover whole statements (e.g. the entire `function X() end`).

`textDocument/rename` is built on top of the same helper (prepare_rename + aggregated references with `strict_shadow=true`), so rename is workspace-wide but safer than find-refs against same-named file-locals.

## PreResolvedGlobals::build() phases (in `pre_globals/mod.rs`)
Built once at startup, shared via `Arc` across all files:
1. **Register class names** — Create empty `TableInfo` for each `@class`
2. **Populate @field entries** — Resolve annotation types, add to table fields
3. **Build method functions** — Create `Function` entries for methods, add to tables
4. **Resolve inheritance** — Fixpoint loop copying parent fields to children (handles 5+ levels), then substitutes parameterized class type params (e.g. `@class C<S>` with `@field __super S` → `S` replaced with concrete parent)
5. **Build global functions** — Create `Function` + `Symbol` entries, add to `scope0_symbols`
6. **Register non-class tables** — `math`, `string`, `table`, etc.

## Workspace scanning passes (in `main_loop.rs:scan_paths_with_overrides`)
Run before `PreResolvedGlobals::build()` to collect classes, aliases, and globals from all files:
1. **Pass 1** — `scan_all_annotations()` + `scan_file_globals()` per file: collect `@class`/`@alias` declarations and top-level function/variable globals
2. **Pass 2** — `scan_defclass_calls()` per file: discover classes from `@defclass` factory calls, extract constructor self-fields
3. **Pass 3** — `scan_built_name_calls()` per file: discover `@built-name` classes, merge with `@class` overlays
4. **Pass 4** — `scan_method_typed_self_fields()` per file: scan colon-method bodies for `self.field = expr ---@type Type` assignments on known classes. Captures both preceding-line and inline `---@type` annotations. Per-field file paths stored in `ClassDecl.field_paths` for cross-file definition locations.

## Metatable type inference (`setmetatable` + `__index`)
`setmetatable(tbl, mt)` is detected during Phase 2 resolution via `setmetatable_func_idx` stored on `PreResolvedGlobals`. When detected, `resolve_setmetatable()` in `resolve_call.rs`:
1. Resolves arg[0] (the table) and arg[1] (the metatable)
2. Looks up `__index` on the metatable via `resolve_metatable_index_field()`
3. Mutates the table in-place, setting `metatable_index` to the resolved `__index` target, `metatable` to the raw metatable, and `call_func` from `__call` if present

Field lookups (`get_field` in `mod.rs`) check three levels: direct fields → `parent_classes` → `metatable_index` chain. The `get_field_via_metatable()` helper walks the chain with `HashSet<TableIndex>` cycle detection, supporting chained metatables (e.g. `inst → Child → Base`).

`getmetatable(obj)` is detected via `getmetatable_func_idx` and returns `obj.metatable` (the raw metatable table).

Operator metamethods (`__add`, `__sub`, `__mul`, `__div`, `__mod`, `__pow`, `__concat`, `__unm`, `__len`) are resolved via `resolve_metamethod_return()` in `resolve.rs`. The function checks the table's `metatable` first, then the table itself (for `@class` tables with metamethods as direct fields). The metamethod function's `@return` annotation determines the operator's result type.

Key fields: `TableInfo.metatable_index: Option<TableIndex>`, `TableInfo.metatable: Option<TableIndex>`, `PreResolvedGlobals.setmetatable_func_idx: Option<FunctionIndex>`.

Class name propagation from `setmetatable()` uses three sources (in priority order):
1. `__index` as a direct table reference with `class_name` (e.g. `{ __index = MyClass }`)
2. The metatable itself having `class_name` (e.g. `---@class Foo \n local MT = { __index = function ... }`)
3. `__index` as a function whose return expressions access a class-typed table (e.g. `__index = function(self, key) if METHODS[key] then return METHODS[key] end end` where METHODS has `@class`).

**Limitations**: `setmetatable` mutates the table in-place — this means field assignments on a `setmetatable`-created table after the call ARE visible, but the metatable won't be set on external tables (idx >= EXT_BASE).

## Expression lowering — split identifier nodes (in `lower_expression.rs`)
The parser produces distinct node kinds for identifier access patterns instead of a single `Identifier` catch-all. The `Expression::Identifier` handler dispatches on node kind:
1. **NameRef** → `lower_name_ref()`: simple symbol lookup with type narrowing
2. **DotAccess** → `lower_dot_access()`: lower base expression, create `FieldAccess`
3. **BracketAccess** → `lower_bracket_access()`: lower base and key, create `BracketIndex`
4. **MethodCall** → `lower_method_call_as_callee()`: fully lower the base (including nested calls), then create `FieldAccess` for the method name. This is called when a MethodCall is used as a callee inside `lower_function_call`.

For chained method calls like `obj:A("x"):B("y")`, the parser nests MethodCall nodes. Each level's base is lowered as a complete FunctionCall before the next method name is resolved. Long chains (≥50 links) use `lower_function_call_chain()` for iterative processing to avoid stack overflow.

A legacy 4-way dispatch for old-style flat `Identifier` nodes is retained below the new handlers but is no longer exercised by the current parser.

## Parameterized classes (`@class Name<S>`)
Classes can declare type parameters: `@class BaseClass<S>`. Fields referencing type params (e.g. `@field __super S`) are stored with `annotation_type_raw` and re-resolved during substitution. The substitution chain:
1. A `@defclass T : P` factory declares `@generic T: BaseClass<P>` — binding class type param `S` to function generic `P`
2. At each call site, `P` resolves to the concrete parent class (e.g. `Animal`)
3. Fields with `annotation_type_raw` are re-resolved with `{S → Animal}`, so `__super` becomes `Animal`

Substitution happens in two places:
- **Per-file**: `prescan.rs:substitute_class_type_params()` for local defclass calls
- **Workspace-wide**: `pre_globals.rs` pass 3b for `scan_defclass_calls()`-discovered classes, using `ClassDecl.constraint_type_arg_subs`

## Generic argument inference (call-site `@generic T` binding)
Binding `@generic T` from call-site arguments happens in three layers in `resolve_call.rs` around `resolve_function_call`:

1. **Direct param types** (lines ~1459–1520): if the param's `resolved_type` is `TypeVariable(T)`, bind T to the arg type. If it's `Union(..., TypeVariable(T), ...)` (optional params, or explicit unions), extract the TypeVariable alternative and bind. Strip nil first so optional args don't pollute T.
2. **Structural inference** via `prescan.rs:infer_generics_from_annotation` (called at line ~1524): walks the raw `AnnotationType` to handle:
   - `T[]` — mine T from the arg's array element type
   - `table<K,V>` — mine V from table values, K = string
   - `` `T` `` (backtick) — resolve a string literal arg as a class name
   - `Fun(_, returns, _)` — if a return annotation is `Simple(T)`, extract T from the arg. The arg can be a function (use its first `@return`; fall back to `FunctionRet.resolved_type`, then `type_source`) or a named `@class` table (callable as constructor — T is the class itself). Plain non-class tables are excluded so `{}` literals don't silently bind T.
   - `Union(members)` — recurse into every member (no short-circuit), so multi-generic params like `(fun(): T) | U` can bind T from the Fun member AND U from the Simple member in one pass. Bare `Simple(T)` members bind T directly to the arg type.
   - `Simple(T)` when T is a generic — bind directly.
   - `NonNil(inner)` — recurse.
3. **Receiver `type_args`** (runs BEFORE the per-arg loop): for method calls whose `@param self Class<T>` is `Parameterized`, look up the receiver's `type_args` via `get_expr_type_args` and bind T from there. Runs first so class-generic `T` is bound from the explicit `---@type Class<X>` annotation before direct-arg binding can clobber it with the (rarely useful) arg's runtime type. Receiver-bound generics also join `substitutable_generic_names` so the type-mismatch loop substitutes them.

4. **Variadic generics** (after the per-arg loop): if a generic name starts with `"..."` (e.g. `"...M"` from `@generic T, ...M`), collect the types of all arguments past the positional param count. If any exist, bind the variadic generic to `Intersection(types)` (or the single type if only one). This powers `Mixin(obj, M1, M2, ...)` returning `T & M1 & M2 & ...` with no limit.

**`substitutable_generic_names`** (previously `structural_generic_names`) is the set of generics whose binding is trusted enough to substitute into sibling param types for the type-mismatch check. Populated from structural inference (`T[]`, `table<K,V>`, `fun(): T`), direct-TypeVariable-param inference, receiver-binding, and variadic generics. Explicitly NOT populated from promotional patterns (`` `T` `` backtick, `@defclass T`) where the bound value intentionally differs from the arg.

**`(fun(): T) | T` pre-emption** (lines ~1493–1510): when the raw annotation is a union containing a `Fun(..)` member, run structural inference *before* the eager Union-direct-bind. Otherwise the direct-bind would pick the `TypeVariable(T)` alternative and bind T to the arg itself (e.g. `T = Function(_)` when the user passes a callable), never giving the `Fun` member a chance.

## Function-type projections (`params<F>` / `returns<F>`)
Utility-type projections referencing the shape of a generic `F` bound to a `fun(...)` type. Declared in source as `AnnotationType::Parameterized("params" | "returns", [Simple(name)])` and stored on `Function` as per-slot overlays (`return_projections: HashMap<usize, ProjectionKind>` + `vararg_projection: Option<ProjectionKind>`), NOT as new `ValueType` variants. The `ProjectionKind` enum (`src/types.rs`) has `Params(String)` and `Return(String)` variants naming the referenced generic.

**Validation** at `prescan.rs::check_annotation_type_names` in the `Parameterized(base, args)` arm: `base == "params" || base == "returns"` requires exactly 1 arg of `Simple(name)` where name is a declared `@generic`. Violations emit `malformed-annotation`. `params<F>` outside the vararg slot (positional `@param x params<F>`, or `@return params<F>`) emits `malformed-annotation` during `insert_function_definition`. Nested projections (`returns<returns<F>>`) fail the `Simple` shape check.

**Population** (`build_ir.rs::insert_function_definition`):
- In the `@param ...` vararg branch, `match_projection` detects `params<F>` / `returns<F>` and sets `func.vararg_projection`.
- In the `@return` loop (legacy multi-line branch), each return slot that matches `returns<F>` gets `func.return_projections.insert(ret_index, Return(name))`.

**Resolver-level placeholder** (`prescan.rs::resolve_annotation_type_mut_gen`): when resolving a projection annotation with F still bound as an unresolved generic, returns `ValueType::Any` so the return/vararg slot exists in the IR. Call-site resolution replaces it with F's concrete type.

**Call-site resolution** (`resolve_call.rs::resolve_function_call`):
- `projected_f_idx` is computed early (before the per-arg loop) by looking up F from the receiver's type_args. Used by the arity check AND the per-arg type-mismatch loop.
- Arity check: when `projected_arity` is non-None, `expected_count = non_vararg_count + F.args.len()`; `effective_is_vararg = false`. Missing-param name uses F's arg name at the out-of-range position.
- Type-mismatch loop: for vararg positions (`i >= non_vararg_count`), pull expected type from `F.args[i - non_vararg_count].resolved_type`.
- Return resolution: when `return_projections[ret_index]` is `Return(name)` and `generic_subs[name]` is `Function(Some(f_idx))`, return `f.return_annotations[0]`. If F has multiple return annotations OR the function has tuple-union overloads, emit `multi-return-projection` warning (column 0 is still picked).

**Diagnostics**:
- `malformed-annotation` — shape errors (wrong arity, wrong arg kind, wrong position, nested projection, unknown generic).
- `multi-return-projection` (WARNING, `src/diagnostics/multi_return_projection.rs`) — `returns<F>` truncates when F has >1 return annotation. Suppressible via `@diagnostic disable:multi-return-projection`.

**Hover** (`queries.rs::format_function_decl`): class-declaration hover shows the raw `params<F>` / `returns<F>` via the existing `param_annotation_text` path (no special expansion). Call-site hover on the receiver's call expression already reflects the bound F's concrete return type via the normal resolve path. Signature help at call sites shows `func: F` unsubstituted — further expansion is a v2 enhancement.

## Carrying `type_args` from parameterized return types (`@return Pool<T>`)
When a generic function's return annotation is `Parameterized("Pool", [Simple("T")])`, the call's inferred T has to survive through the assignment so that subsequent method calls on the receiver (e.g. `pool:Get()`) can bind T from the receiver's type_args.

`ValueType::Table(Option<TableIndex>)` doesn't carry type_args, so we keep them outside the value:
- `Function.return_annotations_raw: Vec<AnnotationType>` — preserves the raw `Parameterized(..)` structure alongside the resolved `return_annotations: Vec<ValueType>` (populated in `build_ir.rs`, `prescan.rs`, and `pre_globals.rs`; `#[serde(default)]` for backward compatibility).
- `Analysis.call_type_args: HashMap<ExprId, Vec<ValueType>>` — per-call cache of substituted type_args. Populated in `resolve_function_call` whenever `generic_subs` is non-empty and the raw first-return annotation is `Parameterized`. The type_args are resolved using the function's own `generic_constraints_raw` so that `Simple("T")` becomes `TypeVariable("T")`, then `substitute_generics_deep` substitutes to concrete types.

`get_expr_type_args` (in `resolve_call.rs`) checks this cache:
1. Direct cache hit for the ExprId (covers `FunctionCall` receivers)
2. `SymbolRef(sym, ver)` — first check the version's `type_args` (set by `---@type Pool<Concrete>` in build_ir), then follow `type_source` ExprId into the cache
3. `FieldAccess { table, field }` — check the field's `annotation_type_raw`, then the field's stored `expr` in the cache (covers `private = { pool = New(...) }` table-field patterns)

Bump `pre_globals.rs::BLOB_VERSION` when changing any field on a serialized type (`Function`, `ClassDecl`, etc.).

## Builder pattern (`@builds-field` + `@return built`)
Builder methods use `@builds-field <param_idx> <type>` with `@return self` to progressively add typed fields to a shadow `built_table` on `TableInfo`. `@return built [: Parent]` returns the accumulated type.

Resolution in `resolve_call.rs`:
- **`@builds-field` + `@return self`**: `clone_table_with_built_field()` clones the receiver table with an updated `built_table` containing the new field. Each chained call produces a new table clone.
- **`@return built`**: Returns the `built_table` from the receiver. If `@return built : Parent` is specified, the parent class is added to the built table's `parent_classes`.

Key fields: `Function.builds_field: Option<(usize, ValueType, bool)>` (param_index, resolved_type, lateinit), `Function.built_name: Option<usize>`, `Function.built_extends: bool`, `Function.returns_built: bool`, `Function.returns_built_parent: Option<String>`, `TableInfo.built_table: Option<TableIndex>`.

The type in `@builds-field` supports `T!` (NonNil/lateinit): `@builds-field 1 T!` creates fields with `FieldInfo.lateinit = true`, allowing nil assignment without `field-type-mismatch`. The `!` is detected at three sites: `build_ir.rs` (per-file), `pre_globals.rs` build function resolution (cross-file `ClassDecl.fields`), and `pre_globals.rs` `build_on_stubs` (workspace overlay).

### Naming built types (`@built-name`)
`@built-name <param_idx>` on the chain entry point function sets the `built_table`'s `class_name` from the string literal at parameter `param_idx`. This allows the built type to be referenced by name in `@param`/`@type` annotations.

Resolution in `resolve_call.rs`:
- `clone_table_with_built_name()` creates a built table with the specified class name and registers it in `ir.classes`
- Subsequent `clone_table_with_built_field()` calls preserve the name and re-register the latest built table in `ir.classes`
- A post-fixpoint step re-resolves param annotations that reference newly discovered `@built-name` classes

Cross-file visibility: `scan_built_name_calls()` in `annotations.rs` scans workspace files for calls to `@built-name` functions, extracting class names and registering them as empty `ClassDecl` entries in `PreResolvedGlobals`.

### `@class` overlays on `@built-name` types
A `@class Foo` declaration that re-uses a name already created via `@built-name` merges its `@field` annotations with the builder-pattern fields. Overlay `@field` types take precedence over built field types for matching names. The overlay must be standalone (not directly preceding a `local` statement, which would be interpreted as typing the variable).

Resolution happens at three levels:
- **Per-file** (`resolve_call.rs`): `clone_table_with_built_name()` checks `ir.classes` for a local `@class` table with the same name and merges its `@field` annotations (identified by `annotation_type_raw.is_some()`) into the built table. `clone_table_with_built_field()` skips overwriting fields that have `annotation_type_raw` (from overlays).
- **Per-file prescan** (`prescan.rs`): After populating local class fields, external class fields (from `ext.classes`) are imported into local `@class` overlay tables for any matching names.
- **Workspace** (`main_loop.rs`): When merging `ws_file_defclasses` (from `scan_built_name_calls()`) into `ws_classes`, built-name fields are merged into existing `@class` entries instead of being skipped.

### Extending builder schemas (`@built-extends`)
`@built-extends` on a method (combined with `@built-name`) makes the new built type inherit from the receiver's current built type instead of copying its fields. This supports schema extension patterns where a base schema is defined and subclasses extend it:

```lua
---@param name string
---@built-name 1
---@built-extends
---@return self
function Schema:Extend(name)
    return self
end

local BASE = Schema:AddString("baseName"):Commit()
local CHILD = BASE:Extend("ChildState"):AddString("childField"):Commit()
-- ChildState inherits baseName from BASE's built type
```

Resolution in `resolve_call.rs`:
- `clone_table_with_built_name()` with `extends=true` creates a new built table whose `parent_classes` include the receiver's existing built table plus all its ancestors (flattened for single-level FieldAccess resolution)
- Subsequent `clone_table_with_built_field()` calls preserve the parent chain, so fields added after `Extend` still inherit from the base
- Multi-level extension works: grandchild → child → base, with all ancestor fields accessible

## Variadic generics (`@generic T, ...M`)
A variadic generic parameter (prefixed with `...`) collects excess positional arguments into an intersection type. Stored as `("...M", None)` in the existing `generics: Vec<(String, Option<String>)>` — no new data structures.

**Parsing**: `...M` in type positions parses as `VarArgs(Simple("M"))` via existing `parse_type()`. In `resolve_annotation_type` (annotation_scanning.rs), the `VarArgs` arm checks if `"...{name}"` is in the generics list → returns `TypeVariable("...M")`.

**Binding** (resolve_call.rs `infer_generic_subs`): after the per-arg binding loop, finds any generic name starting with `"..."`. Collects types of all arguments beyond the positional param count. Binds to `Intersection(types)` if multiple, or the single type if one.

**Substitution** (resolve_call.rs `substitute_generics_deep`): the `Intersection` arm flattens nested intersections after substitution (when `...M` substitutes to `Intersection(A, B)` inside `T & ...M`, the result is `T & A & B` not `T & Intersection(A, B)`). Unbound variadic generics (`TypeVariable("...M")`) are filtered out by the existing TypeVariable filter.

## Derived mixin classes (`Derived = CreateFromMixins(Base, …)`)
The variadic-generic return above types a *bare* `CreateFromMixins(Base)` call as `Base` (the intersection of its args). But the standard WoW idiom assigns that to a global and defines methods on it — `Derived = CreateFromMixins(Base)` then `function Derived:Method()` — so `Derived` needs to be its **own** class that inherits `Base`, with `self` resolving to `Derived` (not the bare base) and the XML template's `parentKey` children (bound via `mixin="Derived"`) landing on it. Three coordinated pieces achieve this:

1. **Cross-file detection** (`scan_globals.rs::extract_mixin_parents`): a single-name global assigned `CreateFromMixins(A, B, …)` records the identifier args on `ExternalGlobal.mixin_parents` (`#[serde(skip)]`, workspace-only; compared in `rebuild.rs::global_semantic_eq`). The coarse scan already auto-creates a class table for `Derived` because it has methods.

2. **Inheritance + open-mixin marking** (`pre_globals/{mod,build_on_stubs}.rs::apply_mixin_parent_inheritance`): runs *after* `resolve_inheritance` (so each parent's transitive closure is ready) and links each `Derived` class to its mixin parents. Unlike `@class` parents, a mixin base may be a **non-class table** (`Base = {}` with methods, never used as an XML `mixin=`), so parents resolve via *both* `classes` and `non_class_tables` — the missing case that made `resolve_inheritance` (which only links `classes`) leave Auctionator-style bases unlinked. A fixpoint loop propagates ancestors across mixin chains (`Derived → Mid → Base`). Every derived class is also flagged `TableInfo.open_mixin = true`.

3. **Per-file re-anchoring** (`build_ir.rs`, simple-assignment path): when `Derived = CreateFromMixins(…)` and `Derived` is a known external class (`Ir::external_class_table_for`) with no overriding `@class`/`@type`, `self`'s root symbol is re-anchored from the CreateFromMixins return (the base) to `Literal(Table(Derived))`. Methods then register/resolve against `Derived`, which inherits the base and carries the XML augment's `parentKey`/`parentArray`/`KeyValue` fields.

**Permissiveness — the `open_mixin` flag.** A mixin instance receives fields dynamically (XML `parentKey` children attached at frame creation, `self.x = …` runtime writes in inherited methods), exactly the rationale that makes `undefined-field` already skip top-level `Frame & Template` intersections. So `undefined_field.rs` skips any access whose base chain is rooted at an `open_mixin` class (or one inheriting it) via `base_chain_is_open_mixin` — which walks `FieldAccess`/`Grouped` bases to the root, covering nested chains (`self.A.B.C:Method()`). The same flag makes an `open_mixin` value permissively assignable in `is_table_subtype_impl` (so `Base.Init(self)` passing the derived to the base's inferred-self shape doesn't false-positive). **Known limitation**: `self:GetParent().X` stays flagged — a method call breaks the field chain and the parent frame's runtime type is statically unknowable.

## `@narrows-arg N` (in-place argument type mutation)
`@narrows-arg N` on a function means: when the function is called as a bare statement (not assigned), narrow the Nth argument's type to the function's return type. Mirrors the `@cast` mechanism.

**Data**: `narrows_arg: Option<usize>` on both `AnnotationBlock` (annotations/mod.rs) and `Function` (types.rs). 1-based index. Stored on `ExternalGlobal` for workspace scanning.

**Phase 1** (build_ir.rs `try_narrows_arg`): after `Statement::FunctionCall`, looks up the callee in external globals, checks `narrows_arg`. The narrowed argument is one of two shapes:

- **Plain local** (`Mixin(f, M)`, arg is a single name): creates a new `SymbolVersion` on `f` with `type_source = call_expr_id`. During Phase 2 the call resolves to the return type (with generics substituted), which becomes `f`'s new type.
- **Field target** (`Mixin(self.Child, M)` / `Mixin(tbl.field, M)`, arg is a 2-name chain): a field has no symbol versions, and the receiver table (`self`) is only known in Phase 2, so a `DeferredFieldMixin { root_name, field_name, scope_idx, mixin_exprs }` is recorded instead. `mixin_exprs` is every *lowered* call argument except the narrowed target. After the fixpoint, `resolve.rs::resolve_deferred_field_mixins` resolves the receiver table (mirroring `resolve_deferred_field_assignments`), resolves the field's current base type and each mixin, and stores `base & mixin1 & …` as the field's `annotation` — so the mixin's members resolve on every read of the field, including from sibling methods (a scope-local flow narrowing can't reach those). Reads cached against the pre-mixin base during the fixpoint are invalidated targetedly (not a full cache wipe, so builder-chain results the read-only resolver relies on the cache for are left intact): the seed is `FieldAccess` exprs on the augmented field, matched on the **resolved receiver table** (not just the field name — so a same-named field on an unrelated table is spared), then transitively their wrappers (`AssignNarrow`/`StripNil`/casts) and `SymbolRef` reads of locals whose `type_source` is itself stale. A local that *copied* the field (`local c = self.Child`) also has its version's `resolved_type` cleared (it was set to the pre-mixin base during the fixpoint), so `reresolve_unresolved_after_deferred` — which only revisits symbols whose type stayed unknown — recomputes it against the intersection. The base assignment doesn't fire `field-type-mismatch` against the fuller intersection — a deferred assignment has `had_annotation_at_build = false`, and an eager scanned field is skipped by `field_type_mismatch.rs` when the expected type is an `Intersection`, the field is `from_scan`, and the assigned value matches one member.

The field path augments the per-file IR only; it does **not** register the field cross-file, so reads in *other* files still see the bare base type (the same limitation that already applies to addon-ns mixin self-fields).

**Limitation**: `try_narrows_arg` only handles single-name external callees (e.g. `Mixin`). Namespace-qualified calls (`ns.Mixin`) and user-defined local functions are not supported. This suffices for WoW's built-in `Mixin()` function.

**Validation** (malformed_annotation.rs): `@narrows-arg` requires a numeric argument >= 1.

## Tuple-union `@return` syntax
`@return (A name, B) | (C, D) desc` lowers to a `Function` with:
- `return_annotations` = per-position column union (each position's type = union of that position across all cases)
- `return_annotations_raw` = per-position raw `AnnotationType` column (often a `Union`)
- `return_labels` = names from the first tuple's positions (parallel to column types)
- `overloads` += one `ResolvedOverload { is_return_only: true, description }` per case

Parser: `AnnotationType::Tuple(Vec<TuplePosition>, Option<String>)` represents a single tuple with optional per-case description. A union of tuples is the correlated form. `parse_return_line()` in `annotations.rs` is the line-level entry point — it detects the tuple form by `s.starts_with('(')` + top-level comma inside, then captures any trailing text after `)` as the case description (with optional `@` prefix). `---|` continuation lines merge into the last `@return` line when that line parsed as tuple-form; otherwise they extend an active `@alias`. Tuple-form aliases (`@alias Foo (A) | (B)`) are stored in `IrBuilder.tuple_form_aliases` / `PreResolvedGlobals.tuple_form_aliases` rather than `aliases` (tuples have no single `ValueType`); use-site expansion via `expand_tuple_form_alias()` happens in `build_ir.rs`'s `@return` processing.

The actual lowering work (cases → column-union `ValueType`/raw + labels + return-only `ResolvedOverload`s) lives in the shared `lower_tuple_form_cases()` helper in `annotations.rs`, which takes a resolver closure so every caller can plug in its own type-resolution context. Four call sites invoke it: `build_ir.rs::insert_function_definition` (local functions), `prescan.rs::materialize_fun_type` (inline `fun()` types), `pre_globals.rs::build_function` (external stubs + cross-file workspace scan), and `pre_globals.rs::materialize_fun_type` (stub `fun()` types). Each site is responsible only for the surrounding scope setup — symbol insertion for return slots, overload merging, and IR-specific index adjustments.

Legacy `@return T name` parsing is still accepted; mixing it with a tuple-form `@return` on the same function emits `malformed-annotation`. `@overload return:` parsing was removed.

Single-position tuple cases (`(T)`) are supported in two contexts: (a) `---|` continuation lines, where we know we're extending a tuple union — controlled by the `force_tuple` flag on `parse_return_line`; and (b) base `@return (T)` lines with *no* trailing text (the trailing check preserves the legacy `@return (string|number) name` form, which would otherwise lose its name to case-description parsing). Outside `@return`, `(T)` still parses as plain grouping in `parse_type`.

Arity mismatch across tuple-union cases is allowed. Shorter cases are implicitly nil-padded at missing positions — `lower_tuple_form_cases` uses `max` arity across cases (not first-case arity), computes per-column types by padding with `nil`, and picks labels from the first case that provides a name at each column. Each case's return-only overload keeps its declared arity (no padding); the overload-narrowing lookup (`resolve_overload_narrow`) and `return_overload_may_nil` handle out-of-bounds positions by returning implicit nil, or by falling through to the last declared position's type when `ResolvedOverload.has_vararg_tail` is set (case ended in `...T`). Tuple cases with a `...T` tail also set `Function.has_vararg_return`, same as legacy `@return ...T` — this is detected in all three lowering paths (`build_ir.rs`, `prescan.rs::materialize_fun_type`, `pre_globals.rs::build_function`).

Hover rendering (`queries.rs::format_function_decl`) shows labels inline (`-> name: type, level: type`) and renders return-only overloads as a `cases:` table under the primary signature instead of stacking each as a separate `function name()` block.

**Narrowing implementation** (unchanged from the old `@overload return:`): `multi_return_siblings` in `Analysis` tracks which symbols came from the same function call. `narrow_siblings()` in `narrowing.rs` hooks into all narrowing points (`analyze_nil_guard`, `analyze_early_exit_guard`, assert narrowing). It checks `check_return_only_overloads_from_siblings()` to only activate for functions with `is_return_only` overloads. Return-only overloads are filtered out of arg-count matching in `resolve_call.rs`.

**Overload-based narrowing**: When a sibling is narrowed, `narrow_siblings()` creates `Expr::OverloadNarrow` versions for ALL other siblings. The OverloadNarrow stores `(ret_index, func_expr, narrowed)` where `narrowed` is a list of `(sibling_ret_index, NarrowKind)` entries for each directly-guarded sibling. `NarrowKind` has four variants: `StripNil` (e.g. `x ~= nil`), `StripFalsy` (e.g. `if x then`), `StripTruthy` (e.g. `if not x then` or `else` of `if x then`), and `ClassEq(String)` (e.g. `if x == Foo.MEMBER then` where `Foo.MEMBER` is class-typed). During resolve, `resolve_overload_narrow()` filters return-only overloads whose type at each narrowed sibling's position is compatible with the `NarrowKind` (`overload_type_survives_{strip_nil,strip_falsy,strip_truthy}` / `overload_type_matches_class`), then computes the union of types at `ret_index` across compatible overloads. Overload-narrowed siblings are NOT added to `narrowed` to avoid double-stripping nil in `narrow_type_for_display`. For cross-file calls (deferred case), the narrowed_info is stored in `deferred_sibling_narrowings` and processed during the resolve fixpoint loop. `push_overload_narrow_version()` uses `version_for_scope_ancestors_only` for the base version so that a narrowing created inside a sibling branch scope can't become the base of an outer-scope narrowing.

**Class-equality narrowing (`x == Foo.MEMBER`)**: Detected during `analyze_nil_guard` by `record_class_eq_deferral()` when the non-symbol side is a pure identifier chain (a bare `Identifier` / `DotAccess` — see *Expression lowering — split identifier nodes* above). The RHS is lowered and queued in `deferred_class_eq_narrowings` as `(sym_idx, expr_id, scope_idx)`. At resolve, `resolve_deferred_class_eq_narrowings()` resolves the RHS: if its type is (or contains) a class table with a `class_name`, it sets `class_narrowed_symbols[scope][sym] = class_name`, inserts into `type_filtered_symbols` (for symbol-level display), pushes a `TypeFilter` version, and propagates to multi-return siblings via `push_overload_narrow_version` with `NarrowKind::ClassEq(name)`. To reach references lowered before resolve, `rewrite_sym_refs_in_subtree()` walks `sym_ref_sites` (a reverse index `SymbolIndex → Vec<(ExprId, token_offset)>` populated at build time) and redirects `Expr::SymbolRef` exprs in the affected scope subtree to the new version, updates `symbol_version_at`, invalidates the `resolved_expr_cache`, and prunes stale `need-check-nil` / `type-mismatch` diagnostics emitted pre-narrowing. Restricting detection to pure identifier chains avoids re-lowering embedded name references (e.g. `name` inside `strlower(name)`) which would clobber the original `symbol_version_at` entries.

**Narrowing tracking maps (convention)**: The narrowing maps live in the `NarrowingState` struct (`analysis/mod.rs`), embedded in both `Analysis` (mutable phase) and `AnalysisResult` (query/diagnostic phase) and accessed through thin forwarder methods. Each map's name describes what the guard *stripped* to produce the narrowing, not what the value is: `narrowed` = nil stripped; `falsy_narrowed` = nil AND false stripped (a subset of `narrowed`); `truthy_narrowed_symbols` = truthy stripped, so the value is `nil | false`; `class_narrowed_symbols` = equated to a class (value IS that class). So "truthy_narrowed" reads as "truthy-ness stripped" → value is falsy. The `narrowed`, `falsy_narrowed`, `type_narrowed`, and `type_stripped` maps are keyed by `NarrowTarget` (an enum with `Symbol(SymbolIndex)` and `Field(SymbolIndex, Vec<String>)` variants), so a single map covers both bare symbols and field chains; `NarrowTarget`'s `Eq`/`Hash` gives exact-match semantics for free (narrowing `a.b` never matches `a.b.c`). The remaining maps (`truthy_narrowed_symbols`, `class_narrowed_symbols`, `num_compare_narrowed_symbols`, `type_filtered_symbols`) stay symbol-only.

**Temporary insert/restore protocol for `and`/`or` RHS**: `analyze_nil_guard` inserts into the tracking maps at a branch scope (then/else), so entries disappear naturally once the branch scope is out of view. The short-circuit `and`/`or` lowering in `lower_expression.rs::BinaryExpression` operates in the *same* scope as the containing expression, so it uses a temporary-insert-then-remove protocol: (1) record what was inserted (`Vec<(SymbolIndex, bool, bool)>` flags whether each map actually took a new entry); (2) call whatever consumes the maps (`narrow_siblings`, etc.); (3) after RHS lowering, remove *only* the entries you added. Sibling `OverloadNarrow` versions pushed during this window are also scope-persistent — pair each narrow call with a pre-narrow version snapshot, then call `ir.push_alias_version(sym, pre_ver, scope)` at teardown to revert the symbol's current version to the pre-`and` state. Any future code that inserts into these maps mid-RHS must follow the same pattern or the cleanup will under- or over-remove.

**Callee enforcement**: The `grouped-return-mismatch` diagnostic (deferred check in `checks.rs`) verifies that each `return` statement matches one of the return-only overloads. The `missing-return-value` diagnostic is suppressed for functions with a nil return-only overload.

## Literal boolean return type union discrimination
When a union type `A | B` has a method where `A:Method()` is annotated `@return false` and `B:Method()` is annotated `@return true`, the LS automatically narrows the union in conditional branches — then-branch keeps the `true`-returning types, else-branch keeps the `false`-returning types.

**Implementation** (`narrowing.rs`):
- `resolve_expr_to_tables()` — like `resolve_expr_to_table()` but returns ALL table indices from a union type
- `extract_bool_discriminator()` — given a method call on a union receiver, checks if all union member tables define the method with complementary literal boolean `@return` annotations. Returns `(sym_idx, true_type, false_type)`.
- Integrated into `analyze_nil_guard` (then + else branches), `analyze_early_exit_guard`, and `narrow_assert_expr`

**Conditions**: all union members must define the method, every return annotation must be literal `true` or `false` (not generic `boolean`), and at least one of each must exist. Works with 3+ member unions.

## Correlated nil fields (`@correlated`)
`@correlated field1, field2, ...` on a `@class` declares that listed optional fields are always nil/non-nil together. Stored as `correlated_groups: Vec<Vec<String>>` on `TableInfo`. Multiple `@correlated` lines per class create independent groups. Groups are inherited by child classes during prescan pass 3.

**Narrowing**: In `try_narrow_field()` and `try_narrow_field_falsy()` (narrowing.rs), after inserting the primary narrowing, `narrow_correlated_fields()` resolves the field's table via `resolve_field_chain_table()`, looks up its `correlated_groups`, and inserts sibling narrowings into `narrowed` (and `falsy_narrowed` if applicable) as `NarrowTarget::Field` entries. Works for both `self.field` (chain len 1) and `self.sub.field` (chain len 2+) patterns, and with early-exit narrowing.

## Correlated locals (inferred from if/elseif branches)
When multiple local variables are assigned in every explicit branch of an if/elseif chain (without else), they form a correlation group. Stored as `correlated_locals: Vec<Vec<SymbolIndex>>` on `Analysis`. When one member is narrowed via a nil guard or early-exit guard, all siblings in the group are narrowed too.

**Detection**: In the `PendingBranchMerge` processing (build_ir.rs), after collecting symbols assigned in branch scopes, symbols that are assigned (not just narrowed) in ALL explicit branches of a `has_implicit_else=true` merge are collected into a correlation group.

**Narrowing**: `narrow_correlated_locals()` in narrowing.rs is called from `narrow_symbol_strip_nil()`, `narrow_symbol_strip_falsy()`, and direct narrowing insertion points in `analyze_nil_guard()`, `analyze_early_exit_guard()`, and `narrow_assert_expr()`. It looks up the symbol in `correlated_locals` groups and inserts sibling narrowings into `narrowed` (and `falsy_narrowed` if applicable) as `NarrowTarget::Symbol` entries.

## Guard implications (`if A and not B then return`)
An early-return guard `if A1 and ... and An and not B then return end` establishes, for code reached past it, the implication `(A1 and ... and An) ⟹ B is truthy` (the negation of the guard condition is `not A1 or ... or B`; `not B` being false means B is truthy). Stored as `guard_implications: Vec<(Vec<SymbolIndex>, SymbolIndex, ScopeIndex)>` on `Analysis` (antecedents, single consequent, the scope the guard lives in).

**Detection**: `detect_guard_implications()` in narrowing.rs runs from build_ir.rs alongside `detect_complementary_exit_guards`, for every exiting branch of the early-exit prefix. It reuses `extract_and_truthiness_shape()` and records an entry only when there is ≥1 truthy antecedent and exactly one negated term (multiple negated terms give `B1 or B2 or ...`, which pins down no single consequent). Duplicate entries are avoided by checking existing implications before pushing.

**Application**: `apply_guard_implications()` fires from `narrow_symbol_strip_falsy()`, the bare-identifier then-branch insert in `analyze_nil_guard_inner()`, and `narrow_or_coalesce_derived()` (falsy branch). When a symbol that is an antecedent gets narrowed truthy in a scope, and (a) that scope is within the guard's scope subtree (`is_ancestor_scope`, guarding against guards nested in conditional branches leaking out) and (b) every antecedent is now `falsy_narrowed` in that scope, the consequent is `narrow_symbol_strip_falsy`-ed. The consequent is claimed truthy (falsy-stripped), not merely non-nil: the guard `if A and not B then return` means `not B` was false past the guard, i.e. B is truthy.

**Invalidation**: `invalidate_guard_implications()` runs at both reassignment sites in build_ir.rs (alongside `invalidate_correlated_locals`), dropping any implication that mentions the reassigned symbol as antecedent or consequent.

## `x = x or y` coalesce narrowing
The idiom `x = x or y` makes `x` non-nil whenever `y` is non-nil: either the old `x` was truthy (kept) or `y` is used (and `y` non-nil means the result is non-nil). Narrowing is one-directional — narrowing `y` narrows `x`, but narrowing `x` tells you nothing about `y`. Stored as `or_coalesce_derivations: HashMap<SymbolIndex, Vec<SymbolIndex>>` (source `y` → derived `x`s).

**Detection**: `maybe_register_or_coalesce()` runs at every simple-name assignment `x = expr` in `narrowing.rs`. When `expr` is `BinaryOp(Or, NameRef(x), NameRef(y))` and both sides resolve to symbols (with the LHS matching the target), it registers `(y, x)`. Any other assignment to `x` invalidates prior `(*, x)` entries — the coalesce relationship is tied to this specific assignment.

**Narrowing**: Propagated from `narrow_symbol_strip_nil()` / `narrow_symbol_strip_falsy()` via `narrow_or_coalesce_derived()`, from the direct-insert narrowing sites in `analyze_nil_guard_inner` (then-branch `if x then` / `if x ~= nil then` / `if type(x) ~= "nil" then` / `if type(x) == "T" then`) and `narrow_assert_expr` (`assert(x ~= nil)`, `assert(type(x) ...)`), and from the temporary `and`/`or`-guard narrowings in `lower_expression.rs`'s `BinaryExpression` arm (lines near `coalesce_pre_narrow`). Guard-path propagation pushes a transient StripNil/StripFalsy version on each derived symbol alongside the primary/extra guard narrowings, then restores them in the same reverse-order pass that restores the primary guard.

## Flavor filtering (`flavors` config + TOC detection + `@flavor-narrows` + `wrong-flavor-api`)
Projects declare target WoW flavors in `.wowluarc.json` via `flavors: [...]` (accepting `retail`, `classic`, `classic_era` — the three WoW install-folder names), or flavors are auto-detected from TOC file listings. Each `Function` carries a `flavors: u8` (the 3-bit mask `crate::flavor`) and a `flavor_guard: u8` (from the `@flavor-narrows` annotation). `Symbol` and `FieldInfo` also carry `flavor_guard: u8` for boolean variable/field guards.

Stub gen: `src/stub_gen/` reads Ketho's `flavor.ts` (4-bit mainline/mists/bcc/classic_era mask) and passes each entry through `crate::flavor::from_ketho_mask` to collapse mists+bcc into our `classic` bit. `stub_gen::classic::apply_flavor_data` writes the translated mask into each matching `ExternalGlobal.flavors`, and `PreResolvedGlobals::build_function` pipes it through to `Function.flavors`.

Narrowing: `Analysis` carries `project_flavors: u8` and `scope_flavors: HashMap<ScopeIndex, u8>`. `try_flavor_narrow()` in narrowing.rs detects three patterns: `WOW_PROJECT_ID == WOW_PROJECT_*` comparisons, `@flavor-narrows` guard function calls (`flavor_guard_mask_for_call`), and `@flavor-narrows` boolean variables/fields (`flavor_guard_mask_for_ident`). All three call `narrow_scope_flavors()` or `exclude_scope_flavors()` on the target scope. `active_flavors_at(scope)` walks ancestor scopes to the first explicit override; if none, returns `project_flavors`.

Because annotation guards on local functions aren't typed at build-ir time, `flavor_guard_mask_for_call` uses `find_function_def(type_source)` to walk the symbol's `type_source` to a `FunctionDef` directly (bypassing `resolved_type`, which is only populated in Phase 2). Boolean guards (`flavor_guard_mask_for_ident`) check `Symbol.flavor_guard` for single-name identifiers and `FieldInfo.flavor_guard` for dotted field access.

Short-circuit `and` narrowing: `collect_and_chain_flavor_guards()` and `detect_and_lhs_flavor_guard_leaf()` in narrowing.rs detect `@flavor-narrows` guards in the LHS of `and` expressions. During expression lowering (`lower_expression.rs`), all `FunctionCall` exprs in the RHS range are recorded in `Ir.and_guarded_flavor_exprs: HashMap<ExprId, u8>` with the effective narrowed flavor mask. The `wrong-flavor-api` diagnostic checks this map before falling back to `active_flavors_at(scope_idx)`.

TOC-based detection: `parse_toc_files()` in `config.rs` reads all `.toc` files in a directory and extracts per-file flavor masks. Three mechanisms are supported: (1) TOC filename suffixes (`_Mainline`, `_Classic`, `_Vanilla`, `_Cata`, `_Wrath`, `_TBC`, `_Mists`) mapped via `flavor::parse_toc_suffix()`; (2) `## AllowLoadGameType:` header restricting the entire TOC; (3) `[AllowLoadGameType ...]` per-line directives restricting individual files. The unsuffixed (base) TOC covers flavors not claimed by any suffixed TOC. Per-file flavor masks are stored in `ProjectConfigs.toc_file_flavors` and intersected with project-level `flavors` in `flavors_for()`. `FLAVOR_ALL` entries are pruned (no restriction needed). Game type names (`mainline`, `vanilla`, `cata`, etc.) are parsed via `flavor::parse_game_type_name()`, which is more permissive than `parse_flavor_name()`.

Path variables `[Family]` and `[Game]` in TOC file listings are expanded via `expand_toc_path_variables()`. Each expansion value maps to a flavor (e.g. `[Game]` → `Standard`=retail, `Vanilla`=classic_era, `Cata`=classic). Only expansions whose file exists on disk are included. The expansion tables are `flavor::FAMILY_EXPANSIONS` and `flavor::GAME_EXPANSIONS`.

Diagnostic: `wrong_flavor_api.rs` emits `wrong-flavor-api` at the call site when `unsupported_flavors(active, call.flavors)` is non-zero. Fires only when `project_flavors != 0` and the function has non-zero `flavors` (a mask of 0 is treated as "available everywhere").

## DefNode (source location pointers)
Symbol and function definitions store `DefNode { start: u32, end: u32 }` — a simple byte range with no dependency on the syntax tree. External symbols use `DefNode::DUMMY`. `definition_at()` returns `DefinitionResult::External(loc)` for external symbols instead of trying to look up the node.

## `self` parameter handling (cross-cutting invariant)
A parameter named `self` can be **implicit** (colon syntax: `function Foo:bar(x)` → parser sees `[x]`, self injected by `insert_function_definition`) or **explicit** (dot/global: `function handler(self, index)` → parser sees `[self, index]`). Three code paths must agree on this distinction:
1. **Stub scanning** (`annotations.rs:scan_file_globals`) — Only filter `self` from unannotated param lists when `is_call_to_self()` (colon syntax). Global functions with explicit `self` must keep it.
2. **Function building** (`build_ir.rs:insert_function_definition`) — `inject_self` adds a synthetic self param for colon-defined methods.
3. **Call-site `self_offset`** (`resolve_call.rs`) — Offset by 1 when `is_method_call` (colon call) AND the function has any first param (whether named `self` or not, including stored function fields). Plain calls pass all args explicitly, so offset must be 0 regardless of the param name.

## Backward param-type inference
`Analysis::infer_backward_param_types()` in `resolve_call.rs` sets `resolved_type` on unannotated local function parameters based on how they're used in the body. Runs inside the fixpoint loop's fallback branch (same branch that handles `@built-name` late resolution), gated by the `backward_param_types` flag (Analysis field, populated from `inference.backward_param_types` in `.wowluarc.json`; default `true`).

**Hints are treated as upper bounds and intersected.** Each use site implies a constraint that the param value must be assignable to; the inferred type is the narrowest type satisfying every constraint (`intersect_hints`/`intersect_pair` in `resolve_call.rs`). Empty intersection (genuinely conflicting constraints) leaves the param untyped. A hint of `any` causes the pass to bail for that param — `any` is a no-information constraint that shouldn't combine with specific hints, and loose stub annotations like `tostring(v: any)` would otherwise coerce real hints away.

Hints are split into **baseline** and **narrowing**. Baseline hints alone drive inference; narrowing hints only tighten existing baseline hints. Narrowing never strips nil from a baseline that explicitly allowed it (every baseline hint contained nil) — the `?` on `@param a? T` is user intent, and a conditional use inside the body reflects a user-maintained invariant the LS can't verify. If a narrowing hint contradicts the baseline intersection entirely, the baseline-only intersection is used instead so a weak signal can't block inference.

Baseline hints:
- Arithmetic `param + n` / `param * n` / `-param` when the other side resolves to `number` → `number`
- Concatenation `param .. x` / `x .. param` when the other side `can_concat_to_string()` → `string | number`
- Passed as arg to a function whose corresponding param has a non-vararg annotation → that annotation's type (respects `self_offset` for colon calls). When the primary signature's arity doesn't match the call, a unique-arity overload is used instead; hints containing a `TypeVariable` (generic) are skipped.
- Passed as arg to a function whose corresponding param has no annotation but a `resolved_type` from a prior inner-iteration of this pass → that inferred type. This lets `outer(y) → inner(y)` inherit `inner`'s backward-inferred type across iterations. Only applies when there's no annotation (otherwise `resolved_type` would already reflect the annotation).

Narrowing-only hints (tighten but don't create inference):
- Passed as arg matching a target function's **variadic** annotation (e.g. `Log.Info(...)` with `@param ... string`) — stubs frequently over-specify varargs (`"%s" format accepts any` but is annotated `string`), so these can't alone drive inference.
- Assignment to an annotated field (`field_type_checks`), an annotated local (`assign_type_checks`), or as an annotated return value (`return_type_checks`).
- Any baseline hint whose contributing expression is **conditionally reached** — i.e. inside the RHS of a short-circuit `and`/`or`, or inside an if/elseif/else/while/for body. Such expressions may not execute on a given invocation of the enclosing function, so they can't establish a lower bound — they can only tighten one. Tracked via `conditionally_reached_exprs: HashSet<ExprId>` populated during `build_ir` at two sites: after lowering the RHS of any `And`/`Or` `BinaryExpression` (marking the RHS sub-tree's ExprId range), and after each statement lowered in a Frame whose `is_conditional` is true (if/elseif/else/while/for bodies; do-blocks and repeat bodies inherit their parent frame's flag; function-body frames reset to `false`).

The typed-arg signal is overload-aware: it filters the callee's primary + non-return-only `Function.overloads` by arg-count (`required..=total`, `is_vararg` for the primary), then collects hints at the candidate position from every matching signature. Generic `T` / `T[]` params are substituted via `substitute_generics_deep` using generics inferred from the sibling (non-candidate) args of the same call (`infer_array_element_type` for `T[]`, direct arg type for `T`). Unsubstituted type-variables are dropped. This prevents the 3-arg `tinsert(list, pos, value)` primary from infecting a 2-arg `tinsert(list, x)` with `pos: integer` — only the 2-arg `@overload fun(list: T[], value: T)` matches by arity, and `T` is inferred from the first arg's `T[]` type.

Skipped cases: `self` params, params already annotated (`param_annotations[i]` non-empty), params with an existing `resolved_type`, and external (stub) functions (`sym_idx >= EXT_BASE`).

**Multi-site caller bail-out.** Alongside body hints, `collect_backward_inference_hints` also records the actual arg types passed at each external call site of a candidate function (`caller` map on `BackwardInferenceHints`). These aren't mixed into the body-hint intersection (they're lower bounds, not upper bounds). Instead, `caller_types_mutually_compatible` runs `intersect_pair` on every pair; if any pair is disjoint, inference bails for that param and it stays untyped. Example: `register(GameTooltip)` + `register(ItemRefTooltip)` at top level → two disjoint class tables → body-inferred `GameTooltip` would spuriously reject the `ItemRefTooltip` site, so the param is left as `?`. A single conflicting caller (e.g. `f(nil)` where the body infers `number`) still goes through so the type-mismatch diagnostic fires — only *caller-vs-caller* disagreement bails, not caller-vs-body. `nil` arg types are dropped (signals optionality rather than a type), as are types containing a `TypeVariable`.

Because the pass runs inside the fixpoint fallback, expressions using the param re-resolve naturally on the next iteration via the existing cache-clear + pending-calls repopulation logic.

## Correlated return-only overload inference
`Analysis::synthesize_correlated_return_overloads()` in `narrowing.rs` adds synthetic return-only `ResolvedOverload` entries to a function whose return statements form a clear all-set-or-all-nil pattern. On by default; gated by `correlated_return_overloads` (Analysis field, populated from `inference.correlated_return_overloads` in `.wowluarc.json`; default `true`).

Trigger point: invoked from the `stack.pop()` handler in `build_ir()` when the popped frame's `func_id` differs from the new top-of-stack's `func_id` (i.e. the function body completed, not just a nested if/do block within it). Doing this BEFORE later statements that call the function is critical — `narrow_siblings` checks `is_return_only` at call sites, so the synthesized overloads must be in place before any later narrowing-triggering reference fires.

Detection groups `func.rets` versions by `(def_node.start, def_node.end)` (each group = one return statement). Requires:
- No `@return` annotations, no existing return-only overloads, not `has_vararg_return`, not `explicit_void_return`.
- ≥ 2 distinct return statements with matching arity ≥ 2.
- Every tuple is either entirely `Expr::Literal(Nil)` or has no `Nil` literals — mixed tuples (`return "x", nil`) are skipped to avoid false correlations where the "set" branch's nil position would survive narrowing on a sibling.
- ≥ 1 all-nil tuple AND ≥ 1 non-all-nil tuple (otherwise nothing to discriminate).

For each unique tuple a `ResolvedOverload { is_return_only: true }` is emitted. Position types are derived from each lowered return expression via `synthesized_return_type()`: `Nil` → `Nil`, string/number/boolean literals normalize to their generic types (avoiding ugly literal unions across branches), everything else → `Any`. Duplicate tuples are deduped by `returns` vector equality.

Two downstream consumers pick these up:
1. `narrow_siblings` — finds them via the existing `is_return_only` check; creates `OverloadNarrow` versions for the call's other return values exactly as it does for a hand-written tuple-union `@return`.
2. `resolve_function_call` — the FunctionRet base-type lookup at `func_scope` is replaced by an overload union when `func.return_annotations.is_empty() && any(is_return_only)`. This is required because the existing `get_symbol(FunctionRet, func_scope)` only finds returns at the function-body scope, not nested-if returns; for unannotated functions whose every return is in a nested branch, the lookup would otherwise produce no type. The synthesized overloads encode types for ALL return statements, so the union gives a useful base type. Use `self.func(func_idx).return_annotations` directly here — the local `return_annotations` variable in `resolve_function_call` is only cloned for generic functions.

## Unknown-type diagnostics (strict typing)
Four HINT-severity, default-disabled diagnostics fire at sites whose `resolved_type` ended up as `None` after all inference passes. `resolved_type = Some(Any)` is treated as an explicit author-written `@type any` / `@type unknown` / `@param x any` and skipped — there's no engine-level distinction between user-written `any` and resolver-produced `Any`, so `None` vs `Some(Any)` is the discriminator.

- `unknown-param-type` — unannotated, non-`self`, local function parameter whose type couldn't be inferred from the body (arithmetic/concat hints, typed-arg calls, etc.) or reconciled with caller types.
- `unknown-return-type` — a return expression with no resolvable type, **and** the function has no `@return` annotation at that ret_index. When `@return Foo` is declared, the annotation is authoritative — body mismatches are `return-type-mismatch` territory.
- `unknown-local-type` — `local x = expr` where `expr` resolves to `None`. Explicit `---@type Foo` produces `Some(_)` and is skipped.
- `unknown-field-type` — field assignment on a `@class` table (local or overlay) where the RHS resolves to `None` **and** the field has no `annotation_type_raw` (no `@field` declaration).

All four are implemented as `DiagnosticPass` trait implementations in their respective modules under `src/diagnostics/`, running as post-analysis passes via `run_all()`. Param emission walks AST Parameter tokens (mirrors `incomplete_signature_doc`) since the param symbol's `def_node` points at the whole function, not the param name.

## Implicit protected for `_`-prefixed names
Runtime-discovered data fields starting with `_` are implicitly `Protected` when no explicit visibility annotation is present. **This behavior is configurable and disabled by default.** Set `inference.implicit_protected_prefix: true` in `.wowluarc.json` to enable it. This does **not** apply to explicit `@field` declarations — those default to `Public` since the author had the opportunity to write `@field protected`. This does **not** apply to methods — only data fields. The helper `default_visibility_for_name()` in `annotations.rs` centralizes the implicit protected logic and takes an `implicit_protected_prefix: bool` parameter. It is called from:
- Table constructor fields in `build_ir.rs`
- All FieldInfo construction sites in `pre_globals.rs` and `prescan.rs`
- `self._foo` assignments inside class methods (the class is defining its own field)
`@field` annotation parsing does **not** call `default_visibility_for_name()` — explicit declarations always use `Public` as the default, with `@field protected`/`@field private` for explicit restriction.
Runtime field assignments from outside the class (in `build_ir.rs` and `resolve.rs`) use `Visibility::Public` — ad-hoc injected fields should not get implicit protected since there is no `@field` declaration asserting protection.

## Source-file deep notes

Deep behavior of individual source files whose one-line entry in [CLAUDE.md — Source files](../CLAUDE.md#source-files) points here.

### Semantic-token classification (`analysis/semantic_tokens.rs`)
LSP semantic-token classification. Narrow by design: (1) walks bare `Name` tokens (skips field/method *access* and parameters) and emits a `function` token when the symbol resolves to a function value; (2) classifies the name chain of a function/method *definition* header (`collect_function_def_name_tokens`) — the root receiver as `class` when it resolves to a `@class` type (`value_type_is_class`, via `symbol_resolved_type_at`), intermediate field/accessor segments as `property`, and the defined name as `method` (colon-defined) or `function` (dot-defined). A simple `function foo()` has no chain wrapper and falls to (1); a non-class receiver's root token is left to the grammar while intermediate and terminal segments are still classified. **Middle segments are `property` syntactically, never by resolved type** — a transparent `@accessor` resolves to its owning class type (that's how `self` is typed in the method body), so a type-based check would mis-color the accessor as `class`; only the root's type is consulted. This is the case a type-blind TextMate grammar can't get right in a dotted chain (it can't tell a class receiver from a namespace or an accessor field), so the analysis owns it. Everything else is left to the editor's built-in Lua grammar. Modifiers: `defaultLibrary` for stub symbols (via `is_stub_symbol()` — `idx - EXT_BASE < stub_symbols_end`, boundary captured at `load_precomputed_stubs()` time), `deprecated` when the resolved function is `@deprecated` **and not flavor-suppressed** — it reuses the same `is_stub_function(f) && crate::flavor::deprecation_suppressed(addon_flavors, func(f).flavors)` gate as the `deprecated` diagnostic, so a WoW-stub API that's still live on a flavor the addon targets is neither warned nor struck through (the two must agree at the same position). Legend is the `SEMANTIC_TOKEN_TYPES` (`function`, `variable`, `keyword`, `number`, `operator`, `class`, `property`, `method`) / `SEMANTIC_TOKEN_MODIFIERS` arrays; encoded into LSP wire format by `main_loop/semantic_token_encoding.rs::encode_semantic_tokens`.

### Multi-definition go-to-definition (`pre_globals/mod.rs`)
The per-key `symbol_locations`/`class_locations`/`alias_locations` maps keep one *primary* location each (last write wins), while the runtime-only (`#[serde(skip)]`, no blob impact) `symbol_locations_by_name`/`class_locations_all`/`alias_locations_all` maps accumulate *every* distinct workspace definition site per name (deduped by `(path, start)`). These are populated in `build_on_stubs` from `ws_globals`/`ws_classes`/`ws_aliases` independent of the name-dedup that registration applies (global functions are deduped to the first occurrence at registration, so the second file's site is only recoverable here). `analysis/queries/definition.rs::definitions_at`/`type_definitions_at` return the primary first then the extras; the single-result `definition_at`/`type_definition_at` are `.into_iter().next()` wrappers (primary == old behavior, so existing `def:`/`typedef:` tests are unaffected).

### Built-in stub name reuse is additive (`pre_globals/build_on_stubs.rs`)
`BuildOnStubsContext` drives workspace incremental builds on precomputed stubs: scope/symbol/function/table cloning, type parameter substitution, field resolution. **Built-in stub name reuse is ADDITIVE**: a workspace `@class` that collides with a stub class name (`PreResolvedGlobals::stub_class_names`, populated at load from `PrecomputedStubs.stub_classes`) merges its fields onto the existing stub `TableInfo` via `populate_class_fields` — `register_classes_and_aliases` never replaces the stub table. This deliberately preserves the common *augmentation* idiom (`@class ScrollBoxListMixin : CallbackRegistryMixin` whose `.Event` enum is synthesized onto the stub overlay by `@generates-events`; a namespace/frame typed as a builtin; a library adding a field) — a destructive replace would strip the stub's (and any synthesized) fields workspace-wide and there'd be no warning on suppressed-origin files. The *fresh-record-collides-with-a-builtin* case (the motivating false `missing-fields`, e.g. `@class CVarInfo` with its own `@field`s used as `@type table<K, CVarInfo>`) is handled WITHOUT touching the stub table: (a) per-file `prescan` skips importing stub fields into a local `@class` that declares its own explicit `@field` contract (`!declared_field_names.is_empty()`), so same-file `local` constructors check the local shape; (b) `diagnostics::missing_fields::is_required_contract_field` scopes the required-field set to `declared_class_fields[name]` (the workspace's explicit `@field`s) for any class in `stub_class_names`, so the cross-file `@type table<K, Class>` case (which resolves the value class against the *external*, additively-merged table) ignores the stub's required fields. The discriminator throughout — warning, prescan, missing-fields — is **explicit `@field` declarations** (`declared_field_names` / `declared_class_fields`), NOT inferred constructor keys, so namespaces/frames/mixins reusing a stub name are neither warned nor scoped. The `class-shadows-builtin` diagnostic warns only at a reuse that declares explicit `@field`s.

### Self-field scanners (`annotations/annotation_scanning.rs`)
Shared types (`ExternalGlobal`/`ExternalGlobalKind`/`FieldValueKind`), constants (`ADDON_NS_NAME`), shared helpers (`extract_type_annotation_for_assign`, `extract_inline_type_annotation`, `is_select_varargs`), `scan_method_typed_self_fields()`, `scan_diagnostic_directives()`, type conversion (`resolve_annotation_type`, `reduce_to_fun_alias`). **Self-field scanners (3 tiers, all gated on recognized local `@class` receivers):** `scan_method_typed_self_fields` (`self.x = v ---@type T`), `scan_method_funcall_self_fields` (`self.x = Foo()` → `FieldValueKind::FunctionCall`, resolved cross-file by `build_on_stubs`), and `scan_method_bare_self_fields` (literals/params/tables → inferred). A *chained* funcall (`self.x = Foo():Bar()`, `self.x = LibStub("X"):New()`) has a call as its receiver, so the funcall scanner (which needs a plain-name callee) can't resolve it — `funcall_has_chained_receiver()` makes the two scanners' coverage disjoint: the funcall scanner skips chained receivers and the bare scanner registers them existence-only as `any` (the honest "unknown"). **Deliberately `any`, NOT a bare `table`:** the chain can return *any* type, and a concrete `table` placeholder leaks into reads — a non-table result (`f():GetHeight()` → number, a string) passed to a typed parameter then false-positives as `type-mismatch` (`got table`), and a call on it as `cannot-call`. The `any`-over-`table` choice is empirically validated (`build_on_stubs.rs` bare-local note: across 18 real addons `any` added zero diagnostics, `table` added false `cannot-call`/`type-mismatch`); the theoretical `any and x` → `x?` `return-mismatch` never lands on these existence-only fields in practice. Self-fields on *mixin* tables (no local `@class`) are out of scope here — they're handled by `scan_globals.rs`'s descendants pass (below).

### Coarse global scan & descendants pass (`annotations/scan_globals.rs`)
`scan_file_globals[_with_synth]()`, `scan_dynamic_global_prefixes()` (detects `_G["PREFIX"..k]` patterns for auto-allowed globals), and `scan_created_globals()` (detects `@creates-global` calls; see [Annotation & type-system feature reference — `@creates-global N`](#creates-global-n-implicit-named-global-side-effect)). The coarse global scan is **detection-only** for deferred functions: `build_func_external()` records a function's location, annotation-derived signature, and deferred-ness (no `@return`), but does **not** infer body-derived return types or synthesize correlated return-only overloads. All body-derived returns/overloads are produced lazily by the per-file harvest (`analysis/deferred.rs::resolve_deferred_sig`), the single source of truth. The `_with_synth` variant's `correlated_return_overloads` parameter is retained for caller compatibility but no longer affects scanning (the harvest reads the flag from the file's config directly). The main statement loop only walks top-level + control-flow statements (`collect_statements_recursive` does **not** descend into function bodies), because the coarse scan can't reliably *type* values produced inside functions (locals, `Mixin`'d frames, chained builder results). A final pass over `root.descendants()` registers the **existence** of namespace/`@class` fields assigned *inside* function bodies, of multi-target assignments (`a.x, a.y = f()`) anywhere, and of bare single-name globals created via `X = ...` *inside* function bodies (e.g. a saved-variable global initialized only in an event/initializer function) — all gaps the main loop leaves. Field/multi-target writes emit existence-only `TableField(_, _, FieldValueKind::Unknown)` entries (no type); the build's Unknown-field pass registers an otherwise-absent field as `any` (the honest "unknown" — suppresses cross-file `undefined-field` on reads without fabricating a shape; unlike a bare `table` it never false-positives a non-table value passed to a typed parameter as `type-mismatch`, or a call as `cannot-call`) and skips fields that already carry a precise type or methods. A function-literal RHS (`self.cb = function() end`) instead emits `FieldValueKind::Function` so the field registers callable (avoids a false `cannot-call` when it is later invoked). A **forwarded** RHS — another field or a parameter (`ns.Foo = current.func`, `ns.Cb = callback`), i.e. an `Expression::Identifier` — on a *non-`self`* namespace/`@class` field emits `FieldValueKind::MaybeCallable`, materialized by the Unknown-field pass as `ValueType::callable_or_unknown()` (the `function & table` intersection): the forwarded value may be a callable, so calling the field must not false-positive as `cannot-call`, while the intersection keeps reads as permissive as a bare table (callable, truthy, assignable to/from any table — see the `callable_or_unknown` doc-comment). This is **scoped to non-`self` roots** (the reported namespace-field case): a `self.` field is left `Unknown` (materialized as `any`), because self data-fields are usually data, also collect a per-file overlay / self-field-scanner type, and forcing them callable would union `function & table` with that real type and turn clean reads into `type-mismatch`. A function-*call* RHS (`CreateFrame(...)`) likewise stays `Unknown` → `any`; the Unknown-field pass's skip guard (it never overwrites a field that already carries a concrete non-`Any`/non-`Table(None)` annotation) lets a competing concrete type (e.g. a merged-in `Frame`) win, so the placeholder only materializes when no better type exists. The descendants pass also handles `self.field = ...` writes by resolving `self` to the nearest enclosing *named* colon method's receiver (skipping anonymous callbacks, which close over the outer `self`): an addon-namespace receiver routes to `ADDON_NS_NAME` (forwarded to its `@class` by `merge_addon_ns_into_classes`), a function-local receiver is skipped (can't be a cross-file mixin), and anything else routes to the receiver's single (leaf) name — but **only when that leaf unambiguously identifies the table**. A *single-name* receiver (`function Mixin:M()`, the method's own root) is the non-`@class` *mixin*-table pattern (promoted to a class only via an XML `mixin=` / `Mixin()` reference) and routes to the receiver name; a *deeply-nested* receiver (`function A.B.C:M()`) is keyed by its bare leaf, which can collide with an unrelated same-named global, so it is registered only when that leaf names a known `@class` (via `var_to_class`, matching the typed/bare scanners) and is **skipped otherwise** rather than misattributed. This is the only path that tracks self-fields on mixin tables; the typed/funcall/bare self-field scanners (`annotation_scanning.rs`) gate on recognized local `@class` names and so miss them. For a *redeclared* local `@class` receiver those scanners (which feed `ClassDecl.fields`) own it instead, since the existence-only `Unknown` entry here would be dropped by the prescan overlay-import filter. In-function single-name globals emit existence-only `Variable(FieldValueKind::Unknown)` entries so cross-file *reads* don't false-positive as `undefined-global` (no fabricated type). To avoid leaking function-scoped locals as phantom globals, `collect_all_local_names()` gathers every name bound as a local anywhere in the file (`local`/`local function`/params/for-vars, including inside function bodies) and a bare write to such a name is treated as a local reassignment and skipped; an explicit `_G.X = ...` write bypasses this and always registers.

### Callback registries (`annotations/scan_callback.rs`)
`scan_callback_registries()` collects callback registries (`Receiver:GenerateCallbackEvents(...)` where the method is `@generates-events`) as `CallbackRegistryDecl` (canonical receiver path + inline events + optional `events_ref` for a table-reference arg) plus `path = { "a", "b" }` `StringArrayConstDecl`s, both keyed by `canonicalize_member_path` (addon-ns alias rewritten to `ADDON_NS_NAME`). `build_generates_events_methods[_iter]()` builds the leaf-method→spec map shared with `DefclassContext`. Receiver/const keys are canonicalized via `canonicalize_member_path` then addon-scoped via `scope_addon_ns_path` (so multi-addon `addonTable.X` paths don't collide). Merged at `PreResolvedGlobals::merge_callback_registries` (joins registries with constants, marking incomplete on unresolved refs/collisions); consumer methods registered via `register_callback_consumer_methods` (`@callback-event-arg`). Powers event-name completion + `unknown-callback-event`. See also [Annotation & type-system feature reference — `@generates-events`](#generates-events-n-field-synthesized-event-enum-table-on-a-class) and [`@callback-event-arg`](#callback-event-arg-n-callback-registry-event-name-validationcompletion).

### Cross-file analysis cache (`lsp/main_loop/hierarchy.rs`)
`hierarchy.rs` holds call/type hierarchy, workspace symbol search, and cross-workspace references/implementations. `find_references_across_workspace` reuses `WorkspaceState::xfile_analysis_cache`, a per-`ws_generation` cache of parsed+resolved *unopened* files, so repeated/batched code-lens "N usages" resolves don't re-read+re-parse+re-`resolve_types` the whole workspace each time; the cache self-invalidates when `ws_generation` advances, i.e. on any `rebuild()`.

## Configuration internals (`config.rs`)

Deep behavior behind the one-line `config.rs` entry in [CLAUDE.md — Source files](../CLAUDE.md#source-files).

### `flavors_for` vs `addon_flavors_for`
`flavors_for(path)` (config `flavors` ∩ flavor-specific TOC suffix/`AllowLoadGameType`) is the strict declared set that powers `wrong-flavor-api` — 0 disables flavor filtering. `addon_flavors_for(path)` is the addon's *declared breadth* used only by flavor-aware `deprecated`: it returns `flavors_for` when nonzero, else falls back to the nearest ancestor `.toc`'s `## Interface:` versions (`toc_interface_flavors`, a dir-keyed map populated in `try_load_toc` from `TocParseResult.interface_flavor`). The `## Interface:` fallback is deliberately kept OUT of `flavors_for`/`project_flavors` so it never turns on `wrong-flavor-api` for a config-less addon — only `deprecated` consults it. Unlike `toc_file_flavors`, `toc_interface_flavors` keeps `FLAVOR_ALL` entries (a multi-version addon's breadth still matters for deprecation).

### Nested-config policy (isolate vs inherit)
Settings that affect diagnostics are **isolated** — a file's effective value comes solely from its single nearest ancestor `.wowluarc.json` (via `nearest_config()` / `nearest_bool()`); unset keys fall back to hardcoded defaults, *not* parent values. This keeps `check` results invocation-independent (subdir scan == root scan filtered to that subdir). Isolated: `diagnostics.*`, `globals.*`, `framexml`, `flavors`, `inference.*`, `ignore`, `plugins` (filtered per-file by source path in `PluginEngine::run_plugins`). Editor-preference settings that don't affect diagnostics (`hint.*`, `code_lens.*`, `completion.snippets`, `editor.auto_insert_end`) keep the inherited `deepest_bool` behavior. Config discovery only walks *downward* from the scan root (no walk-up), so the "no nearest config" case is intentionally best-effort. When adding a new `*_for` accessor, slot it deliberately into the isolate (`nearest_*`) or inherit (`deepest_bool`) bucket. **`library` is the deliberate exception to isolation**: `ProjectConfigs::is_library()` checks *every* config entry, not just the nearest — relative patterns from any ancestor config apply to all descendants, and absolute external dirs stay workspace-wide via `external_library_dirs()`. This is "inherited downward": marking a subtree as a library means "this whole subtree is vendored third-party code," so a nested config deeper in the tree (e.g. a vendored library's own `.wowluarc.json`, possibly pulled in via a symlink) must not be able to un-mark it. The tradeoff is that this one setting is *not* invocation-independent — a scan rooted *inside* a library subtree won't see the ancestor's `library` declaration. **Built-in default ignore**: `ProjectConfigs::is_ignored()` skips default-ignored directories (the `DEFAULT_IGNORED_DIRS` list, currently just `.github/`) with or without a config. `.github/` holds CI/build tooling run by the standalone Lua interpreter (real `io`/`os` stdlib), never in-game addon code, so analyzing it as WoW Lua only yields false `undefined-global`s. The match is **anchored to the workspace** so a default-ignored segment *above* the scan root can't silently prune the whole project: in the configured branch it runs `is_default_ignored()` (any-component match) on the config-relative remainder (`strip_prefix(config_dir)`), exactly like the `ignore`/`library` patterns; in the no-config branch it matches a directory *entry* by its own name via `is_default_ignored_component()` (so the walk prunes a `.github` entry when it reaches it, but the ancestor segment that appears in every entry's absolute path never matches). Both helpers are free fns near `matches_path_patterns`. Adding a new default-ignored dir = extend `DEFAULT_IGNORED_DIRS`.

### Flavor-aware `deprecated` (`flavor.rs`)
`flavor.rs` also owns flavor-aware `deprecated` (`deprecation_suppressed(addon_flavors, fn_flavors)`): WoW deprecations are retail-side (`DEPRECATION_ORIGIN_FLAVORS = FLAVOR_RETAIL`), so a `@deprecated` API still available outside retail in a flavor the addon targets is "live there" and the warning is suppressed. And `interface_number_flavor(n)` / `parse_interface_flavors(value)` map a `.toc` `## Interface:` version (`MAJOR*10000+...`) to a flavor by major version (1 → classic_era, 2–5 → classic, ≥6 → retail). The call-site and semantic-token consumers of this gate are the `deprecated` diagnostic (`annotation_metadata.rs`) and the `deprecated` semantic-token modifier; the two must agree at the same position. See also [Flavor filtering](#flavor-filtering-flavors-config--toc-detection--flavor-narrows--wrong-flavor-api).

## Inlay hints (`analysis/queries/inlay_hints.rs`)
`inlay_hints(tree, config)` collects six categories of inline annotations controlled by `InlayHintConfig` (from `.wowluarc.json` `hint.*` fields, enabled by default unless noted):
1. **Parameter names** (`collect_param_name_hints`) — iterates `call_resolutions`, emits `InlayHintKind::PARAMETER` before each argument. Suppressed when: arg text matches param name (case-insensitive), param is `self`, param is vararg, or param name is empty.
2. **Variable types** (`collect_local_type_hints`) — walks `LocalAssignStatement` nodes, emits `InlayHintKind::TYPE` after each name token. Suppressed when: variable has `@type` annotation, resolved type is `Any`/`Nil`/`Function`, or RHS is a function literal. Per-variable check (not per-statement).
3. **Function return types** (`collect_function_return_hints`) — matches functions by `def_node.start`, emits after the parameter list close paren. Suppressed when: function has `@return` annotation, `returns_self`, or `explicit_void_return`.
4. **For-loop variable types** (`collect_forin_type_hints`) — walks `ForInLoop` nodes, emits after each name token. Suppressed when: variable has `@type` annotation or resolved type is `Any`.
5. **Parameter types** (`collect_param_type_hints`, **disabled by default**, `hint.parameterTypes`) — walks `FunctionDefinition` nodes, emits `InlayHintKind::TYPE` after each parameter name token. Suppressed when: parameter has a `@param` annotation, resolved type is `Any`/`Nil`, or parameter is `self`.
6. **Chained method return types** (`collect_chained_return_hints`, **disabled by default**, `hint.chainedReturnTypes`) — iterates `call_resolutions`, emits `InlayHintKind::TYPE` at the closing `)` of calls whose return value is used as the receiver of a subsequent field/method access. Suppressed when: return type is `Any`/`Nil` or formats to `"?"`. Only intermediate calls in a chain get hints (the final call is covered by variable type hints).

All type hints use `format_type_depth(resolved, 1)` (depth 1) to avoid expanding table fields with newlines — inlay hints show class names only, not field listings.

LSP handler in `main_loop/handlers.rs` converts `InlayHintData` (byte offsets) to LSP `InlayHint` (line/column positions). Config is built from `ProjectConfigs` per-file hierarchy.

## IR read-only query surface — enforcement detail
This expands the rule-of-thumb summary in [CLAUDE.md — IR read-only query surface](../CLAUDE.md#ir-read-only-query-surface-post-build-consumers).

**The boundary is enforced at the type level.** The five arena fields on `Ir` are **private to the `analysis` module** (plain private, not `pub(crate)`): builders and queries in descendant modules (`build_ir`, `resolve`, `prescan`, `narrowing`, `queries/*`, …) still touch them directly, but every consumer *outside* the module — `diagnostics` (a sibling module in the same `wowlua_analysis` crate) and `doc_gen`/`stub_gen`/`lsp/main_loop`/`plugins/query` (in higher crates) — is forced through the surface methods. Those methods are now `pub` (they are the analysis crate's public API across the crate boundary), while the arena fields stay private. `PreResolvedGlobals` got the symmetric treatment: its five arena fields are **private to `pre_globals`** behind matching `pub` routing accessors (`sym`/`func`/`expr`/`table`/`scope`, fallible `try_expr`/`try_scope`/`try_table`, `iter_symbols`, and the existing `*_len`). The `Ir` routing layer reads its `ext` through those (`self.ext.sym(idx)`, never `self.ext.symbols[…]`), so the `analysis`→`pre_globals` arena dependency is also encapsulated. `doc_gen`'s unit-test helpers build a synthetic `PreResolvedGlobals` via the `push_ext_{symbol,function,expr,table}` methods (gated on `any(test, feature = "test-util")` since `doc_gen` is a higher crate) rather than pushing to the (now private) arenas. Keep these fields private; do not widen them — add a new accessor on the relevant surface instead.

## Annotation & type-system feature reference

Deep notes for the annotation-syntax and `ValueType` features whose one-line entry under [CLAUDE.md — Conventions](../CLAUDE.md#conventions) points here. Several closely-related features already have their own top-level section above and are cross-linked, not duplicated: [Variadic generics (`@generic T, ...M`)](#variadic-generics-generic-t-m), [`@narrows-arg N`](#narrows-arg-n-in-place-argument-type-mutation), [Tuple-union `@return` syntax](#tuple-union-return-syntax), [Parameterized classes (`@class Name<S>`)](#parameterized-classes-class-names), and the [Builder pattern](#builder-pattern-builds-field--return-built).

These remain **conventions** — they are load-bearing rules for anyone touching the type system. They were relocated here only to keep CLAUDE.md's always-loaded footprint small; treat them with the same authority as the conventions that stayed inline.

### `@field name? type` (optional fields)
The `?` is stripped from the field name at parse time in `annotations.rs` and the type is wrapped in `Union(type, nil)`. Field HashMap keys never contain `?`. Same pattern as `@param name?` handling.

### `@alias (opaque) Name type` (nominal aliases)
Creates a nominally distinct type alias: `ValueType::OpaqueAlias(String, Box<ValueType>)`. Parsed in `annotations/mod.rs` by stripping the `(opaque)` prefix. Stored in `ir.aliases` / `ext.aliases` as the `OpaqueAlias` variant (not the raw inner type). Different opaque aliases with the same inner type are NOT assignable to each other. Base values/literals are assignable into an opaque alias (Rule 2). Opaque aliases flow out to their base type freely. Arithmetic and other operators unwrap to the inner type; results decay to the base type. Hover displays the alias name.

### `@class (partial) Foo` / `(exact)`
Parsed in `annotations/mod.rs` by stripping the `(partial)` prefix before the class name. `(exact)` is also recognized. Currently parse-only — the modifier is accepted but has no effect on diagnostics.

### `@class Foo : table<K, V>` (dictionary inheritance)
Class inheriting dictionary key/value types. Parent name parsing in `annotations/mod.rs` uses `split_at_top_level` to handle commas inside `<>`. During inheritance resolution (prescan.rs pass 3, pre_globals/mod.rs, pre_globals/build_on_stubs.rs), `table<K,V>` parents set `key_type`/`value_type` on the child's `TableInfo` without adding to `parent_classes`. This enables typed `pairs()` iteration over class-typed tables. The `undefined-doc-class` diagnostic skips parameterized parents (those containing `<`).

### `T & U` (intersection type)
`AnnotationType::Intersection(Vec<AnnotationType>)` / `ValueType::Intersection(Vec<ValueType>)`. Parsed via `&` with higher precedence than `|` (split `|` first, then `&` inside each union member). An intersection is assignable to X if ANY member is; X is assignable to an intersection if assignable to ALL members. Field access checks all member tables. Used by `CreateFrame` stub to combine frame type with template mixin (`T & Tp`).

### `T!` (non-nil assertion / lateinit)
`AnnotationType::NonNil(Box<inner>)` wraps the inner type. Resolves to the inner type with nil stripped. On `@field` or `---@type`, sets `FieldInfo.lateinit = true`, which suppresses `field-type-mismatch` for nil assignments and ensures the field's resolved type is non-nil (no `need-check-nil` on access). Hover shows `T!`. In `fun()` return types (e.g. `fun(): K!, V!`), the `NonNil` wrapper is preserved in `Function.return_annotations_raw` even though the resolved `return_annotations` loses it. For-in variable resolution (`Expr::ForInVar` in `resolve.rs`) checks `return_annotations_raw` for `NonNil` to strip nil from iteration variables — this is how `pairs`/`ipairs`/`next` stubs (`stubs/overrides/`) express that Lua tables cannot store nil values, avoiding hardcoded position checks.

### `{field: type, ...}` (anonymous table shape)
`AnnotationType::TableLiteral(Vec<(String, AnnotationType)>)`. Parsed in `parse_type()` when the string starts with `{` and ends with `}`, splitting on `,` at top level and then `field: type` pairs. Resolves via `materialize_table_literal()` in `prescan.rs` which creates a `TableInfo` with the specified fields. Supports optional fields (`field?: type`) which become `Union(type, nil)`. Works in `@param`, `@return`, `@type`, `@alias`, and inside intersections (`T & {field: type}`).

### `[T1, T2, ...]` (LuaLS tuple syntax)
Parsed in `parse_type()` and **lowered to a `TableLiteral` with integer-literal keys** (`{ [1]: T1, [2]: T2 }`), reusing the existing materialization/field-access/`undefined-doc-name` machinery rather than a dedicated tuple value type. `split_at_top_level`/`extract_type_prefix` track `[`/`]` (alongside `<>`/`()`/`{}`) so the inner commas are protected and a `T[]` array segment stays intact. Element access (`x[1]`) resolves the per-position type via the `[1]`/`[2]` field-name convention shared with `{ [1]: ... }`.

### `?T` (prefix-optional shorthand)
Some authors (and LuaLS) write the optional marker as a prefix (`?number`) rather than the canonical suffix (`number?`). `parse_type()` strips a leading `?` and returns `Union(T, nil)`, identical to `T?` — accepted leniently with no diagnostic (no type legitimately starts with `?`). On `@param`/`@field` this makes the slot optional via `annotation_type_is_nullable`, suppressing the `missing-parameter` false positive the dropped marker used to cause.

### Comma-separated `@return T1, T2` (LuaLS single-line multi-return)
One return slot per top-level comma segment, each `<type> [name]`, with an optional trailing `# description` or free-text description on the final segment. Parsed by `annotation_types::parse_multi_return_line()` (left-to-right, matching LuaLS: the type list continues only when a `<type> [name]` is *immediately* followed by a top-level comma), so a single return whose **description** contains a comma (`@return number red Red color, from 0 to 1`) stays one return — the helper returns `None` there and the caller falls back to `parse_return_line` (which still owns parenthesized tuple / tuple-union / vararg / `built` forms). The old "comma-separated return types are not supported" `malformed-annotation` was removed.

### LuaLS-only `@diagnostic` codes
`diagnostics/mod.rs::LUALS_ONLY_CODES` lists codes LuaLS defines but wowlua_ls has no equivalent for (e.g. `lowercase-global`). They're folded into `known_codes()` so `unknown-diag-code` accepts them silently; they suppress nothing (no mapping). Codes that *do* map to one of ours go in `CODE_ALIASES` instead.

### Number-literal types (`0` / `-1` / `0xFF`)
`ValueType::NumberLiteral(String)` (the variant is kept **last** in the `ValueType` enum so it doesn't shift serde variant indices in the precomputed-stub blob — adding it mid-enum breaks blob deserialization). The String preserves the source spelling (kept `Eq`/`Hash`; f64 is not). Detected by `is_number_literal()` in `annotations/annotation_scanning.rs` (decimal/float/hex, optional leading `-`), used by `resolve_annotation_type` and the `undefined-doc-name` validator (`check_annotation_type_names`). Mutually assignable with plain `number`. Decays to `Number` under arithmetic/comparison via `into_decay_number_literal()` (called in `resolve_binary_op_standalone`); `make_union` collapses `number | 0` → `number`. Hover shows the literal spelling. Primary use is tuple-union sentinel cases like `@return (number, string) | (0, nil)`. **Only annotation literals become `NumberLiteral`; source-code numeric literals stay `Literal(ValueType::Number)`.**

### `ValueType::FunctionSig(Box<FunctionShape>)` (inline function signature)
A self-contained structural function type carrying `FunctionShape { params: Vec<ShapeParam>, returns: Vec<ValueType>, is_vararg }` (`ShapeParam { name, ty, optional }`). Unlike `ValueType::Function(Some(FunctionIndex))`, it holds no arena index, so it survives cross-file. **Runtime-only — never serialized into the precomputed-stub blob**, so it can live mid-enum without breaking blob deserialization (it's constructed lazily during the cross-file harvest, not from stubs). Built by `analysis/deferred.rs::lift_local_func_to_shape()` when a defining file's harvested return type is a local function (`Function(Some(local_idx))`): the lift converts the per-file `Function` into a `FunctionSig` (depth-guarded by `LIFT_MAX_DEPTH`) instead of decaying to bare `function`, making `Test() -> fun(a: string)` survive cross-file. Treated as callable (`cannot_call.rs`), truthy, and mutually assignable with `Function`. Formatted by `format_function_shape()` in `queries/format.rs`.

### `ValueType::TableShape(Box<TableShape>)` (inline anonymous table shape)
A self-contained structural table type carrying `TableShape { fields: Vec<(String, ValueType)> }` (sorted by name). Like `FunctionSig`, it holds no arena index so it survives cross-file, and is **runtime-only — never serialized into the precomputed-stub blob** (appended after `FunctionSig` so blob variant indices stay stable; constructed lazily by the cross-file harvest, not from stubs). Built by `analysis/deferred.rs::wrap_overlay_shape()` when a deferred function returns an external class instance carrying per-file overlay fields (`frame.DropDown = ...` on a `CreateFrame` result): the return is lifted to `Class & TableShape` so the injected fields resolve, complete, and hover cross-file, and the surrounding `Intersection` suppresses `undefined-field` (concrete-instance policy). **Per-file aggregation (pre-existing):** `overlay_fields` is keyed by the *shared* external class index, so the shape carries the union of every same-classed instance's injected fields in the defining file — matching what same-file hover already shows for a `Frame`-typed local. Field access (`resolve.rs` `FieldAccess`) and completion/hover (`queries/completion.rs`, `queries/hover.rs::shape_field_hover_at`) read shape members via `ValueType::collect_shape_field_types`/`collect_shape_field_names`; treated as a table (truthy, table-kind, permissively assignable to/from any table — no structural rejection). Formatted as `{ field: type, ... }` in `queries/format.rs`.

### `NarrowKind::NumCompare { op, bound }` (numeric case discrimination)
A `if x > 1` style guard on a bare symbol records `(Operator, bound_string)` in `Analysis::num_compare_narrowed_symbols` (detected in `narrowing.rs`, oriented via `flip_comparison` when the literal is on the LHS). An ordered comparison (`<`/`>`/`<=`/`>=`) errors at runtime on nil, so reaching the then-branch proves `x` is non-nil: for a plain scalar the narrowing block also adds `x` to `narrowed` and pushes a `StripNil` version (`narrow_siblings` returns `false` for non-tuple-union symbols, gating this — for tuple-union members the `OverloadNarrow` machinery owns the narrowing and the crude `StripNil` is skipped). `narrow_kind_for` returns `NumCompare` after `ClassEq`/`StripTruthy`/`StripFalsy` but **before** plain `StripNil`, since `NumCompare` now implies non-nil and is at least as strong (a compound `x and x > 0` still keeps the stronger truthiness narrowing). In `overload_type_compatible_with` → `overload_type_survives_num_compare` (resolve.rs), a tuple-union case is eliminated when its slot is `Nil` or when a `NumberLiteral` value fails `value <op> bound` (parsed via `parse_num_literal_str`); plain `number`/`any` slots always survive (no range tracking).

### `...T` (variadic return)
`AnnotationType::VarArgs(Box<AnnotationType>)`. When `@return ...T` is the last return annotation, it fills all remaining return slots with type `T`. Stored as `Function.has_vararg_return: bool`. The vararg portion is optional (no `missing-return-value` for it) and `redundant-return-value` is suppressed. Multiple returns must use separate `@return` lines (comma-separated multi-return on a single `@return` line is not supported). Tuple-union cases ending in a `...T` tail also set this flag — see [Tuple-union `@return` syntax](#tuple-union-return-syntax).

### `@returns-class-name` (return-value class guard)
`Function.returns_class_name: bool`. Marks a method whose string return value names its receiver's runtime class (the canonical case is WoW's `FrameScriptObject:GetObjectType`, spec'd in `stubs/overrides/GetObjectType.lua` — **never hard-code the method name in the engine**). At narrowing time (`narrowing.rs::extract_returns_class_name_guard`), an equality comparison `recv:m() == "Class"` (either operand order) narrows `recv` to `Class` via `apply_type_narrows` — the return-value-comparison counterpart to the boolean-guard `@type-narrows`. Positive direction only: then-branch `==` / else-branch `~=` (and the `~=` early-exit + `assert(... == ...)` forms); the inverse can't cleanly subtract a class. The receiver must be a single-name `recv:m()` (no deeper field-chain receivers); unknown/non-literal class names degrade gracefully (no narrowing). Plumbed exactly like `@type-narrows`: `AnnotationBlock` → `ExternalGlobal` → `FnMeta` (`returns_class_name_raw`) → `Function`, applied to local funcs in `build_ir.rs`. **Rides the stub blob** (so the `GetObjectType` override carries it), hence in `global_semantic_eq` and bumped `BLOB_VERSION`; also added to the `try_resolve_call_function` global-search fallback gate so cross-file unresolved receivers still resolve the method.

### `@creates-global N` (implicit named-global side effect)
Marks a function (e.g. `CreateFrame`, `CreateFont`) whose call creates a global named by the string-literal at 1-based param `N`. Stored as `ExternalGlobal.creates_global: Option<CreatesGlobalSpec>` (just `name_param`; defined in `annotations/annotation_scanning.rs`), parsed in `annotations/mod.rs`. **Never hard-code creating-function names in the engine** — this annotation is the mechanism (CreateFrame's spec lives in `stubs/overrides/CreateFrame.lua`, the Font functions' in `stubs/overrides/CreateFont.lua`). Detection: `scan_globals.rs::scan_created_globals(root, &CreatesGlobalMap)` emits an `ExternalGlobal` (typed `Variable(FunctionCall)`, **empty `returns`**, `deferred_call_type = true`, `def_start` = the creating call's offset) for each match; the spec map is built from stub globals via `build_creates_global_map()` (also cached on `PreResolvedGlobals::creates_global_specs`). Detection runs in the **stub-aware pass**, not pass 1: `complete_directory_scan` (LSP full scan, over cached trees), `scan_paths_with_overrides` via `scan_file_globals_with_synth` (scan_workspace path), and the incremental rebuild (`rebuild.rs`/`handlers.rs`). Pass 1 (`scan_lua_file_cached`) passes an empty map because it runs before stubs load. Entries are registered first/prepended so they win over a same-named assignment.

- **Type harvest (no type token).** The created global's type is **not** reconstructed from annotations — it is harvested from the creating call's *resolved* return type, so a `CreateFrame(..., "Template")` call yields `Frame & Template` (the template overload's intersection), matching the same-file local-assignment type. Mechanism mirrors deferred function returns (`analysis/deferred.rs`): `build`/`build_on_stubs` record each created global in `PreResolvedGlobals::deferred_call_globals` (`SymbolIndex → DeferredCallGlobal { path, call_offset }`, plus `_by_path` and a `deferred_call_global_cache` RwLock memo — all `#[serde(skip)]`, no blob impact) and leave the symbol's `resolved_type` unset (the mod.rs deferred-funcall pass skips them). `resolve_deferred_call_global_type()` lazily re-analyzes the defining file, matches `Expr::FunctionCall.call_range.0 == call_offset` (ret_index 0), reads `resolved_expr_cache`, and lifts to ext space. Reads route through a per-file **symbol overlay** (`Ir::symbol_overlay`, checked in `sym()` exactly like `func()`'s `overlay`), warmed by `Ir::ensure_symbol_overlay()` in a one-time pre-pass at the top of `resolve_types()` (so aliased locals get the type too). `deferred_call_type` is `#[serde(skip)]` (workspace-only). **Detection is stub-only**: only functions defined in stubs are scanned as `@creates-global` sources (the spec map is built from stub globals).
- When adding `CreatesGlobalSpec` semantic fields, update `global_semantic_eq` in `rebuild.rs` and bump `BLOB_VERSION`.

### `@generates-events N [Field]` (synthesized event enum table on a class)
Marks a **method** (e.g. `CallbackRegistryMixin:GenerateCallbackEvents`) whose call `Receiver:Method({ "OnFoo", SomeEvents.OnBar, ... })` synthesizes an enum-like `Field` table (default `Event`) on the receiver class. Stored as `ExternalGlobal.generates_events: Option<GeneratesEventsSpec>` (`events_param` + `field_name`; defined in `annotations/annotation_scanning.rs`), parsed in `annotations/mod.rs`. **Never hard-code the method/field names in the engine** — the spec lives in `stubs/overrides/CallbackRegistryMixin.lua`. Detection **rides the defclass scan**, not a separate pipeline: `DefclassContext` carries a `generates_events` map (leaf method name → spec, built from all_globals), and `scan_defclass_calls_with_context` (`scan_defclass.rs::collect_generated_event_classes`) emits a `ClassDecl` for the receiver carrying one field of type `TableLiteral([(event, string), …])` per call. Array entries: string literals use their value; field references (`SomeEvents.OnBar`) use the leaf name (value==name convention). Only **single-name receivers** (`names.len() == 2`: `[Receiver, Method]`) are handled. The synthesized `ClassDecl` merges into the receiver's `@class` overlay via `merge_defclass_into_overlays` (field added only if absent — multiple `GenerateCallbackEvents` calls on one receiver don't union). Because it rides the defclass scan, the defclass gating is widened to include `generates_events`: `needs_defclass` in `scan.rs` (both full-scan paths), and `rebuild_caches` in `state.rs` adds the method leaf names to `cached_defclass_func_names` and forces `cached_needs_defclass`. Add new `GeneratesEventsSpec` semantic fields to `global_semantic_eq` and bump `BLOB_VERSION`. The synthesized `TableLiteral` field is **not closed** (accessing an absent event resolves leniently, no `undefined-field`).

### `@callback-event-arg N` (callback-registry event-name validation/completion)
Marks a **consumer** method (`CallbackRegistryMixin:RegisterCallback`/`UnregisterCallback`/`TriggerEvent`/`RegisterCallbackWithHandle`) whose 1-based arg `N` is an event name. Stored as `ExternalGlobal.callback_event_arg: Option<usize>`, parsed in `annotations/mod.rs`, spec lives in `stubs/overrides/CallbackRegistryMixin.lua` (never hard-code method names). Pairs with `@generates-events` (the producer): `scan_callback.rs` collects `path → event-set` registries (keyed by canonical receiver path, addon-ns alias rewritten via `canonicalize_member_path`; events come from an inline array **or** a cross-file `addonTable.Constants.Events`-style string-array constant resolved at `merge_callback_registries` time). The consumer side is **purely syntactic** — `addonTable.CallbackRegistry` types as bare `table` cross-file so call resolution can't be used; `AnalysisResult::callback_event_set_for_string` reads the receiver path/method/arg-index from the AST, re-detects the addon-ns var on the same tree (`detect_addon_ns_var`) and canonicalizes identically to scan time, then looks up `ext.callback_registries`. Drives string-literal completion (`queries/completion.rs`, always on) and the default-off `unknown-callback-event` diagnostic (suppressed unless the event set is `complete`, so unresolved/colliding registries never false-positive). Workspace maps (`callback_registries`, `callback_event_methods`) are `#[serde(skip)]` on `PreResolvedGlobals`; the `@generates-events`/`@callback-event-arg` fields ride the stub blob, so adding semantic fields needs `global_semantic_eq` + `BLOB_VERSION`. **Multi-addon scoping:** addon-namespace canonical paths are prefixed with the file's addon identity (`config.addon_name_for` at scan time / `Ir::addon_folder_name` at query time, via `scope_addon_ns_path`), so two addons' `addonTable.CallbackRegistry` don't collide. **Incremental refresh:** `rebuild.rs::maybe_rebuild_workspace` re-scans the edited file (gated on a `@generates-events` method name in the text or a prior contribution) using the cached `WorkspaceState::cached_generates_events_methods` map, updates `ws_file_callback_registries`/`ws_file_string_consts`, and forces a Full warm on change. **Validated on real addons: 0 false positives, found genuine unregistered-event bugs in shipping code.** Scope limits: receiver matched by canonical name (global/class/addon-ns reliable; bare locals are recorded but collide cross-file under a shared name → merge marks incomplete → suppressed); inline-array or resolvable string-array-constant event lists only (dynamic/computed lists → incomplete → suppressed).

### `@requires T: Constraint` (method availability gating)
`Function.requires_constraints: Vec<(String, String)>` (param name + raw constraint string). Parsed in `annotations/mod.rs`. At call-resolution time, `CallResolution.receiver_param_subs: HashMap<String, ValueType>` records the receiver class's type-param bindings (from `class_type_param_subs`). The `param-constraint-mismatch` diagnostic checks each `requires_constraints` entry against the bound type via `resolve_class_constraint` + `is_assignable_to`/`is_table_subtype`, emitting at the method-name range. Constraints use subtype (`:`) semantics, not equality.

### Mixin-object params accept the data type; methods-typed params are strict
(Replaces the former `@shape` annotation.) Several WoW types are modeled (by Ketho) as a data struct + a behavior mixin: `colorRGB : ColorRGBData, ColorMixin`, `ItemLocation : ItemLocationData, ItemLocationMixin`. A value *produced* as the combined type is a real instance (has the methods); a C API that *accepts* one only reads its data fields in C++ (never calls Lua methods). Two coordinated pieces:

- **Strict rule (analysis).** A plain table *literal* passed as a function argument where a class with methods is expected cannot be a real instance — its methods would be nil at runtime — so it must provide the class's (possibly inherited) methods. `resolve_call.rs::infer_table_constructor_field_params` records argument-context constructors in `Ir::tc_arg_constructors` (**only Case 2** — direct call args; the lenient `@type`/assignment/bracket/field contexts that also feed `tc_expected_class` are excluded, so explicit `@type` casts and the construct-then-Mixin idiom still type-check). `diagnostics/missing_fields.rs::check_missing_fields`, for an arg-context ctor, walks `collect_class_fields` (inherited included) and emits `missing-fields` ("table literal cannot satisfy class …") when any required (non-nullable, non-lateinit) `Function`/`FunctionSig` field is absent from the literal. A real object is a `SymbolRef`/call result, not a `TableConstructor`, so it is never in `tc_arg_constructors` and stays assignable nominally. `tc_arg_constructors` is per-file analysis state (not serialized).
- **Param remap (stub-gen).** `stub_gen/data_params.rs::remap_mixin_data_params` (applied after each scan in `orchestrate.rs`, both passes, before `PreResolvedGlobals::build`) retypes mixin-object types in **parameter** positions only — `ExternalGlobal.params` + overload params, and class-method `Fun`-typed fields — to a **`Data | Object` union** via the `MIXIN_DATA_PARAM_TYPES` map (`colorRGB→ColorRGBData | colorRGB`, `colorRGBA→ColorRGBAData | colorRGBA`, `ItemLocation→ItemLocationData | ItemLocation`). Returns and data-field values keep the object type (a produced `colorRGBA` is a real object). The union — not the bare data type — is deliberate: the object member is **load-bearing for backward inference**, so an unannotated caller variable inferred from the argument position keeps a methods-bearing alternative (`local loc = ...; C_Item.IsBound(loc); loc:IsValid()` — the data-only type would strip `loc`'s methods). Net: a data-reading C API accepts a plain `{r,g,b}`/`{bagID,slotIndex}` table (the data member) *or* a real object (either member), a hand-written `@param colorRGBA` stays strict, and a typo'd/foreign literal (`{red,green,blue}`, `{foo}`) is flagged by `missing_fields::check_missing_fields_union`. That union check collects **inherited** fields (`collect_class_fields`) and requires a candidate member both to have all its required data fields present **and** to share ≥1 known field with the literal — the share guard is what catches garbage when the object member has only inherited fields (`colorRGB`) or the data member is all-optional (`ItemLocationData`); without it a foreign literal would vacuously satisfy a member. A `table<string_literal_keys, V>` config dict still reaches a data class via the `table<keys,V> → @class` key-coverage path in `is_table_subtype_impl`. The map is a small documented constant (the data/mixin split only exists for Color and ItemLocation in Ketho's annotations); extend it if more split types appear.

### `@return self<X>` (re-parameterized self return)
`Function.returns_self_type_args: Option<Vec<AnnotationType>>` alongside `returns_self = true`. Parsed from `AnnotationType::Parameterized("self", args)` in `build_ir.rs` and `pre_globals/mod.rs`. At call resolution (`resolve_call.rs` `returns_self` branch), the args are resolved + generic-substituted and stored in `call_type_args` so the call result displays as `Receiver<X>` regardless of the receiver's actual type args. If a type arg uses `T!` (NonNil), nil is stripped from the substituted type after generic substitution (e.g. `@return self<T!>` on `Publisher<string?>` yields `Publisher<string>`). Plain `@return self` (without type args) also propagates the receiver's type args into `call_type_args` so that chained calls can resolve class type params from intermediate results. Hover/signature formats it via `Self::self_return_text(func)` in `queries/format.rs`.

### `@alias Foo<K,V> V[]` (parameterized alias)
`AliasDecl` has `type_params: Vec<String>`. Stored in `ir.parameterized_aliases` / `ext.parameterized_aliases` as `(Vec<String>, AnnotationType)` (type params + raw body). When `Parameterized("Foo", args)` is encountered during resolution, `substitute_alias_type_params()` replaces type param names in the body with concrete args, then the substituted type is resolved normally. Supports colon syntax (`@alias Foo<K,V>: V[]`) and spaces in type params (`@alias Foo<K, V>`). Type params may carry constraints (`@alias Box<T: Frame> { value: T }`): the constraint is split off at parse time so the stored param name stays bare (`T`), and `AliasDecl.type_param_constraints` (parallel `Vec<Option<String>>`, mirroring `ClassDecl`) records the raw constraint string. `undefined_doc_class.rs` validates the constraint type name (`undefined-doc-name`). Constraints are pre-parsed and stored at registration time as `Vec<Option<(String, AnnotationType)>>` into `Ir::parameterized_alias_constraints` / `PreResolvedGlobals::parameterized_alias_constraints` (the latter is `#[serde(skip)]` — populated at runtime during workspace scanning, not from the stub blob, so no `BLOB_VERSION` bump; stub aliases carry no constraints). The pre-parsed `AnnotationType` avoids re-parsing the same constraint string at every use site. They are **enforced at every annotation use site** by `Ir::check_parameterized_alias_constraints` (called from the central `check_annotation_type_names` recursion, so `@type`/`@param`/`@return`/`@field`/`@cast`/alias-body positions are all covered): each `Foo<Arg>` type argument must satisfy its param's constraint (resolved via `Ir::resolve_annotation_type_for_check`, the shared core used by both alias and class constraint checking), emitting `generic-constraint-mismatch`. Args that are unresolved type variables/generics (e.g. an alias body forwarding its own param, or a function generic flowing into an alias arg) are skipped — no false positives. Aliases are still purely *substituted* (there's no call site like classes have), so this annotation-walk is the **only** place alias type-arg bounds are checked; the constraint is not re-enforced after substitution.

### Deferred constructor self-field type args (cross-file generic recovery)
A defclass `__init` self-field whose RHS is a chained/generic call the coarse cross-file scan can't resolve (e.g. `self._manager = UIManager.Create(state):SuppressActionLog(...)`) is typed `any` by `scan_defclass.rs`, losing the `<State>` type argument. The per-file engine *does* resolve it, so the precise type args are harvested lazily — same pattern as `@creates-global` deferred globals. Detection: `extract_self_fields_inner` records the RHS call's full byte range on `SelfFieldEntry.deferred_call_range` (only when inferred type is `any` and RHS is a funcall) → `ClassDecl.deferred_field_call_ranges` (field → range, `#[serde(skip)]`). `build_on_stubs::finish()` builds `PreResolvedGlobals::deferred_field_type_args` (`(class, field) → DeferredFieldTypeArgs { path, call_range }`, plus `_by_path` and a `deferred_field_type_args_cache` RwLock memo — all `#[serde(skip)]`, no blob impact). Harvest: `deferred.rs::resolve_deferred_field_type_args()` re-analyzes the defining file, matches `Expr::FunctionCall.call_range == call_range` (ret_index 0), reads the call's type args via `get_type_args_for_expr`, and lifts them to ext space (skipping `any`). Applied lazily in `get_expr_type_args`'s `FieldAccess` branch when the field's own lookups yield no args, caching into the per-file `field_type_args_cache` so both resolve-time (`class_type_param_subs` for `expression<T & ...>`) and query-time hover see the recovered `<State>`. `deferred_field_call_ranges` is positional and **excluded** from `class_semantic_eq` (a stale offset just degrades to the incomplete-context suppression until the next real rebuild). The harvest only *adds* type args; the `expression_type.rs` incomplete-context suppression is the fallback when it can't resolve. **Cache discipline in `get_expr_type_args`'s `FieldAccess` branch:** only *stable* sources are written to `field_type_args_cache` — the field's `Parameterized` annotation and the deferred-harvest result. Type args read from `call_type_args` (the field's own assignment expr / `extra_exprs`) are returned **uncached**, because a generic bound from a builder-built argument (`Mgr.Create(state)` where `state` is `@return built: Base`) refines from the declared base to the precise built subtype only late in the fixpoint; caching the early base binding poisons every later read (manifests as `expression<T & ...>` resolving `T` to the base, or `keyof T` constraint false positives). `get_type_args_for_expr` (the query-time twin) already re-reads `call_type_args` on a cache miss, so hover stays consistent.

## Diagnostics — module reference

Per-module reference for the trait-based diagnostic passes under `src/diagnostics/`. The architecture (the `DiagnosticDef`/`DiagnosticPass` traits, `run_all()` orchestration, the `CATALOG`/`DEFAULT_DISABLED_CODES`/`CODE_ALIASES` tables) and the steps to add a new diagnostic live in [CLAUDE.md — Diagnostics](../CLAUDE.md#diagnostics); the authoritative code list is the `CATALOG` array in `src/diagnostics/mod.rs`, and the user-facing reference is [`docs/reference/diagnostics.md`](../docs/reference/diagnostics.md). This section captures the per-module behavior and false-positive rationale that isn't in those places. (~40 modules implementing `DiagnosticPass` or exporting helpers.)

**Type system checks:**
- `type_mismatch.rs` — argument type mismatches against `@param` (`type-mismatch`)
- `return_mismatch.rs` — return type mismatches against `@return` (`return-mismatch`)
- `field_type_mismatch.rs` — field assignment type mismatches against `@field` (`field-type-mismatch`)
- `assign_type_mismatch.rs` — reassignment type mismatches against `@type` (`assign-type-mismatch`)
- `grouped_return_mismatch.rs` — return values not matching any tuple-union `@return` case (`grouped-return-mismatch`)
- `generic_constraint_mismatch.rs` — generic type constraint violations at call sites and class type params (`generic-constraint-mismatch`)
- `param_constraint_mismatch.rs` — `@requires T: Constraint` violations: a method call where the receiver's class type-param binding doesn't satisfy the constraint (`param-constraint-mismatch`)
- `missing_return_value.rs` — return statements with fewer values than `@return` (`missing-return-value`)
- `missing_return.rs` — functions missing return statements (`missing-return`)
- `missing_fields.rs` — missing required fields when constructing `@class` tables (`missing-fields`)

**Function/call checks:**
- `call_arity.rs` — argument count validation: `redundant-parameter` (extra args) and `missing-parameter` (insufficient args), handles method calls, varargs, optional params, and projected arity from `params<F>`. **Skips field/method calls whose receiver is a multi-type union** (`receiver_is_multi_type_union`: 2+ table-bearing alternatives, e.g. an AceGUI `WidgetA | WidgetB | ...` handle): the runtime type isn't statically known and the accessed function resolves to whichever member declares it — when only one does, the callee collapses to that member's (often stub-incomplete) signature, so arity-checking it is a false positive. Methods present on several members already resolve to `ValueType::Union` and are skipped at the `func_idx` match anyway; this extends the same leniency to the single-member case. **Also skips functions in `ext.conflicting_arity_funcs`** — a namespaced function (`ns.A.B`) or class method merged from multiple cross-file definitions that disagree on arity, the signature of a flavor-split addon defining the same function with different param counts in mutually-exclusive `Source_*` dirs. The workspace merges all files and keeps only the first definition's signature (`pre_globals` drops the unannotated duplicate, recording the function in `conflicting_arity_funcs` via `duplicate_def_arity_conflicts` at the Method-dedup skip branch in both `mod.rs` and `build_on_stubs.rs`); without the skip, `call_arity` would flag the *other* flavor's call sites against that one definition. The set is `#[serde(skip)]` and recomputed per-workspace (mirrors `deferred_returns`), so an arity edit triggers a rebuild (`global_semantic_eq` compares `params`) that regenerates it.
- `cannot_call.rs` — calling a value whose type is not callable (`cannot-call`): warns on non-function, non-constructor types (number, string, boolean, nil, bare table, etc.)
- `invalid_op.rs` — arithmetic, concatenation, or ordered comparison on incompatible types (`invalid-op`): catches `+` on strings (suggests `..`), arithmetic on booleans/nil/tables, concatenation on non-stringable types, ordered comparisons (`<`, `>`, `<=`, `>=`) on nil/boolean/function/mixed types
- `destructure_arity.rs` — destructuring more variables than a function returns (`unbalanced-assignments`): handles annotated arity, inferred body returns, overloads, `returns<F>` projections, and vararg return suppression. **Dynamic-trailing lower bound**: for a *body-inferred* function (no authored `@return`), when the last return slot resolves to `any` — a value the inference couldn't pin to a concrete type (a dynamically-keyed `table<K,V>` access like `return id, t[15], t[17] or 0`, an untyped-receiver field, etc.) — the syntactic value count is only a *lower* bound, so the over-destructure check backs off (`return_arity` returns `None`). This covers both the same-file `func.rets` path (via `return_type_at_slot` on the max slot) and the cross-file deferred-harvest path (`func.return_annotations.last()`, gated on `ext.deferred_returns` membership + no non-return-only overloads so an authored `@return any` is excluded and still warns). Confidently-typed trailing returns (literals, classes, primitives) keep the warning; the inferred *type* display is untouched (only the diagnostic relaxes)
- `discard_returns.rs` — ignored `@nodiscard` return values (`discard-returns`)
- `multi_return_projection.rs` — `returns<F>` truncation when F has >1 return annotation (`multi-return-projection`)

**Variable/field/global checks:**
- `undefined_global.rs` — references to unresolved global names (`undefined-global`)
- `undefined_field.rs` — accessing nonexistent fields on `@class` tables (`undefined-field`); also fires on **module-private record tables** — a `local X = {}` whose complete field set is statically known. Gated by `collect_pure_record_symbols` (single-version local with a `{}` constructor RHS, never referenced bare/escaping) + `is_closed_record` (no `key_type`/`value_type`/metatable/dynamic bracket write/etc.), so parameter-inferred records, the addon namespace, globals, maps, and escaping tables are excluded. **Membership-test suppression** (`collect_membership_suppressions`, a syntactic AST pre-pass returning the set of field-token start offsets to skip): a field read used as a *defensive existence check* — and the access it guards — is not flagged. `analyze_bool` walks boolean/value expressions tracking a `boolean_ctx` flag (true when the value is consumed purely as a boolean: a condition, a `not` operand, a non-final `and`/`or` operand, or an `==`/`~=` comparand). Two shapes are suppressed: (1) **probe reads** — a field-read chain in boolean context (every field token in the chain is suppressed, since the probe dereferences every level); (2) **guarded accesses** — an access whose dotted path *exactly* matches a path proven present by a preceding probe (the `and`-RHS, the `or`-RHS after `not X`, or the `if`/`while` body). `or` operands are all treated as optional reads (the fallback idiom); `and`/`if`-bodies guard via every length-≥2 prefix of the probed chain so a re-read of an inner segment (`if a.b.c then a.b ...`) is covered, while *exact*-match (not prefix) on the guarded side keeps a *deeper* access on a now-known field (`if o.cfg then o.cfg.typo end`) checked. An `==`/`~=` comparison suppresses its field-chain operands in **both** boolean and value context (`if x.f ~= nil then` and `local h = x.f ~= nil` are the same existence check) — the comparison arm is not `boolean_ctx`-gated, and the value-context dispatch covers standalone comparison roots (`is_eq_comparison_node`) as well as `and`/`or`/`not` roots. Method/function *calls* in boolean position are never probe reads (a missing method errors at runtime) — they are only suppressed as guarded accesses. The suppressed offsets equal the IR `FieldAccess.field_range` start; the check is applied **only on the `@class` emit path**, *after* the closed-record fallback's `continue` — a closed module-private record has a fully-known field set that can't grow at runtime, so an unknown field there is a typo even inside a condition and is still flagged
- `unused_local.rs` — unreferenced local variables (`unused-local`, HINT)
- `redefined_local.rs` — same-scope local variable redefinition (`redefined-local`)
- `shadowed_local.rs` — local variable shadows a variable from an outer scope (`shadowed-local`, HINT)
- `duplicate_index.rs` — duplicate keys in table constructors (`duplicate-index`)
- `duplicate_set_field.rs` — setting a field already set on `@class` tables (`duplicate-set-field`)
- `inject_field.rs` — setting undeclared fields on `@class` tables (`inject-field`, HINT). **Skips writes on a multi-type union receiver** (`union_receiver_declares_field`) when the field is declared (own or inherited) on any union member: a `FieldAssignment` carries only a single `table_idx` — the *first* union member (`resolve_deferred_field_assignments` picks the first `Table` in the union) — so a field declared on a *later* member (e.g. `---@param x A|B|C` writing a field declared on `C`) would otherwise be reported as injected into the first. Mirrors `call_arity`'s `receiver_is_multi_type_union` leniency. The receiver's full union type is recovered by resolving the write's `root_symbol` (every `field_assignments` entry's receiver is its root symbol — deep `a.b.c = …` chains never reach this list); suppression uses `class_has_annotated_field` (author `@field` only), since the deferred resolver injects the field onto the first member *un*annotated, so a bare existence check would also swallow a genuinely-injected control field
- `create_global.rs` — implicit global creation via assignment or function definition (`create-global`)
- `mixed_enum_values.rs` — `@enum` with mixed number/string values or unsupported value types (`mixed-enum-values`)
- `class_shadows_builtin.rs` — workspace `@class` that declares its own explicit `@field`s reuses a built-in WoW API stub class name (`class-shadows-builtin`); bare stub-name reuse with no explicit `@field` (mixins, namespaces, frames) is not flagged

**Access control:**
- `access.rs` — `@private`/`@protected` visibility violations (`access-private`, `access-protected`)
- `need_check_nil.rs` — field/method access on possibly-nil values (`need-check-nil`, default-disabled)
- `nil_index.rs` — possibly-nil table key in bracket access (`nil-index`)
- `nil_table_key.rs` — nil in table key type annotations, e.g. `table<string?, V>` (`nil-table-key`)
- `wrong_flavor_api.rs` — calls to APIs not available in project-declared flavors (`wrong-flavor-api`)
- `callback_events.rs` — string-literal event names passed to a callback-registry consumer (`:RegisterCallback("…")`/`:TriggerEvent("…")`, marked `@callback-event-arg`) that the registry never declared via `GenerateCallbackEvents` (`unknown-callback-event`, default-disabled). Purely syntactic via `AnalysisResult::callback_event_set_for_string` (shared with completion): the registry receiver usually types as bare `table` cross-file, so the receiver path/method/arg-index are read from the AST and the receiver is canonicalized (`canonicalize_member_path`) to match the producer site. Suppressed unless the registry's event set is `complete`.
- `expression_type.rs` — undefined variables and return type mismatches in `expression<C, R>` string arguments (`undefined-field`, `type-mismatch`). Exports the shared, closure-based `infer_expression_type(tree, name_type)` result-type inference (parses `return <expr>` and walks operators/identifiers); `resolve_call.rs` reuses it to infer a generic `R` from the expression body and bind it into the call's generic substitutions when `R` is one of the method's `@generic` params (so `@return Schema<R>` becomes `Schema<number>` etc.). **Incomplete context degradation:** when `C` contains a `self`/generic-param member that can't be bound to a concrete table (`resolve_expression_tables` returns `incomplete = true`), `ExpressionArg.context_incomplete` is set and this pass skips — the resolved members (e.g. a `Builtins` half) still power hover/completion, but emitting `undefined-field` would be a false positive on names the unbound member would supply. This is the fallback when the deferred field-type-args harvest can't recover the binding.

**Annotation validation:**
- `function_annotation_checks.rs` — comprehensive function-level annotation validation: `@param` name mismatches (`undefined-doc-param`), duplicate `@param` (`duplicate-doc-param`), `@return` type resolution, `@overload` type resolution, `@generic` on class methods (`redundant-class-generic`), `params<F>` position/shape validation, `@requires` param-name-must-be-a-class-type-param + constraint type resolution (`malformed-annotation`/`undefined-doc-name`), and `@return self<X>` type-arg arity vs the class's type params (`malformed-annotation`)
- `annotation_metadata.rs` — annotation comment scanning: duplicate `@constructor` (`duplicate-constructor`), `@constructor` return validation (`constructor-return`), `@builds-field` without `@return self` (`builds-field-not-self`), `@return ClassName` instead of `@return self` (`return-self-class-name`, HINT), bare `return` with all-optional `@return` types (`implicit-nil-return`, HINT), duplicate `@field` (`duplicate-doc-field`), duplicate `@alias` (`duplicate-doc-alias`), and the **`deprecated` call-site check** ("Part 3" of `run()`: a call whose callee resolves to a `@deprecated` `Function`). The deprecated check is **flavor-aware** — gated by `crate::flavor::deprecation_suppressed(analysis.addon_flavors, func.flavors)` so a retail-sourced deprecation that's still the live API on a flavor the addon targets (e.g. `GetMerchantItemInfo` on Classic) is not flagged. `addon_flavors` (from `config.addon_flavors_for`) is the addon's declared breadth incl. the `.toc` `## Interface:` fallback; `0` (no flavor signal) disables suppression and preserves prior behavior.
- `malformed_annotation.rs` — unknown or incomplete `---@` annotations (`malformed-annotation`)
- `doc_field_no_class.rs` — `@field` annotations not preceded by `@class` (`doc-field-no-class`)
- `doc_func_no_function.rs` — function-level annotations (`@param`, `@return`, `@overload`, `@generic`, `@nodiscard`, `@deprecated`, `@constructor`, `@builds-field`, `@built-name`, `@built-extends`, `@flavor-narrows`, `@type-narrows`, `@narrows-arg`, `@defclass`) not attached to a function definition (`doc-func-no-function`)
- `undefined_doc_class.rs` — undefined class names in `@class Foo: Parent` inheritance, circular inheritance chains, and inheriting from primitive/literal types (`undefined-doc-class`, `circle-doc-class`, `invalid-class-parent`)
- `undefined_doc_name.rs` — undefined type names in annotations (`undefined-doc-name`)
- `unknown_diag_code.rs` — unknown code in `@diagnostic` directives (`unknown-diag-code`)
- `incomplete_signature_doc.rs` — functions with partial `@param`/`@return` annotations (`incomplete-signature-doc`, HINT, default-disabled)
- `missing_annotation.rs` — non-file-local functions (globals + cross-file table methods/fields) entirely missing `@param`/`@return` (`missing-param-annotation`/`missing-return-annotation`, HINT, default-disabled). Unlike `incomplete-signature-doc`, fires even with zero annotations, but is scoped to functions reachable beyond their file. Skips `local function` and anonymous literals. **Bare `function foo()`**: `bare_name_is_file_local()` skips it only when `foo` rebinds a forward-declared local (detected via `is_local_declaration_site` on the binding symbol's first version). **Dotted/colon methods (`A.B`/`A:B`)**: escape is decided by `escaping_methods()`, which reuses `scan_file_globals()` as the source of truth — a method is checked iff the workspace scan registers it as a cross-file symbol (root is a global/`@class`/addon-ns, or a local table assigned onto the addon ns via `ns.Field = LocalTable`), matched back to the per-file `Function` by `def_node.start` byte offset. Methods on file-private local tables are therefore skipped. The escaping-offset set is built lazily on the first dotted candidate.

**AST & style checks:**
- `ast_checks.rs` — AST-traversing pass consolidating: empty blocks (`empty-block`, HINT), unbalanced assignments (`unbalanced-assignments`), redundant values (`redundant-value`), redundant return values (`redundant-return-value`), code after break (`code-after-break`, HINT), unreachable code after return (`unreachable-code`, HINT), count-down loops (`count-down-loop`), unused functions (`unused-function`, HINT), redundant return (`redundant-return`, HINT), deprecated symbol usage (`deprecated`)
- `trailing_space.rs` — lines ending with whitespace (`trailing-space`, HINT)
- `not_precedence.rs` — `not x <cmp> y` parsing as `(not x) <cmp> y` (`not-precedence`, HINT)
- `redundant_logical.rs` — redundant `or`/`and` where LHS truthiness/falsiness makes RHS unreachable, plus `and` whose LHS is always truthy (the operator is a no-op). The `x and y or z` Lua ternary idiom suppresses only the outer `or`; the inner `and` is still flagged when `x` is statically truthy. (`redundant-or`, `redundant-and`, HINT)
- `redundant_condition.rs` — `if`/`elseif`/`while`/`repeat...until` condition whose type is always truthy or always falsy (`redundant-condition`, HINT, default-disabled)
- `unused_vararg.rs` — functions declaring `...` but never referencing it (`unused-vararg`, HINT, default-disabled)
- `unused_function.rs` — cross-file unused function/method detection (`unused-function`, HINT): tracks references via call_resolutions + field-access token walk with inheritance; skips library files, `_`-prefixed names, and interface methods (2+ tables sharing the same method name)

**Unknown-type diagnostics (strict typing, all default-disabled):**
- `unknown_param_type.rs` / `unknown_return_type.rs` / `unknown_local_type.rs` / `unknown_field_type.rs` — sites whose type couldn't be inferred (`unknown-param-type`, `unknown-return-type`, `unknown-local-type`, `unknown-field-type`, HINT). See [Unknown-type diagnostics (strict typing)](#unknown-type-diagnostics-strict-typing).

**Special:**
- `safety-limit` (ERROR) — emitted when analysis is incomplete due to safety limits

Some modules are "hybrid": they implement `DiagnosticPass` for the post-analysis phase AND export `pub(crate)` helper functions called from `build_ir.rs` / `resolve.rs` during IR construction.
