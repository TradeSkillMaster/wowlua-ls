# wowlua_ls — WoW Lua Language Server

A Language Server Protocol implementation for Lua (World of Warcraft API dialect). Provides hover, go-to-definition, completion, signature help, find references, rename, and diagnostics.

For deep architecture internals (type inference, narrowing, generics, builder pattern, cross-file references, metatable inference, flavor filtering, etc.), see [ARCHITECTURE.md](.claude/ARCHITECTURE.md).

For Neovim diagnostic integration details (push/pull namespaces, `workspace_diagnostics` flag, line-shifting, edit zone handling), see [NEOVIM_DIAGNOSTICS.md](.claude/NEOVIM_DIAGNOSTICS.md).

## Architecture

### Source files
- `src/main.rs` — CLI entry point: `evaluate` subcommand, `test-query` subcommand (hover/def/sig/completions/diagnostics), `dump-types` subcommand (hover regression baselines), `doc` subcommand (markdown API doc generation), otherwise starts LSP
- `src/doc_gen.rs` — Documentation data model: `DocNamespace`, `DocDefine`, `DocField`, `DocParam` structs, standalone type formatter operating on `PreResolvedGlobals`, class/field iteration with visibility and source locations
- `src/doc_gen_md.rs` — Markdown documentation renderer: takes `Vec<DocNamespace>` and produces VitePress-compatible `.md` files (one per class + `index.md`)
- `src/types.rs` — IR type definitions: `ValueType`, `Expr`, `Symbol`, `Scope`, `Function`, `TableInfo`, `FieldInfo`, `EnumKind`, deferred check structs, index aliases, `EXT_BASE`
- `src/analysis/` — Core per-file analysis engine (`Analysis` struct):
  - `mod.rs` — `Ir` struct definition, scope-chain walking helpers, two-tier lookups, core helpers
  - `prescan.rs` — Phase 0: class/alias pre-scan, annotation type resolution, generic inference
  - `build_ir.rs` — Phase 1: AST walk, scope/symbol/function/table creation, correlated return inference
  - `lower_expression.rs` — Expression lowering from AST to IR `Expr`: literals, identifiers (`NameRef`, `DotAccess`, `BracketAccess`, `MethodCall`), function calls, binary ops, table constructors, inline `@as` casts
  - `narrowing.rs` — Type narrowing from control flow guards: `GuardNarrow` enum, `OrTermEffect`, flavor narrowing detection, `@flavor-narrows`, type filter/strip for scope-specific refinement
  - `resolve.rs` — Phase 2: fixpoint type resolution loop, expression resolver, backward param-type inference
  - `resolve_call.rs` — Function call resolution: `CallSiteInfo`, argument count/type checking, return type determination, overload matching, generic binding
  - `checks.rs` — Diagnostic check orchestration via `run_diagnostics()`, name-token collection for access diagnostics
  - `queries/` — LSP query methods, split per feature (all `impl AnalysisResult` blocks over the `pub(crate)` fields defined in `mod.rs`). `mod.rs` holds shared imports/re-exports (`ReferenceTarget`, `HighlightKind`, `CallSiteResult`, `OutgoingCallResult`, `DATA_REPLACE_START/END`) and the cross-module helpers (`return_type_at_slot`, `format_vararg_return`, `format_vararg_param`). Submodules: `nav.rs` (token/symbol/field resolution helpers like `find_symbol_at`, `find_field_at`, `scope_at_offset`), `format.rs` (type/signature formatting), `hover.rs`, `definition.rs`, `completion.rs`, `signature.rs`, `references.rs`, `rename.rs`, `highlights.rs` (document highlights), `inlay_hints.rs`, `code_lens.rs`, `call_hierarchy.rs`, `document_symbols.rs`, `embedded_strings.rs` (event/`expression<C,R>` string hover/completion/definition)
  - `semantic_tokens.rs` — LSP semantic-token classification. Narrow by design: (1) walks bare `Name` tokens (skips field/method *access* and parameters) and emits a `function` token when the symbol resolves to a function value; (2) classifies the name chain of a function/method *definition* header (`collect_function_def_name_tokens`) — the root receiver as `class` when it resolves to a `@class` type (`value_type_is_class`, via `symbol_resolved_type_at`), intermediate field/accessor segments as `property`, and the defined name as `method` (colon-defined) or `function` (dot-defined). A simple `function foo()` has no chain wrapper and falls to (1); a non-class receiver's root token is left to the grammar while intermediate and terminal segments are still classified. **Middle segments are `property` syntactically, never by resolved type** — a transparent `@accessor` resolves to its owning class type (that's how `self` is typed in the method body), so a type-based check would mis-color the accessor as `class`; only the root's type is consulted. This is the case a type-blind TextMate grammar can't get right in a dotted chain (it can't tell a class receiver from a namespace or an accessor field), so the analysis owns it. Everything else is left to the editor's built-in Lua grammar. Modifiers: `defaultLibrary` for stub symbols (via `is_stub_symbol()` — `idx - EXT_BASE < stub_symbols_end`, boundary captured at `load_precomputed_stubs()` time), `deprecated` when the resolved function is `@deprecated` **and not flavor-suppressed** — it reuses the same `is_stub_function(f) && crate::flavor::deprecation_suppressed(addon_flavors, func(f).flavors)` gate as the `deprecated` diagnostic, so a WoW-stub API that's still live on a flavor the addon targets is neither warned nor struck through (the two must agree at the same position). Legend is the `SEMANTIC_TOKEN_TYPES` (`function`, `variable`, `keyword`, `number`, `operator`, `class`, `property`, `method`) / `SEMANTIC_TOKEN_MODIFIERS` arrays; encoded into LSP wire format by `main_loop/semantic_token_encoding.rs::encode_semantic_tokens`.
- `src/pre_globals/` — Precomputed global type database:
  - `mod.rs` — `PreResolvedGlobals` struct, 5-phase build from WoW API stubs, type parameter substitution, class/alias/function registration. **Multi-definition go-to-definition**: the per-key `symbol_locations`/`class_locations`/`alias_locations` maps keep one *primary* location each (last write wins), while the runtime-only (`#[serde(skip)]`, no blob impact) `symbol_locations_by_name`/`class_locations_all`/`alias_locations_all` maps accumulate *every* distinct workspace definition site per name (deduped by `(path, start)`). These are populated in `build_on_stubs` from `ws_globals`/`ws_classes`/`ws_aliases` independent of the name-dedup that registration applies (global functions are deduped to the first occurrence at registration, so the second file's site is only recoverable here). `analysis/queries/definition.rs::definitions_at`/`type_definitions_at` return the primary first then the extras; the single-result `definition_at`/`type_definition_at` are `.into_iter().next()` wrappers (primary == old behavior, so existing `def:`/`typedef:` tests are unaffected).
  - `build_on_stubs.rs` — `BuildOnStubsContext` for workspace incremental builds on precomputed stubs: scope/symbol/function/table cloning, type parameter substitution, field resolution. **Built-in stub name reuse is ADDITIVE**: a workspace `@class` that collides with a stub class name (`PreResolvedGlobals::stub_class_names`, populated at load from `PrecomputedStubs.stub_classes`) merges its fields onto the existing stub `TableInfo` via `populate_class_fields` — `register_classes_and_aliases` never replaces the stub table. This deliberately preserves the common *augmentation* idiom (`@class ScrollBoxListMixin : CallbackRegistryMixin` whose `.Event` enum is synthesized onto the stub overlay by `@generates-events`; a namespace/frame typed as a builtin; a library adding a field) — a destructive replace would strip the stub's (and any synthesized) fields workspace-wide and there'd be no warning on suppressed-origin files. The *fresh-record-collides-with-a-builtin* case (the motivating false `missing-fields`, e.g. `@class CVarInfo` with its own `@field`s used as `@type table<K, CVarInfo>`) is handled WITHOUT touching the stub table: (a) per-file `prescan` skips importing stub fields into a local `@class` that declares its own explicit `@field` contract (`!declared_field_names.is_empty()`), so same-file `local` constructors check the local shape; (b) `diagnostics::missing_fields::is_required_contract_field` scopes the required-field set to `declared_class_fields[name]` (the workspace's explicit `@field`s) for any class in `stub_class_names`, so the cross-file `@type table<K, Class>` case (which resolves the value class against the *external*, additively-merged table) ignores the stub's required fields. The discriminator throughout — warning, prescan, missing-fields — is **explicit `@field` declarations** (`declared_field_names` / `declared_class_fields`), NOT inferred constructor keys, so namespaces/frames/mixins reusing a stub name are neither warned nor scoped. The `class-shadows-builtin` diagnostic warns only at a reuse that declares explicit `@field`s.
- `src/annotations/` — Annotation system (types, parsing, cross-file scanning):
  - `mod.rs` — Core types (`AnnotationType`, `ParamInfo`, `Visibility`, `ClassDecl`, `AliasDecl`, `AnnotationBlock`), comment extraction (`extract_annotations`), full-file class/alias discovery (`scan_all_annotations`), line-level `@tag` dispatch (`parse_annotation_lines`), tuple-union lowering (`lower_tuple_form_cases`), re-exports from submodules
  - `annotation_types.rs` — Type expression parsing: `parse_type()`, `parse_overload()`, `parse_return_line()`, `format_annotation_type()`, `substitute_alias_type_params()`, `match_projection()`, and internal helpers (`split_at_top_level`, `extract_type_prefix`, etc.)
  - `annotation_scanning.rs` — Shared types (`ExternalGlobal`/`ExternalGlobalKind`/`FieldValueKind`), constants (`ADDON_NS_NAME`), shared helpers (`extract_type_annotation_for_assign`, `extract_inline_type_annotation`, `is_select_varargs`), `scan_method_typed_self_fields()`, `scan_diagnostic_directives()`, type conversion (`resolve_annotation_type`, `reduce_to_fun_alias`). **Self-field scanners (3 tiers, all gated on recognized local `@class` receivers):** `scan_method_typed_self_fields` (`self.x = v ---@type T`), `scan_method_funcall_self_fields` (`self.x = Foo()` → `FieldValueKind::FunctionCall`, resolved cross-file by `build_on_stubs`), and `scan_method_bare_self_fields` (literals/params/tables → inferred). A *chained* funcall (`self.x = Foo():Bar()`, `self.x = LibStub("X"):New()`) has a call as its receiver, so the funcall scanner (which needs a plain-name callee) can't resolve it — `funcall_has_chained_receiver()` makes the two scanners' coverage disjoint: the funcall scanner skips chained receivers and the bare scanner registers them existence-only as a bare `table` (not `any` — `any` would propagate through `self.x.y and z` as `boolean?`, causing a spurious `return-mismatch`; a truthy `table` short-circuits `and` to the RHS type). Self-fields on *mixin* tables (no local `@class`) are out of scope here — they're handled by `scan_globals.rs`'s descendants pass.
  - `scan_globals.rs` — `scan_file_globals[_with_synth]()`, `scan_dynamic_global_prefixes()` (detects `_G["PREFIX"..k]` patterns for auto-allowed globals), and `scan_created_globals()` (detects `@creates-global` calls; see the convention note below). The coarse global scan is **detection-only** for deferred functions: `build_func_external()` records a function's location, annotation-derived signature, and deferred-ness (no `@return`), but does **not** infer body-derived return types or synthesize correlated return-only overloads. All body-derived returns/overloads are produced lazily by the per-file harvest (`analysis/deferred.rs::resolve_deferred_sig`), the single source of truth. The `_with_synth` variant's `correlated_return_overloads` parameter is retained for caller compatibility but no longer affects scanning (the harvest reads the flag from the file's config directly). The main statement loop only walks top-level + control-flow statements (`collect_statements_recursive` does **not** descend into function bodies), because the coarse scan can't reliably *type* values produced inside functions (locals, `Mixin`'d frames, chained builder results). A final pass over `root.descendants()` registers the **existence** of namespace/`@class` fields assigned *inside* function bodies, of multi-target assignments (`a.x, a.y = f()`) anywhere, and of bare single-name globals created via `X = ...` *inside* function bodies (e.g. a saved-variable global initialized only in an event/initializer function) — all gaps the main loop leaves. Field/multi-target writes emit existence-only `TableField(_, _, FieldValueKind::Unknown)` entries (no type); the build's Unknown-field pass registers an otherwise-absent field as a bare `table` (no annotation → suppresses cross-file `undefined-field` on reads without fabricating a type that would cause `field-type-mismatch`) and skips fields that already carry a precise type or methods. A function-literal RHS (`self.cb = function() end`) instead emits `FieldValueKind::Function` so the field registers callable (avoids a false `cannot-call` when it is later invoked). A **forwarded** RHS — another field or a parameter (`ns.Foo = current.func`, `ns.Cb = callback`), i.e. an `Expression::Identifier` — on a *non-`self`* namespace/`@class` field emits `FieldValueKind::MaybeCallable`, materialized by the Unknown-field pass as `ValueType::callable_or_unknown()` (the `function & table` intersection): the forwarded value may be a callable, so calling the field must not false-positive as `cannot-call`, while the intersection keeps reads as permissive as a bare table (callable, truthy, assignable to/from any table — see the `callable_or_unknown` doc-comment). This is **scoped to non-`self` roots** (the reported namespace-field case): a `self.` field is left `Unknown`/bare-`table`, because self data-fields are usually data, also collect a per-file overlay / self-field-scanner type, and forcing them callable would union `function & table` with that real type and turn clean reads into `type-mismatch`. A function-*call* RHS (`CreateFrame(...)`) likewise stays a bare `table` so a competing concrete type (e.g. a merged-in `Frame`) can still subsume it in a union via `make_union`. The descendants pass also handles `self.field = ...` writes by resolving `self` to the nearest enclosing *named* colon method's receiver (skipping anonymous callbacks, which close over the outer `self`): an addon-namespace receiver routes to `ADDON_NS_NAME` (forwarded to its `@class` by `merge_addon_ns_into_classes`), a function-local receiver is skipped (can't be a cross-file mixin), and anything else routes to the receiver's single (leaf) name — but **only when that leaf unambiguously identifies the table**. A *single-name* receiver (`function Mixin:M()`, the method's own root) is the non-`@class` *mixin*-table pattern (promoted to a class only via an XML `mixin=` / `Mixin()` reference) and routes to the receiver name; a *deeply-nested* receiver (`function A.B.C:M()`) is keyed by its bare leaf, which can collide with an unrelated same-named global, so it is registered only when that leaf names a known `@class` (via `var_to_class`, matching the typed/bare scanners) and is **skipped otherwise** rather than misattributed. This is the only path that tracks self-fields on mixin tables; the typed/funcall/bare self-field scanners (`annotation_scanning.rs`) gate on recognized local `@class` names and so miss them. For a *redeclared* local `@class` receiver those scanners (which feed `ClassDecl.fields`) own it instead, since the existence-only `Unknown` entry here would be dropped by the prescan overlay-import filter. In-function single-name globals emit existence-only `Variable(FieldValueKind::Unknown)` entries so cross-file *reads* don't false-positive as `undefined-global` (no fabricated type). To avoid leaking function-scoped locals as phantom globals, `collect_all_local_names()` gathers every name bound as a local anywhere in the file (`local`/`local function`/params/for-vars, including inside function bodies) and a bare write to such a name is treated as a local reassignment and skipped; an explicit `_G.X = ...` write bypasses this and always registers.

This keeps `check` results invocation-independent (subdir scan == root scan filtered to that subdir). Isolated: `diagnostics.*`, `globals.*`, `framexml`, `flavors`, `inference.*`, `ignore`, `plugins` (filtered per-file by source path in `PluginEngine::run_plugins`). Editor-preference settings that don't affect diagnostics (`hint.*`, `code_lens.*`, `completion.snippets`, `editor.auto_insert_end`) keep the inherited `deepest_bool` behavior. Config discovery only walks *downward* from the scan root (no walk-up), so the "no nearest config" case is intentionally best-effort. When adding a new `*_for` accessor, slot it deliberately into the isolate (`nearest_*`) or inherit (`deepest_bool`) bucket. **`library` is the deliberate exception to isolation**: `ProjectConfigs::is_library()` checks *every* config entry, not just the nearest — relative patterns from any ancestor config apply to all descendants, and absolute external dirs stay workspace-wide via `external_library_dirs()`. This is "inherited downward": marking a subtree as a library means "this whole subtree is vendored third-party code," so a nested config deeper in the tree (e.g. a vendored library's own `.wowluarc.json`, possibly pulled in via a symlink) must not be able to un-mark it. The tradeoff is that this one setting is *not* invocation-independent — a scan rooted *inside* a library subtree won't see the ancestor's `library` declaration. **Built-in default ignore**: `ProjectConfigs::is_ignored()` skips default-ignored directories (the `DEFAULT_IGNORED_DIRS` list, currently just `.github/`) with or without a config. `.github/` holds CI/build tooling run by the standalone Lua interpreter (real `io`/`os` stdlib), never in-game addon code, so analyzing it as WoW Lua only yields false `undefined-global`s. The match is **anchored to the workspace** so a default-ignored segment *above* the scan root can't silently prune the whole project: in the configured branch it runs `is_default_ignored()` (any-component match) on the config-relative remainder (`strip_prefix(config_dir)`), exactly like the `ignore`/`library` patterns; in the no-config branch it matches a directory *entry* by its own name via `is_default_ignored_component()` (so the walk prunes a `.github` entry when it reaches it, but the ancestor segment that appears in every entry's absolute path never matches). Both helpers are free fns near `matches_path_patterns`. Adding a new default-ignored dir = extend `DEFAULT_IGNORED_DIRS`.

### Two-tier index space (EXT_BASE)
External globals (WoW API stubs) use indices >= `EXT_BASE` (1,000,000). Per-file locals use indices < `EXT_BASE`. All lookup functions (`sym()`, `func()`, `table()`, `expr()`) route via `idx >= EXT_BASE` check. This avoids cloning ~9000 external symbols per file.

### IR read-only query surface (post-build consumers)
The `Ir` arenas (`symbols`, `functions`, `tables`, `exprs`, `scopes`) carry load-bearing, convention-only invariants — the EXT_BASE two-tier routing, the symbol/function overlay fallbacks, and scope-chain walking. **Post-build consumers (everything in `src/diagnostics/`) must read the IR through the narrow query surface on `Ir` (and the delegators re-exposed on `AnalysisResult`/`Analysis`), never by indexing the raw arena `Vec`s directly.** The surface is defined in `analysis/mod.rs` next to the existing routing helpers:

- **Indexed routing accessors** (handle EXT_BASE + overlays): `sym(idx)`, `func(idx)`, `expr(idx)`, `try_expr(idx)` (fallible), `table(idx)`, `scope(idx)`, `try_scope(idx)` (fallible).
- **Local-arena iterators** (per-file entries only; external stub entries live in `self.ext` and are deliberately excluded): `local_symbols()`, `local_functions()`, `local_tables()`, `local_exprs()`, `local_scopes()` — each yields `(TypedIndex, &T)` so callers get the correctly-typed local index for free instead of reconstructing `FunctionIndex(usize)` etc.
- **Scope-0 (global/file-level) helpers**: `scope0_local_symbols()` (iterate the file's own scope-0 bindings as `(&SymbolIdentifier, SymbolIndex)`; external `scope0_symbols` are chained explicitly by the `_G`-redirect completion path), `scope0_global_symbol(id)` (local scope-0 lookup with the external `scope0_symbols` fallback, no framexml — encapsulates the manual `scopes[0].symbols.get(..).or(ext.scope0_symbols.get(..))` pattern).
- **Scope-chain walk**: `get_symbol(...)`, `scope_at_offset(...)`, `ancestor_scopes(start)`.
- **Resolved-type cache**: `resolved_expr_cache_get(ExprId) -> Option<&ValueType>` on `AnalysisResult` — avoids raw `resolved_expr_cache[id.val()]` indexing in diagnostics.

The shared `unwrap_to_inner_expr(ir: &Ir, id)` helper in `diagnostics/mod.rs` also takes `&Ir` (not a raw `&[Expr]` slice) so its expr lookups route through `expr()`. Rule of thumb when writing a new diagnostic: iterate user code via `analysis.local_*()`, resolve any index via `analysis.sym/func/expr/table/scope(idx)`, and never write `analysis.ir.symbols[...]` / `.functions[...]` / `.exprs[...]` / `.tables[...]` / `.scopes[...]`. (Method calls like `analysis.ir.expr(idx)`, `.ir.get_field(...)`, `.ir.classes`, and the deferred-check collection fields such as `.ir.call_resolutions` / `.ir.binary_op_sites` are fine — they don't bypass routing.)

**The boundary is now enforced at the type level.** The five arena fields on `Ir` are **private to the `analysis` module** (not `pub(crate)`): builders and queries in descendant modules (`build_ir`, `resolve`, `prescan`, `narrowing`, `queries/*`, …) still touch them directly, but every consumer *outside* `analysis` (diagnostics, `doc_gen`, `stub_gen`, `lsp/main_loop`, `plugins/query`) is forced through the surface methods, which stay `pub(crate)`. `PreResolvedGlobals` got the symmetric treatment: its five arena fields are **private to `pre_globals`** behind matching `pub(crate)` routing accessors (`sym`/`func`/`expr`/`table`/`scope`, fallible `try_expr`/`try_scope`/`try_table`, `iter_symbols`, and the existing `*_len`). The `Ir` routing layer reads its `ext` through those (`self.ext.sym(idx)`, never `self.ext.symbols[…]`), so the `analysis`→`pre_globals` arena dependency is also encapsulated. `doc_gen`'s unit-test helpers build a synthetic `PreResolvedGlobals` via the `#[cfg(test)]` `push_ext_{symbol,function,expr,table}` methods rather than pushing to the (now private) arenas. Do not re-widen any of these fields to `pub(crate)`; add a new accessor on the relevant surface instead.

### Key query functions (in `queries/nav.rs`)
- `find_symbol_at(offset)` — Resolves direct names: gets token at offset → scope lookup → returns `(SymbolIndex, name)`
- `find_field_at(offset)` — Resolves dot/colon chains (`x.y.z`): walks table fields to find the target field's `ExprId`
- `scope_at_offset(offset)` — Finds innermost scope containing offset via `block_scopes` ranges
- `get_symbol(id, scope_idx)` — Walks scope hierarchy upward; at scope 0 also checks `ext.scope0_symbols` (in `analysis/mod.rs`)

### Inlay hints (in `queries/inlay_hints.rs`)
`inlay_hints(tree, config)` collects six categories of inline annotations controlled by `InlayHintConfig` (from `.wowluarc.json` `hint.*` fields, enabled by default unless noted):
1. **Parameter names** (`collect_param_name_hints`) — iterates `call_resolutions`, emits `InlayHintKind::PARAMETER` before each argument. Suppressed when: arg text matches param name (case-insensitive), param is `self`, param is vararg, or param name is empty.
2. **Variable types** (`collect_local_type_hints`) — walks `LocalAssignStatement` nodes, emits `InlayHintKind::TYPE` after each name token. Suppressed when: variable has `@type` annotation, resolved type is `Any`/`Nil`/`Function`, or RHS is a function literal. Per-variable check (not per-statement).
3. **Function return types** (`collect_function_return_hints`) — matches functions by `def_node.start`, emits after the parameter list close paren. Suppressed when: function has `@return` annotation, `returns_self`, or `explicit_void_return`.
4. **For-loop variable types** (`collect_forin_type_hints`) — walks `ForInLoop` nodes, emits after each name token. Suppressed when: variable has `@type` annotation or resolved type is `Any`.
5. **Parameter types** (`collect_param_type_hints`, **disabled by default**, `hint.parameterTypes`) — walks `FunctionDefinition` nodes, emits `InlayHintKind::TYPE` after each parameter name token. Suppressed when: parameter has a `@param` annotation, resolved type is `Any`/`Nil`, or parameter is `self`.
6. **Chained method return types** (`collect_chained_return_hints`, **disabled by default**, `hint.chainedReturnTypes`) — iterates `call_resolutions`, emits `InlayHintKind::TYPE` at the closing `)` of calls whose return value is used as the receiver of a subsequent field/method access. Suppressed when: return type is `Any`/`Nil` or formats to `"?"`. Only intermediate calls in a chain get hints (the final call is covered by variable type hints).

All type hints use `format_type_depth(resolved, 1)` (depth 1) to avoid expanding table fields with newlines — inlay hints show class names only, not field listings.

LSP handler in `main_loop/handlers.rs` converts `InlayHintData` (byte offsets) to LSP `InlayHint` (line/column positions). Config is built from `ProjectConfigs` per-file hierarchy.

### Code lens (in `queries/code_lens.rs` + `main_loop/handlers.rs`)
Three lens kinds: "N usages" (two-stage resolve via `code_lens_targets`), "N implementations" on `@class` declarations, and "overrides Parent" on methods (both pre-resolved via `code_lens()`). See [ARCHITECTURE.md](.claude/ARCHITECTURE.md#code-lens-in-queriesrs--main_looprs) for algorithm details.

### Per-file analysis phases (in `src/analysis/`)
1. **Phase 0: prescan_classes_and_aliases** — Import external classes/aliases, scan local `@class`/`@alias` declarations
2. **Phase 1: build_ir** — Walk AST, create scopes/symbols/functions/tables, lower expressions to `Expr` IR
3. **Phase 2: resolve_types** — Fixpoint loop resolving expressions until no progress

### Diagnostics
Diagnostics use a trait-based architecture with a centralized catalog in `src/diagnostics/mod.rs`:
- `DiagnosticDef` struct (`code: &str`, `severity`) with `emit()` method for creating `WowDiagnostic` instances
- `DiagnosticPass` trait with `visit_node()` (AST walk), `run()` (full-analysis pass), and `run_inject()` (inject-field pipeline) methods
- `run_all()` orchestrates all passes in three phases: `run` passes, `visit_node` passes (AST walk), and `run_inject` passes (type-mismatch → inject-field pipeline)
- All diagnostic code constants are defined centrally in `mod.rs` (e.g. `DEPRECATED`, `TYPE_MISMATCH`, `SAFETY_LIMIT`)
- `CATALOG` array collects all definitions for validation; `DEFAULT_DISABLED_CODES` lists opt-in codes; `CODE_ALIASES` maps LuaLS codes to ours

Diagnostic modules under `src/diagnostics/` (40 modules implementing `DiagnosticPass` or exporting helpers):

**Type system checks:**
- `type_mismatch.rs` — argument type mismatches against `@param` (`type-mismatch`)
- `return_mismatch.rs` — return type mismatches against `@return` (`return-mismatch`)
- `field_type_mismatch.rs` — field assignment type mismatches against `@field` (`field-type-mismatch`)
- `assign_type_mismatch.rs` — reassignment type mismatches against `@type` (`assign-type-mismatch`)
- `grouped_return_mismatch.rs` — return values not matching any tuple-union `@return` case (`grouped-return-mismatch`)
- `generic_constraint_mismatch.rs` — generic type constraint violations at call sites and class type params (`generic-constraint-mismatch`)
- `param_constraint_mismatch.rs` — `@requires T: Constraint` violations: a method call where the receiver's class type-param binding doesn't satisfy the constraint (`param-constraint-mismatch`)
- `missing_return_value.rs` — return statements with fewer values than `@return` (`missing-return-value`)
- `missing_return.rs` — functions missing return statements (`missing-return`)
- `missing_fields.rs` — missing required fields when constructing `@class` tables (`missing-fields`)

**Function/call checks:**
- `call_arity.rs` — argument count validation: `redundant-parameter` (extra args) and `missing-parameter` (insufficient args), handles method calls, varargs, optional params, and projected arity from `params<F>`. **Skips field/method calls whose receiver is a multi-type union** (`receiver_is_multi_type_union`: 2+ table-bearing alternatives, e.g. an AceGUI `WidgetA | WidgetB | ...` handle): the runtime type isn't statically known and the accessed function resolves to whichever member declares it — when only one does, the callee collapses to that member's (often stub-incomplete) signature, so arity-checking it is a false positive. Methods present on several members already resolve to `ValueType::Union` and are skipped at the `func_idx` match anyway; this extends the same leniency to the single-member case.
- `cannot_call.rs` — calling a value whose type is not callable (`cannot-call`): warns on non-function, non-constructor types (number, string, boolean, nil, bare table, etc.)
- `invalid_op.rs` — arithmetic, concatenation, or ordered comparison on incompatible types (`invalid-op`): catches `+` on strings (suggests `..`), arithmetic on booleans/nil/tables, concatenation on non-stringable types, ordered comparisons (`<`, `>`, `<=`, `>=`) on nil/boolean/function/mixed types
- `destructure_arity.rs` — destructuring more variables than a function returns (`unbalanced-assignments`): handles annotated arity, inferred body returns, overloads, `returns<F>` projections, and vararg return suppression. **Dynamic-trailing lower bound**: for a *body-inferred* function (no authored `@return`), when the last return slot resolves to `any` — a value the inference couldn't pin to a concrete type (a dynamically-keyed `table<K,V>` access like `return id, t[15], t[17] or 0`, an untyped-receiver field, etc.) — the syntactic value count is only a *lower* bound, so the over-destructure check backs off (`return_arity` returns `None`). This covers both the same-file `func.rets` path (via `return_type_at_slot` on the max slot) and the cross-file deferred-harvest path (`func.return_annotations.last()`, gated on `ext.deferred_returns` membership + no non-return-only overloads so an authored `@return any` is excluded and still warns). Confidently-typed trailing returns (literals, classes, primitives) keep the warning; the inferred *type* display is untouched (only the diagnostic relaxes)
- `discard_returns.rs` — ignored `@nodiscard` return values (`discard-returns`)
- `multi_return_projection.rs` — `returns<F>` truncation when F has >1 return annotation (`multi-return-projection`)

**Variable/field/global checks:**
- `undefined_global.rs` — references to unresolved global names (`undefined-global`)
- `undefined_field.rs` — accessing nonexistent fields on `@class` tables (`undefined-field`); also fires on **module-private record tables** — a `local X = {}` whose complete field set is statically known. Gated by `collect_pure_record_symbols` (single-version local with a `{}` constructor RHS, never referenced bare/escaping) + `is_closed_record` (no `key_type`/`value_type`/metatable/dynamic bracket write/etc.), so parameter-inferred records, the addon namespace, globals, maps, and escaping tables are excluded
- `unused_local.rs` — unreferenced local variables (`unused-local`, HINT)
- `redefined_local.rs` — same-scope local variable redefinition (`redefined-local`)
- `shadowed_local.rs` — local variable shadows a variable from an outer scope (`shadowed-local`, HINT)
- `duplicate_index.rs` — duplicate keys in table constructors (`duplicate-index`)
- `duplicate_set_field.rs` — setting a field already set on `@class` tables (`duplicate-set-field`)
- `inject_field.rs` — setting undeclared fields on `@class` tables (`inject-field`, HINT)
- `create_global.rs` — implicit global creation via assignment or function definition (`create-global`)
- `mixed_enum_values.rs` — `@enum` with mixed number/string values or unsupported value types (`mixed-enum-values`)
- `class_shadows_builtin.rs` — workspace `@class` that declares its own explicit `@field`s reuses a built-in WoW API stub class name (`class-shadows-builtin`); bare stub-name reuse with no explicit `@field` (mixins, namespaces, frames) is not flagged

**Access control:**
- `access.rs` — `@private`/`@protected` visibility violations (`access-private`, `access-protected`)
- `need_check_nil.rs` — field/method access on possibly-nil values (`need-check-nil`, default-disabled)
- `nil_index.rs` — possibly-nil table key in bracket access (`nil-index`)
- `nil_table_key.rs` — nil in table key type annotations, e.g. `table<string?, V>` (`nil-table-key`)
- `wrong_flavor_api.rs` — calls to APIs not available in project-declared flavors (`wrong-flavor-api`)
- `callback_events.rs` — string-literal event names passed to a callback-registry consumer (`:RegisterCallback("…")`/`:TriggerEvent("…")`, marked `@callback-event-arg`) that the registry never declared via `GenerateCallbackEvents` (`unknown-callback-event`, default-disabled). Purely syntactic via `AnalysisResult::callback_event_set_for_string` (shared with completion): the registry receiver usually types as bare `table` cross-file, so the receiver path/method/arg-index are read from the AST and the receiver is canonicalized (`canonicalize_member_path`) to match the producer site. Suppressed unless the registry's event set is `complete`.
- `expression_type.rs` — undefined variables and return type mismatches in `expression<C, R>` string arguments (`undefined-field`, `type-mismatch`). Exports the shared, closure-based `infer_expression_type(tree, name_type)` result-type inference (parses `return <expr>` and walks operators/identifiers); `resolve_call.rs` reuses it to infer a generic `R` from the expression body and bind it into the call's generic substitutions when `R` is one of the method's `@generic` params (so `@return Schema<R>` becomes `Schema<number>` etc.). **Incomplete context degradation:** when `C` contains a `self`/generic-param member that can't be bound to a concrete table (`resolve_expression_tables` returns `incomplete = true`), `ExpressionArg.context_incomplete` is set and this pass skips — the resolved members (e.g. a `Builtins` half) still power hover/completion, but emitting `undefined-field` would be a false positive on names the unbound member would supply. This is the fallback when the deferred field-type-args harvest (below) can't recover the binding.

**Annotation validation:**
- `function_annotation_checks.rs` — comprehensive function-level annotation validation: `@param` name mismatches (`undefined-doc-param`), duplicate `@param` (`duplicate-doc-param`), `@return` type resolution, `@overload` type resolution, `@generic` on class methods (`redundant-class-generic`), `params<F>` position/shape validation, `@requires` param-name-must-be-a-class-type-param + constraint type resolution (`malformed-annotation`/`undefined-doc-name`), and `@return self<X>` type-arg arity vs the class's type params (`malformed-annotation`)
- `annotation_metadata.rs` — annotation comment scanning: duplicate `@constructor` (`duplicate-constructor`), `@constructor` return validation (`constructor-return`), `@builds-field` without `@return self` (`builds-field-not-self`), `@return ClassName` instead of `@return self` (`return-self-class-name`, HINT), bare `return` with all-optional `@return` types (`implicit-nil-return`, HINT), duplicate `@field` (`duplicate-doc-field`), duplicate `@alias` (`duplicate-doc-alias`), and the **`deprecated` call-site check** ("Part 3" of `run()`: a call whose callee resolves to a `@deprecated` `Function`). The deprecated check is **flavor-aware** — gated by `crate::flavor::deprecation_suppressed(analysis.addon_flavors, func.flavors)` so a retail-sourced deprecation that's still the live API on a flavor the addon targets (e.g. `GetMerchantItemInfo` on Classic) is not flagged. `addon_flavors` (from `config.addon_flavors_for`) is the addon's declared breadth incl. the `.toc` `## Interface:` fallback; `0` (no flavor signal) disables suppression and preserves prior behavior.
- `malformed_annotation.rs` — unknown or incomplete `---@` annotations (`malformed-annotation`)
- `doc_field_no_class.rs` — `@field` annotations not preceded by `@class` (`doc-field-no-class`)
- `doc_func_no_function.rs` — function-level annotations (`@param`, `@return`, `@overload`, `@generic`, `@nodiscard`, `@deprecated`, `@constructor`, `@builds-field`, `@built-name`, `@built-extends`, `@flavor-narrows`, `@type-narrows`, `@narrows-arg`, `@defclass`) not attached to a function definition (`doc-func-no-function`)
- `undefined_doc_class.rs` — undefined class names in `@class Foo: Parent` inheritance, circular inheritance chains, and inheriting from primitive/literal types (`undefined-doc-class`, `circle-doc-class`, `invalid-class-parent`)
- `undefined_doc_name.rs` — undefined type names in annotations (`undefined-doc-name`)
- `unknown_diag_code.rs` — unknown code in `@diagnostic` directives (`unknown-diag-code`)
- `incomplete_signature_doc.rs` — functions with partial `@param`/`@return` annotations (`incomplete-signature-doc`, HINT, default-disabled)
- `missing_annotation.rs` — non-file-local functions (globals + cross-file table methods/fields) entirely missing `@param`/`@return` (`missing-param-annotation`/`missing-return-annotation`, HINT, default-disabled). Unlike `incomplete-signature-doc`, fires even with zero annotations, but is scoped to functions reachable beyond their file. Skips `local function` and anonymous literals. **Bare `function foo()`**: `bare_name_is_file_local()` skips it only when `foo` rebinds a forward-declared local (detected via `is_local_declaration_site` on the binding symbol's first version). **Dotted/colon methods (`A.B`/`A:B`)**: escape is decided by `escaping_methods()`, which reuses `scan_file_globals()` as the source of truth — a method is checked iff the workspace scan registers it as a cross-file symbol (root is a global/`@class`/addon-ns, or a local table assigned onto the addon ns via `ns.Field = LocalTable`), matched back to the per-file `Function` by `def_node.start` byte offset. Methods on file-private local tables are therefore skipped. The escaping-offset set is built lazily on the first dotted candidate.

**AST & style checks:**
- `ast_checks.rs` — AST-traversing pass consolidating: empty blocks (`empty-block`, HINT), unbalanced assignments (`unbalanced-assignments`), redundant values (`redundant-value`), redundant return values (`redundant-return-value`), code after break (`code-after-break`, HINT), unreachable code after return (`unreachable-code`, HINT), count-down loops (`count-down-loop`), unused functions (`unused-function`, HINT), redundant return (`redundant-return`, HINT), deprecated symbol usage (`deprecated`)
- `trailing_space.rs` — lines ending with whitespace (`trailing-space`, HINT)
- `not_precedence.rs` — `not x <cmp> y` parsing as `(not x) <cmp> y` (`not-precedence`, HINT)
- `redundant_logical.rs` — redundant `or`/`and` where LHS truthiness/falsiness makes RHS unreachable, plus `and` whose LHS is always truthy (the operator is a no-op). The `x and y or z` Lua ternary idiom suppresses only the outer `or`; the inner `and` is still flagged when `x` is statically truthy. (`redundant-or`, `redundant-and`, HINT)
- `redundant_condition.rs` — `if`/`elseif`/`while`/`repeat...until` condition whose type is always truthy or always falsy (`redundant-condition`, HINT, default-disabled)
- `unused_vararg.rs` — functions declaring `...` but never referencing it (`unused-vararg`, HINT, default-disabled)
- `unused_function.rs` — cross-file unused function/method detection (`unused-function`, HINT): tracks references via call_resolutions + field-access token walk with inheritance; skips library files, `_`-prefixed names, and interface methods (2+ tables sharing the same method name)

**Unknown-type diagnostics (strict typing, all default-disabled):**
- `unknown_param_type.rs` / `unknown_return_type.rs` / `unknown_local_type.rs` / `unknown_field_type.rs` — sites whose type couldn't be inferred (`unknown-param-type`, `unknown-return-type`, `unknown-local-type`, `unknown-field-type`, HINT). See [ARCHITECTURE.md — Unknown-type diagnostics](.claude/ARCHITECTURE.md#unknown-type-diagnostics-strict-typing).

**Special:**
- `safety-limit` (ERROR) — emitted when analysis is incomplete due to safety limits

To add a new diagnostic: add a `DiagnosticDef` constant to `mod.rs`, create `src/diagnostics/new_thing.rs` implementing `DiagnosticPass`, add `mod new_thing;` to `mod.rs`, register the pass in `run_all()`, and add the constant to `CATALOG`. Suppression via `@diagnostic disable:new-thing` works automatically by matching the code string. Some modules are "hybrid": they implement `DiagnosticPass` for the post-analysis phase AND export `pub(crate)` helper functions called from `build_ir.rs` / `resolve.rs` during IR construction. **Also add the diagnostic to the table in `README.md`.**

## Bug fixes

When fixing a bug, always add a regression test covering the fix. Add test assertions to the appropriate existing test file (see test file layout below) using the annotation format (`hover:`, `def:`, `sig:`, `diag:`, etc.). Run `cargo test` to confirm the new test passes.

### Investigating false positives in real addon code

**CRITICAL**: When reproducing a diagnostic false positive reported in a real addon (e.g. TradeSkillMaster), **always use `--scan-dir` pointing to the FULL addon root** — not a subdirectory. A partial scan misses cross-file classes, defclass calls, inherited fields, and addon namespace resolution, producing many spurious diagnostics that don't exist with the full scan. First reproduce the exact diagnostic with a full scan before investigating the code.

```bash
# WRONG — partial scan produces false positives that mask the real issue:
cargo run -- test-query /path/to/addon/SubLib/Source/File.lua:386:1 --with-stubs --scan-dir /path/to/addon/SubLib

# RIGHT — full workspace scan for accurate diagnostics:
cargo run -- test-query /path/to/addon/SubLib/Source/File.lua:386:1 --with-stubs --scan-dir /path/to/addon
```

## Conventions

- Byte offsets are `u32` throughout the IR (not `usize`)
- `SymbolIndex`, `FunctionIndex`, `TableIndex`, `ExprId` are all `usize` type aliases
- Symbol versions track reassignments: `local x = 1; x = "hi"` creates two versions
- External data is immutable after `PreResolvedGlobals::build()`
- `@meta` files suppress all diagnostics (they're declaration-only stubs)
- `@field name? type` — the `?` is stripped from the field name at parse time in `annotations.rs` and the type is wrapped in `Union(type, nil)`. Field HashMap keys never contain `?`. Same pattern as `@param name?` handling.
- `@alias (opaque) Name type` — Creates a nominally distinct type alias: `ValueType::OpaqueAlias(String, Box<ValueType>)`. Parsed in `annotations/mod.rs` by stripping the `(opaque)` prefix. Stored in `ir.aliases` / `ext.aliases` as the `OpaqueAlias` variant (not the raw inner type). Different opaque aliases with the same inner type are NOT assignable to each other. Base values/literals are assignable into an opaque alias (Rule 2). Opaque aliases flow out to their base type freely. Arithmetic and other operators unwrap to the inner type; results decay to the base type. Hover displays the alias name.
- `@class (partial) Foo` — Parsed in `annotations/mod.rs` by stripping the `(partial)` prefix before the class name. `(exact)` is also recognized. Currently parse-only — the modifier is accepted but has no effect on diagnostics.
- `@class Foo : table<K, V>` — Class inheriting dictionary key/value types. Parent name parsing in `annotations/mod.rs` uses `split_at_top_level` to handle commas inside `<>`. During inheritance resolution (prescan.rs pass 3, pre_globals/mod.rs, pre_globals/build_on_stubs.rs), `table<K,V>` parents set `key_type`/`value_type` on the child's `TableInfo` without adding to `parent_classes`. This enables typed `pairs()` iteration over class-typed tables. The `undefined-doc-class` diagnostic skips parameterized parents (those containing `<`).
- `T & U` (intersection type) — `AnnotationType::Intersection(Vec<AnnotationType>)` / `ValueType::Intersection(Vec<ValueType>)`. Parsed via `&` with higher precedence than `|` (split `|` first, then `&` inside each union member). An intersection is assignable to X if ANY member is; X is assignable to an intersection if assignable to ALL members. Field access checks all member tables. Used by `CreateFrame` stub to combine frame type with template mixin (`T & Tp`).
- `T!` (non-nil assertion / lateinit) — `AnnotationType::NonNil(Box<inner>)` wraps the inner type. Resolves to the inner type with nil stripped. On `@field` or `---@type`, sets `FieldInfo.lateinit = true`, which suppresses `field-type-mismatch` for nil assignments and ensures the field's resolved type is non-nil (no `need-check-nil` on access). Hover shows `T!`. In `fun()` return types (e.g. `fun(): K!, V!`), the `NonNil` wrapper is preserved in `Function.return_annotations_raw` even though the resolved `return_annotations` loses it. For-in variable resolution (`Expr::ForInVar` in `resolve.rs`) checks `return_annotations_raw` for `NonNil` to strip nil from iteration variables — this is how `pairs`/`ipairs`/`next` stubs (`stubs/overrides/`) express that Lua tables cannot store nil values, avoiding hardcoded position checks.
- `{field: type, ...}` (anonymous table shape) — `AnnotationType::TableLiteral(Vec<(String, AnnotationType)>)`. Parsed in `parse_type()` when the string starts with `{` and ends with `}`, splitting on `,` at top level and then `field: type` pairs. Resolves via `materialize_table_literal()` in `prescan.rs` which creates a `TableInfo` with the specified fields. Supports optional fields (`field?: type`) which become `Union(type, nil)`. Works in `@param`, `@return`, `@type`, `@alias`, and inside intersections (`T & {field: type}`).
- `[T1, T2, ...]` (LuaLS tuple syntax) — parsed in `parse_type()` and **lowered to a `TableLiteral` with integer-literal keys** (`{ [1]: T1, [2]: T2 }`), reusing the existing materialization/field-access/`undefined-doc-name` machinery rather than a dedicated tuple value type. `split_at_top_level`/`extract_type_prefix` track `[`/`]` (alongside `<>`/`()`/`{}`) so the inner commas are protected and a `T[]` array segment stays intact. Element access (`x[1]`) resolves the per-position type via the `[1]`/`[2]` field-name convention shared with `{ [1]: ... }`.
- `?T` (prefix-optional shorthand) — some authors (and LuaLS) write the optional marker as a prefix (`?number`) rather than the canonical suffix (`number?`). `parse_type()` strips a leading `?` and returns `Union(T, nil)`, identical to `T?` — accepted leniently with no diagnostic (no type legitimately starts with `?`). On `@param`/`@field` this makes the slot optional via `annotation_type_is_nullable`, suppressing the `missing-parameter` false positive the dropped marker used to cause.
- Comma-separated `@return T1, T2` (LuaLS single-line multi-return) — one return slot per top-level comma segment, each `<type> [name]`, with an optional trailing `# description` or free-text description on the final segment. Parsed by `annotation_types::parse_multi_return_line()` (left-to-right, matching LuaLS: the type list continues only when a `<type> [name]` is *immediately* followed by a top-level comma), so a single return whose **description** contains a comma (`@return number red Red color, from 0 to 1`) stays one return — the helper returns `None` there and the caller falls back to `parse_return_line` (which still owns parenthesized tuple / tuple-union / vararg / `built` forms). The old "comma-separated return types are not supported" `malformed-annotation` was removed.
- LuaLS-only `@diagnostic` codes — `diagnostics/mod.rs::LUALS_ONLY_CODES` lists codes LuaLS defines but wowlua_ls has no equivalent for (e.g. `lowercase-global`). They're folded into `known_codes()` so `unknown-diag-code` accepts them silently; they suppress nothing (no mapping). Codes that *do* map to one of ours go in `CODE_ALIASES` instead.
- `0` / `-1` / `0xFF` (number-literal type) — `ValueType::NumberLiteral(String)` (the variant is kept **last** in the `ValueType` enum so it doesn't shift serde variant indices in the precomputed-stub blob — adding it mid-enum breaks blob deserialization). The String preserves the source spelling (kept `Eq`/`Hash`; f64 is not). Detected by `is_number_literal()` in `annotations/annotation_scanning.rs` (decimal/float/hex, optional leading `-`), used by `resolve_annotation_type` and the `undefined-doc-name` validator (`check_annotation_type_names`). Mutually assignable with plain `number`. Decays to `Number` under arithmetic/comparison via `into_decay_number_literal()` (called in `resolve_binary_op_standalone`); `make_union` collapses `number | 0` → `number`. Hover shows the literal spelling. Primary use is tuple-union sentinel cases like `@return (number, string) | (0, nil)`. **Only annotation literals become `NumberLiteral`; source-code numeric literals stay `Literal(ValueType::Number)`.**
- `ValueType::FunctionSig(Box<FunctionShape>)` (inline function signature) — a self-contained structural function type carrying `FunctionShape { params: Vec<ShapeParam>, returns: Vec<ValueType>, is_vararg }` (`ShapeParam { name, ty, optional }`). Unlike `ValueType::Function(Some(FunctionIndex))`, it holds no arena index, so it survives cross-file. **Runtime-only — never serialized into the precomputed-stub blob**, so it can live mid-enum without breaking blob deserialization (it's constructed lazily during the cross-file harvest, not from stubs). Built by `analysis/deferred.rs::lift_local_func_to_shape()` when a defining file's harvested return type is a local function (`Function(Some(local_idx))`): the lift converts the per-file `Function` into a `FunctionSig` (depth-guarded by `LIFT_MAX_DEPTH`) instead of decaying to bare `function`, making `Test() -> fun(a: string)` survive cross-file. Treated as callable (`cannot_call.rs`), truthy, and mutually assignable with `Function`. Formatted by `format_function_shape()` in `queries/format.rs`.
- `ValueType::TableShape(Box<TableShape>)` (inline anonymous table shape) — a self-contained structural table type carrying `TableShape { fields: Vec<(String, ValueType)> }` (sorted by name). Like `FunctionSig`, it holds no arena index so it survives cross-file, and is **runtime-only — never serialized into the precomputed-stub blob** (appended after `FunctionSig` so blob variant indices stay stable; constructed lazily by the cross-file harvest, not from stubs). Built by `analysis/deferred.rs::wrap_overlay_shape()` when a deferred function returns an external class instance carrying per-file overlay fields (`frame.DropDown = ...` on a `CreateFrame` result): the return is lifted to `Class & TableShape` so the injected fields resolve, complete, and hover cross-file, and the surrounding `Intersection` suppresses `undefined-field` (concrete-instance policy). **Per-file aggregation (pre-existing):** `overlay_fields` is keyed by the *shared* external class index, so the shape carries the union of every same-classed instance's injected fields in the defining file — matching what same-file hover already shows for a `Frame`-typed local. Field access (`resolve.rs` `FieldAccess`) and completion/hover (`queries/completion.rs`, `queries/hover.rs::shape_field_hover_at`) read shape members via `ValueType::collect_shape_field_types`/`collect_shape_field_names`; treated as a table (truthy, table-kind, permissively assignable to/from any table — no structural rejection). Formatted as `{ field: type, ... }` in `queries/format.rs`.
- `NarrowKind::NumCompare { op, bound }` (numeric case discrimination) — a `if x > 1` style guard on a bare symbol records `(Operator, bound_string)` in `Analysis::num_compare_narrowed_symbols` (detected in `narrowing.rs`, oriented via `flip_comparison` when the literal is on the LHS). An ordered comparison (`<`/`>`/`<=`/`>=`) errors at runtime on nil, so reaching the then-branch proves `x` is non-nil: for a plain scalar the narrowing block also adds `x` to `narrowed` and pushes a `StripNil` version (`narrow_siblings` returns `false` for non-tuple-union symbols, gating this — for tuple-union members the `OverloadNarrow` machinery owns the narrowing and the crude `StripNil` is skipped). `narrow_kind_for` returns `NumCompare` after `ClassEq`/`StripTruthy`/`StripFalsy` but **before** plain `StripNil`, since `NumCompare` now implies non-nil and is at least as strong (a compound `x and x > 0` still keeps the stronger truthiness narrowing). In `overload_type_compatible_with` → `overload_type_survives_num_compare` (resolve.rs), a tuple-union case is eliminated when its slot is `Nil` or when a `NumberLiteral` value fails `value <op> bound` (parsed via `parse_num_literal_str`); plain `number`/`any` slots always survive (no range tracking).
- `...T` (variadic return) — `AnnotationType::VarArgs(Box<AnnotationType>)`. When `@return ...T` is the last return annotation, it fills all remaining return slots with type `T`. Stored as `Function.has_vararg_return: bool`. The vararg portion is optional (no `missing-return-value` for it) and `redundant-return-value` is suppressed. Multiple returns must use separate `@return` lines (comma-separated multi-return on a single `@return` line is not supported).
- `@generic T, ...M` (variadic generic) — The `...M` part is stored as `("...M", None)` in `generics: Vec<(String, Option<String>)>`. `...M` in type positions resolves to `TypeVariable("...M")`. During call-site binding, excess arguments past positional params are collected into `Intersection(types)`. Nested intersections are flattened during substitution. See [ARCHITECTURE.md](.claude/ARCHITECTURE.md#variadic-generics-generic-t-m).
- `@narrows-arg N` (in-place argument narrowing) — `Function.narrows_arg: Option<usize>`. When a function with `@narrows-arg N` is called as a bare statement, the Nth argument's symbol gets a new `SymbolVersion` with `type_source = call_expr_id`. Phase 2 resolves the call expression to the return type (with generics substituted), which becomes the argument's new type. See [ARCHITECTURE.md](.claude/ARCHITECTURE.md#narrows-arg-n-in-place-argument-type-mutation).
- `@creates-global N` (implicit named-global side effect) — marks a function (e.g. `CreateFrame`, `CreateFont`) whose call creates a global named by the string-literal at 1-based param `N`. Stored as `ExternalGlobal.creates_global: Option<CreatesGlobalSpec>` (just `name_param`; defined in `annotations/annotation_scanning.rs`), parsed in `annotations/mod.rs`. **Never hard-code creating-function names in the engine** — this annotation is the mechanism (CreateFrame's spec lives in `stubs/overrides/CreateFrame.lua`, the Font functions' in `stubs/overrides/CreateFont.lua`). Detection: `scan_globals.rs::scan_created_globals(root, &CreatesGlobalMap)` emits an `ExternalGlobal` (typed `Variable(FunctionCall)`, **empty `returns`**, `deferred_call_type = true`, `def_start` = the creating call's offset) for each match; the spec map is built from stub globals via `build_creates_global_map()` (also cached on `PreResolvedGlobals::creates_global_specs`). Detection runs in the **stub-aware pass**, not pass 1: `complete_directory_scan` (LSP full scan, over cached trees), `scan_paths_with_overrides` via `scan_file_globals_with_synth` (scan_workspace path), and the incremental rebuild (`rebuild.rs`/`handlers.rs`). Pass 1 (`scan_lua_file_cached`) passes an empty map because it runs before stubs load. Entries are registered first/prepended so they win over a same-named assignment.
  - **Type harvest (no type token).** The created global's type is **not** reconstructed from annotations — it is harvested from the creating call's *resolved* return type, so a `CreateFrame(..., "Template")` call yields `Frame & Template` (the template overload's intersection), matching the same-file local-assignment type. Mechanism mirrors deferred function returns (`analysis/deferred.rs`): `build`/`build_on_stubs` record each created global in `PreResolvedGlobals::deferred_call_globals` (`SymbolIndex → DeferredCallGlobal { path, call_offset }`, plus `_by_path` and a `deferred_call_global_cache` RwLock memo — all `#[serde(skip)]`, no blob impact) and leave the symbol's `resolved_type` unset (the mod.rs deferred-funcall pass skips them). `resolve_deferred_call_global_type()` lazily re-analyzes the defining file, matches `Expr::FunctionCall.call_range.0 == call_offset` (ret_index 0), reads `resolved_expr_cache`, and lifts to ext space. Reads route through a per-file **symbol overlay** (`Ir::symbol_overlay`, checked in `sym()` exactly like `func()`'s `overlay`), warmed by `Ir::ensure_symbol_overlay()` in a one-time pre-pass at the top of `resolve_types()` (so aliased locals get the type too). `deferred_call_type` is `#[serde(skip)]` (workspace-only). **Detection is stub-only**: only functions defined in stubs are scanned as `@creates-global` sources (the spec map is built from stub globals).
  - When adding `CreatesGlobalSpec` semantic fields, update `global_semantic_eq` in `rebuild.rs` and bump `BLOB_VERSION`.
- `@generates-events N [Field]` (synthesized event enum table on a class) — marks a **method** (e.g. `CallbackRegistryMixin:GenerateCallbackEvents`) whose call `Receiver:Method({ "OnFoo", SomeEvents.OnBar, ... })` synthesizes an enum-like `Field` table (default `Event`) on the receiver class. Stored as `ExternalGlobal.generates_events: Option<GeneratesEventsSpec>` (`events_param` + `field_name`; defined in `annotations/annotation_scanning.rs`), parsed in `annotations/mod.rs`. **Never hard-code the method/field names in the engine** — the spec lives in `stubs/overrides/CallbackRegistryMixin.lua`. Detection **rides the defclass scan**, not a separate pipeline: `DefclassContext` carries a `generates_events` map (leaf method name → spec, built from all_globals), and `scan_defclass_calls_with_context` (`scan_defclass.rs::collect_generated_event_classes`) emits a `ClassDecl` for the receiver carrying one field of type `TableLiteral([(event, string), …])` per call. Array entries: string literals use their value; field references (`SomeEvents.OnBar`) use the leaf name (value==name convention). Only **single-name receivers** (`names.len() == 2`: `[Receiver, Method]`) are handled. The synthesized `ClassDecl` merges into the receiver's `@class` overlay via `merge_defclass_into_overlays` (field added only if absent — multiple `GenerateCallbackEvents` calls on one receiver don't union). Because it rides the defclass scan, the defclass gating is widened to include `generates_events`: `needs_defclass` in `scan.rs` (both full-scan paths), and `rebuild_caches` in `state.rs` adds the method leaf names to `cached_defclass_func_names` and forces `cached_needs_defclass`. Add new `GeneratesEventsSpec` semantic fields to `global_semantic_eq` and bump `BLOB_VERSION`. The synthesized `TableLiteral` field is **not closed** (accessing an absent event resolves leniently, no `undefined-field`).
- `@callback-event-arg N` (callback-registry event-name validation/completion) — marks a **consumer** method (`CallbackRegistryMixin:RegisterCallback`/`UnregisterCallback`/`TriggerEvent`/`RegisterCallbackWithHandle`) whose 1-based arg `N` is an event name. Stored as `ExternalGlobal.callback_event_arg: Option<usize>`, parsed in `annotations/mod.rs`, spec lives in `stubs/overrides/CallbackRegistryMixin.lua` (never hard-code method names). Pairs with `@generates-events` (the producer): `scan_callback.rs` collects `path → event-set` registries (keyed by canonical receiver path, addon-ns alias rewritten via `canonicalize_member_path`; events come from an inline array **or** a cross-file `addonTable.Constants.Events`-style string-array constant resolved at `merge_callback_registries` time). The consumer side is **purely syntactic** — `addonTable.CallbackRegistry` types as bare `table` cross-file so call resolution can't be used; `AnalysisResult::callback_event_set_for_string` reads the receiver path/method/arg-index from the AST, re-detects the addon-ns var on the same tree (`detect_addon_ns_var`) and canonicalizes identically to scan time, then looks up `ext.callback_registries`. Drives string-literal completion (`queries/completion.rs`, always on) and the default-off `unknown-callback-event` diagnostic (suppressed unless the event set is `complete`, so unresolved/colliding registries never false-positive). Workspace maps (`callback_registries`, `callback_event_methods`) are `#[serde(skip)]` on `PreResolvedGlobals`; the `@generates-events`/`@callback-event-arg` fields ride the stub blob, so adding semantic fields needs `global_semantic_eq` + `BLOB_VERSION`. **Multi-addon scoping:** addon-namespace canonical paths are prefixed with the file's addon identity (`config.addon_name_for` at scan time / `Ir::addon_folder_name` at query time, via `scope_addon_ns_path`), so two addons' `addonTable.CallbackRegistry` don't collide. **Incremental refresh:** `rebuild.rs::maybe_rebuild_workspace` re-scans the edited file (gated on a `@generates-events` method name in the text or a prior contribution) using the cached `WorkspaceState::cached_generates_events_methods` map, updates `ws_file_callback_registries`/`ws_file_string_consts`, and forces a Full warm on change. **Validated on real addons: 0 false positives, found genuine unregistered-event bugs in shipping code.** Scope limits: receiver matched by canonical name (global/class/addon-ns reliable; bare locals are recorded but collide cross-file under a shared name → merge marks incomplete → suppressed); inline-array or resolvable string-array-constant event lists only (dynamic/computed lists → incomplete → suppressed).
- `@requires T: Constraint` (method availability gating) — `Function.requires_constraints: Vec<(String, String)>` (param name + raw constraint string). Parsed in `annotations/mod.rs`. At call-resolution time, `CallResolution.receiver_param_subs: HashMap<String, ValueType>` records the receiver class's type-param bindings (from `class_type_param_subs`). The `param-constraint-mismatch` diagnostic checks each `requires_constraints` entry against the bound type via `resolve_class_constraint` + `is_assignable_to`/`is_table_subtype`, emitting at the method-name range. Constraints use subtype (`:`) semantics, not equality.
- `@shape <type>` / `@shape <ClassName> <type>` (userdata/mixin plain-table escape) — declares structural plain-table forms a `@class` accepts even though the value lacks the class's methods (e.g. `ItemLocation` accepts `{bagID, slotIndex} | {equipmentSlotIndex}`, `ColorMixin` accepts `{r, g, b, a?}`). Resolved shapes live on `TableInfo.accept_shapes: Vec<ValueType>` (serialized — rides the stub blob, so adding it bumped `BLOB_VERSION`); the raw forms live on `ClassDecl.shape_annotations: Vec<AnnotationType>`. Two surface forms, disambiguated in `annotations/mod.rs` by whether the comment block has a `@class`: **block-tied** `@shape <type>` (no name) attaches to that block's class; **standalone** `@shape <ClassName> <type>` (parsed by `parse_named_shape`) emits a *shape-only* `ClassDecl` (`ClassDecl::shape_only`) that merges onto the named class by name in the build — the standalone form lets stub overrides add a shape **additively** without re-declaring `@class`. Acceptance is in `analysis/mod.rs::is_table_subtype_impl`: when the actual is a non-class table and the expected class has non-empty `accept_shapes`, `table_matches_shape_impl` accepts it if it matches any shape — a table **literal** via the normal structural field walk, or a `table<string_literal_keys, V>` dict via key-coverage (mirroring the existing `table<keys,V>` → `@class` rule). Gated on a non-class actual + a matching shape, so unrelated tables and wrong-typed fields still mismatch (no blanket hole). **A shape-bearing class is matched against plain tables ONLY by its shapes** — the generic structural-field match (`fields_structurally_match`) and the `table<keys,V>` key-coverage path are both skipped when `bt.accept_shapes` is non-empty (else, once fields are nilable per below, an all-optional class would vacuously accept any table). The parallel detail/excess paths follow suit: `structural_mismatch_details_impl` returns `None` for shape classes (so `type_mismatch.rs` doesn't suppress via a vacuous all-`Missing` walk — a non-matching table emits `type-mismatch`), and `checks.rs::check_excess_structural_fields` returns early (a shape-matched plain table's extra fields are tolerated, not flagged `inject-field`). `missing_fields.rs` gates on `accept_shapes` non-empty + `is_table_subtype` so a shape-matched constructor doesn't also report `missing-fields`. **Read-side nilability is derived from the shape**: `pre_globals::apply_shape_field_nilability` (called in all three build paths after `accept_shapes` is populated) nil-unions any class field that is named in the shape but not required (present and non-nil) in *every* member — e.g. ItemLocation's bagID/slotIndex/equipmentSlotIndex all become `number?` (each absent from one union member), ColorMixin's `a` becomes `number?` while r/g/b stay required. This keeps field types honest without re-declaring the vendor-authored `@field`s (`shape_optional_field_names` computes the set; methods and fields not named in any shape member are untouched). **Two build paths both resolve shapes**: `BuildContext::populate_class_fields` (`pre_globals/mod.rs`, the from-scratch regenerate path) AND `BuildOnStubsContext::populate_class_fields` (`build_on_stubs.rs`, the runtime workspace path) — both must call `resolve_field_annotation` on `class.shape_annotations` then `apply_shape_field_nilability`. Per-file local `@class` shapes resolve in `prescan.rs`. Add `shape_annotations` to `class_semantic_eq` (rebuild) and `"shape"` to `malformed_annotation.rs::KNOWN_TAGS`. **Editor support** (like any new annotation): the tag is in the completion `TAGS` list (`queries/completion.rs`, class context) and gets a `@shape` rule in the shared TextMate grammar (`editors/vscode/syntaxes/lua.tmLanguage.json`, copied to the JetBrains vendored grammar — keep both in sync). **Stub-override gotcha**: an override file's *stem* must not collide with a generated/vendor stub file name (`orchestrate.rs` `override_stems` skips the vendor file of the same stem, *replacing* it) — the shapes live in the deliberately-named `stubs/overrides/MixinShapes.lua` (no `@class`, so the generated `ItemLocation`/`ColorMixin` classes are preserved and only augmented).
- `@return self<X>` (re-parameterized self return) — `Function.returns_self_type_args: Option<Vec<AnnotationType>>` alongside `returns_self = true`. Parsed from `AnnotationType::Parameterized("self", args)` in `build_ir.rs` and `pre_globals/mod.rs`. At call resolution (`resolve_call.rs` `returns_self` branch), the args are resolved + generic-substituted and stored in `call_type_args` so the call result displays as `Receiver<X>` regardless of the receiver's actual type args. If a type arg uses `T!` (NonNil), nil is stripped from the substituted type after generic substitution (e.g. `@return self<T!>` on `Publisher<string?>` yields `Publisher<string>`). Plain `@return self` (without type args) also propagates the receiver's type args into `call_type_args` so that chained calls can resolve class type params from intermediate results. Hover/signature formats it via `Self::self_return_text(func)` in `queries/format.rs`.
- `@alias Foo<K,V> V[]` (parameterized alias) — `AliasDecl` has `type_params: Vec<String>`. Stored in `ir.parameterized_aliases` / `ext.parameterized_aliases` as `(Vec<String>, AnnotationType)` (type params + raw body). When `Parameterized("Foo", args)` is encountered during resolution, `substitute_alias_type_params()` replaces type param names in the body with concrete args, then the substituted type is resolved normally. Supports colon syntax (`@alias Foo<K,V>: V[]`) and spaces in type params (`@alias Foo<K, V>`). Type params may carry constraints (`@alias Box<T: Frame> { value: T }`): the constraint is split off at parse time so the stored param name stays bare (`T`), and `AliasDecl.type_param_constraints` (parallel `Vec<Option<String>>`, mirroring `ClassDecl`) records the raw constraint string. `undefined_doc_class.rs` validates the constraint type name (`undefined-doc-name`). Constraints are pre-parsed and stored at registration time as `Vec<Option<(String, AnnotationType)>>` into `Ir::parameterized_alias_constraints` / `PreResolvedGlobals::parameterized_alias_constraints` (the latter is `#[serde(skip)]` — populated at runtime during workspace scanning, not from the stub blob, so no `BLOB_VERSION` bump; stub aliases carry no constraints). The pre-parsed `AnnotationType` avoids re-parsing the same constraint string at every use site. They are **enforced at every annotation use site** by `Ir::check_parameterized_alias_constraints` (called from the central `check_annotation_type_names` recursion, so `@type`/`@param`/`@return`/`@field`/`@cast`/alias-body positions are all covered): each `Foo<Arg>` type argument must satisfy its param's constraint (resolved via `Ir::resolve_annotation_type_for_check`, the shared core used by both alias and class constraint checking), emitting `generic-constraint-mismatch`. Args that are unresolved type variables/generics (e.g. an alias body forwarding its own param, or a function generic flowing into an alias arg) are skipped — no false positives. Aliases are still purely *substituted* (there's no call site like classes have), so this annotation-walk is the **only** place alias type-arg bounds are checked; the constraint is not re-enforced after substitution.
- **Deferred constructor self-field type args** (cross-file generic recovery) — a defclass `__init` self-field whose RHS is a chained/generic call the coarse cross-file scan can't resolve (e.g. `self._manager = UIManager.Create(state):SuppressActionLog(...)`) is typed `any` by `scan_defclass.rs`, losing the `<State>` type argument. The per-file engine *does* resolve it, so the precise type args are harvested lazily — same pattern as `@creates-global` deferred globals. Detection: `extract_self_fields_inner` records the RHS call's full byte range on `SelfFieldEntry.deferred_call_range` (only when inferred type is `any` and RHS is a funcall) → `ClassDecl.deferred_field_call_ranges` (field → range, `#[serde(skip)]`). `build_on_stubs::finish()` builds `PreResolvedGlobals::deferred_field_type_args` (`(class, field) → DeferredFieldTypeArgs { path, call_range }`, plus `_by_path` and a `deferred_field_type_args_cache` RwLock memo — all `#[serde(skip)]`, no blob impact). Harvest: `deferred.rs::resolve_deferred_field_type_args()` re-analyzes the defining file, matches `Expr::FunctionCall.call_range == call_range` (ret_index 0), reads the call's type args via `get_type_args_for_expr`, and lifts them to ext space (skipping `any`). Applied lazily in `get_expr_type_args`'s `FieldAccess` branch when the field's own lookups yield no args, caching into the per-file `field_type_args_cache` so both resolve-time (`class_type_param_subs` for `expression<T & ...>`) and query-time hover see the recovered `<State>`. `deferred_field_call_ranges` is positional and **excluded** from `class_semantic_eq` (a stale offset just degrades to the incomplete-context suppression until the next real rebuild). The harvest only *adds* type args; the `expression_type.rs` incomplete-context suppression is the fallback when it can't resolve. **Cache discipline in `get_expr_type_args`'s `FieldAccess` branch:** only *stable* sources are written to `field_type_args_cache` — the field's `Parameterized` annotation and the deferred-harvest result. Type args read from `call_type_args` (the field's own assignment expr / `extra_exprs`) are returned **uncached**, because a generic bound from a builder-built argument (`Mgr.Create(state)` where `state` is `@return built: Base`) refines from the declared base to the precise built subtype only late in the fixpoint; caching the early base binding poisons every later read (manifests as `expression<T & ...>` resolving `T` to the base, or `keyof T` constraint false positives). `get_type_args_for_expr` (the query-time twin) already re-reads `call_type_args` on a cache miss, so hover stays consistent.
- **Never special-case specific functions** (e.g. `tinsert`, `table.insert`) in the LS engine code. Behavior differences should be expressed through stub annotations (`@generic`, `@overload`, etc.) so the general type system handles them.
- **Workspace rebuild comparisons** (`main_loop/rebuild.rs`): `globals_match()`, `classes_match()`, `aliases_match()`, `events_match()`, and `self_fields_match()` use allow-list field comparisons to ignore positional/display-only fields (e.g. `def_range`, `field_ranges`, `byte_range`). When adding a new semantic field to `ExternalGlobal`, `ClassDecl`, `AliasDecl`, `EventDecl`, or `TypedSelfField`, you **must** add it to the corresponding `*_match()` function — otherwise edits that change that field won't trigger a workspace rebuild.
- **Structured logging**: Use `log::info!`, `log::warn!`, `log::error!`, `log::debug!` instead of `eprintln!`. The logger (`env_logger`) is initialized in `main.rs`; library code uses `log::` macros directly. `RUST_LOG` env var controls filtering at runtime.
- **Zero warnings policy**: Always run `cargo build` and `cargo clippy --lib` after completing changes and ensure there are zero warnings before considering work done. If clippy suggests a fix, apply it. Do not add `#[allow(clippy::...)]` suppressions unless there's a documented reason in a code comment.
- **No real addon code in source**: Never use code from real addons (e.g. TradeSkillMaster) in source comments, test names, or examples. Always generalize to fictional/generic examples.
- **Never `git stash` in a worktree**: All worktrees of a repo share a single stash stack (it lives on the common git dir, not per-worktree). Concurrent workspaces running `git stash push` / `pop` will clobber each other's entries. To shelve changes, use a per-worktree WIP commit (`git commit -m WIP`, reset later) or write to a uniquely-named ref (`git stash create` + `git update-ref refs/wip/<name>`).

## Testing

```bash
# Run all tests
cargo test

# Check all diagnostics across a workspace (the primary way to verify diagnostic behavior)
cargo run -- check /path/to/addon
# Filter to a specific file:
cargo run -- check /path/to/addon | grep "FileName.lua"
# Include hints (default is warnings+errors only):
cargo run -- check /path/to/addon --severity hint

# Evaluate a single file with type info
cargo run -- evaluate tests/annotations.lua

# Test hover/definition/signature/diagnostics at line:col
cargo run -- test-query tests/integration_stubs.lua:4:10 --with-stubs

# Test hover/definition/signature/diagnostics against a real addon project
# Use --scan-dir to load the full workspace so cross-file defclass, @builds-field,
# and addon namespace resolution all work. This is slow but necessary for accurate
# results when investigating issues in real addon code.
cargo run -- test-query /path/to/addon/File.lua:LINE:COL --with-stubs --scan-dir /path/to/addon

# Dump hover types for every identifier in a workspace (for regression baselines)
cargo run --release -- dump-types /path/to/addon --with-stubs > baseline.txt
# After changes, diff to catch hover regressions:
cargo run --release -- dump-types /path/to/addon --with-stubs | diff baseline.txt -

# Smoke test (build + integration tests)
./tests/smoke.sh
```

### Annotation format
Test expectations are embedded as comments below code lines:
```lua
local x = 5
--    ^ hover: x: number  def: local
foo(1, "hello")
--  ^ sig: fun(x: number, y: string): boolean
oldFunc()
-- ^ diag: deprecated
local y = mustUse()
-- ^ diag: none
```
Fields are separated by double-space. Supported fields: `hover:`, `def:`, `defs:`, `typedef:`, `typedefs:`, `sig:`, `diag:`, `refs:`, `comp:`, `tok:`, `hint:`, `lens:`.

The `defs:`/`typedefs:` field values are integers: the expected number of distinct sites returned by `definitions_at`/`type_definitions_at` (multi-result go-to-definition / go-to-type-definition for a global or type defined in multiple files). `def:`/`typedef:` assert the *primary* (first) result; `defs:`/`typedefs:` assert the total count.

The `tok:` field value is the semantic-token classification at the caret: the token type followed by zero or more modifiers in any order (e.g. `tok: function defaultLibrary`, `tok: method deprecated`). Use `tok: none` to assert no token is emitted at the caret.

The `lens:` field value is the expected code lens function name on the code line (e.g. `lens: greet`, `lens: sayHello`). Use `lens: none` to assert no code lens target exists at that line.

The test harness applies `ProjectConfigs::disabled_diagnostics_for()` to filter diagnostics — the same path the LSP server uses in `publish_with_config`. Tests that rely on default-off codes (`need-check-nil`, `implicit-nil-return`, `unused-vararg`, `incomplete-signature-doc`, `unknown-*-type`) must live in a subdirectory with an adjacent `.wowluarc.json` that opts in via `diagnostics.enable`. Existing examples: `tests/need-check-nil/`, `tests/incomplete-signature-doc/`, `tests/unused-vararg/`, `tests/unknown-types/`.

**Exhaustive diagnostic checking**: The harness fails on any diagnostic (WARNING, ERROR, or HINT) that is not covered by a `diag:` assertion. This catches diagnostic regressions automatically — if a code change introduces a new unexpected diagnostic, the test fails even without an explicit `diag: none` on that line. Therefore, **do not write `diag: none`** — it is redundant since the absence of an unexpected diagnostic is already verified by the harness. Only use `diag: <code>` to assert that a specific diagnostic IS emitted. Syntax errors are exempt from exhaustive checking since they are structural parse failures whose codes are free-form messages that can't be suppressed via `---@diagnostic disable`. To suppress incidental diagnostics (e.g. `unused-local`, `unused-function`, `shadowed-local`) in test files that aren't testing those specific codes, use `---@diagnostic disable: code1, code2` at the top of the file. For test files that both test a HINT diagnostic AND have incidental instances of the same code, use `---@diagnostic disable:` at the top with `---@diagnostic enable:` / `---@diagnostic disable:` blocks around the tested section.

## Stubs
WoW API stubs live in `stubs/`. Scanned at startup by `scan_workspace()` / `scan_stubs_for_test()`. Stubs are precomputed and checked in; they are regenerated by `cargo run -- regenerate-stubs`, which clones [Ketho/vscode-wow-api](https://github.com/Ketho/vscode-wow-api) to a temp directory. Local overrides live in `stubs/overrides/`.

Stub generation (including Classic-only globals from the wiki and BlizzardInterfaceResources) is handled by `src/stub_gen/`. Run `cargo run -- regenerate-stubs` to regenerate precomputed stubs. **Any change to `src/stub_gen/` or `stubs/overrides/` requires regenerating stubs and committing the updated `stubs/precomputed.bin.zst` and `stubs/precomputed-files.bin.zst`.** After resolving rebase/merge conflicts on the precomputed stub binaries (accepting either side), always re-run `cargo run -- regenerate-stubs` to ensure the blob is consistent with the current `stubs/overrides/` content.

### Fixing missing or incorrect API stubs
When a WoW API function is missing or has wrong types, **always fix the root cause in `src/stub_gen/`** rather than adding a one-off override in `stubs/overrides/`. There are thousands of WoW API functions; adding overrides for individual missing APIs doesn't scale and masks systemic discovery bugs. Trace the function through the stub generation pipeline (wiki categories, BlizzardInterfaceResources GlobalAPI.lua branches, Blizzard APIDocumentationGenerated, dedup filters) to find where it's being dropped or mistyped. Overrides in `stubs/overrides/` are reserved for cases where the upstream source data is fundamentally wrong (e.g. Blizzard's documentation declares a numeric enum type but the Lua API actually accepts strings) — not for patching gaps in the generation pipeline.

### Embedded vs external stubs (`embedded-stubs` feature)
The `embedded-stubs` Cargo feature (default on) controls how the binary loads precomputed stubs:
- **With feature (default):** Stubs are baked into the binary via `include_bytes!`. Produces a self-contained executable for standalone release downloads.
- **Without feature (`--no-default-features`):** Stubs are loaded at runtime from a `stubs/` directory next to the executable (resolved via `std::env::current_exe()`). Used for universal editor plugin packages that share one copy of the stubs across per-platform binaries.

Both modes use the same version-checking logic (`BLOB_MAGIC`/`BLOB_VERSION`). The implementation is in `src/lsp/main_loop/stub_loading.rs` (`load_precomputed_stubs()`, `stub_file_contents()`, `stubs_dir()`).

## Editor Extensions

### Shared TextMate grammar
Both the VS Code and JetBrains plugins use the same Lua TextMate grammar (`editors/vscode/syntaxes/lua.tmLanguage.json`). The JetBrains plugin vendors a copy at `editors/jetbrains/textmate/lua/syntaxes/lua.tmLanguage.json`. **When updating the grammar, copy it to both locations.**

### VS Code Extension Development

The VS Code extension requires two build steps before launching:
1. `cargo build` — build the language server binary
2. `cd editors/vscode && npm run build` — bundle the extension JS (`extension.js` → `dist/extension.js` via esbuild). The `package.json` `main` field points to `./dist/extension.js`, so the extension will fail to activate without this step.

When using `/vscode`, check whether VS Code already has a window open for the target folder **before** launching. If it does, stop and ask the user to close it — VS Code reuses the existing window and silently ignores the new `--extensionDevelopmentPath`, so the dev build won't load. The `--new-window` flag does not reliably fix this. Warning the user *after* launching is too late; the wrong instance is already foregrounded.
